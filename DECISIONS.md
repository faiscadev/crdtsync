# Design decisions

Log of design changes to [ARCHITECTURE.md](ARCHITECTURE.md) that implementation forced. ARCHITECTURE always reads as the current intended end-state; this file is *why* it changed. Newest first.

`/cs-implement` appends an entry when building a unit reveals a genuine design change ŌĆö a forced realization (the design as written is wrong, impossible, or clearly worse than what the code must do), never a mere preference. The human appends when they revise scope. Format: **date ┬Ę unit/PR ┬Ę what changed ┬Ę why**.

The entries below (2026-07-02) are a backfill: design changes made during the v0.1ŌåÆv0.2 build that predate this log, recovered from the sessions and commit history.

## 2026-07-02 ┬Ę #72 ┬Ę periodic awareness sweep wired into the runtime; timed-TTL + throttle deferred as schema-gated
**Changed:** the tokio runtime now drives `Registry::sweep` on an interval, so the reconnect-grace expiry (the session-TTL half of awareness) fires in production without an external caller ŌĆö previously `sweep` was reachable only from tests. `serve` keeps its signature (5s grace, 1s sweep by default via `ServeConfig::default`); a new `serve_with(listener, server, store, ServeConfig)` exposes the grace + sweep cadence, and the registry actor selects over inbound commands and the sweep tick, flushing outboxes after each. The registry runtime enables tokio time. **Deferred, not built:** per-entry *timed* TTL (auto-expire after silence) and server-side throttle/coalesce ŌĆö ARCHITECTURE ┬¦Awareness makes both **schema-declared** (an entry's type carries its TTL and throttle interval), and the schema layer is unbuilt, so their trigger values have no home yet. Building them now would either invent a wire field ARCHITECTURE doesn't sanction or hardcode a policy the schema is meant to own.
**Why:** the grace window is only useful if something actually sweeps; the runtime is where wall time and the connection set both live. Splitting `serve`/`serve_with` keeps the binary + existing callers untouched while letting tests drive expiry on a sub-second cadence instead of waiting the 5s default. Recognizing timed-TTL/throttle as schema-gated (rather than clock-gated) is the real finding: the clock seam (#71) unblocked *session* TTL, but the remaining two await schema ŌĆö recorded so they are not attempted prematurely.

## 2026-07-02 ┬Ę #70, #71 ┬Ę awareness reconnect grace via an injected server `Clock` + explicit sweep, not clear-on-disconnect
**Changed:** a disconnect no longer clears the client's ephemeral presence at once. The server gains a `Clock` seam (`clock::{Clock, SystemClock, ManualClock}`, wall-millis) ŌĆö the analog of the core's `Host::now`, which the server runtime lacked. On disconnect a client with live presence is marked *stale* with a `now + grace` deadline (default 5s); a reconnect (Hello) within the window cancels it; `Registry::sweep` clears every past-deadline client and fans a new `Message::AwarenessClear { channel, actor }` (wire tag 10, serverŌåÆclient) to each affected room's remaining subscribers on their own channel. Replaces #69's synchronous `Hub::clear_client_awareness` on disconnect; that method now *returns* the cleared `(room, actor)` pairs so the sweep can address the fan-out.
**Why:** ARCHITECTURE ┬¦Awareness prescribes a reconnect grace window so a brief drop (tab reload, flaky network) doesn't flicker a user's cursor out and back for every peer. That needs wall time on the server, which only the core had ŌĆö hence the `Clock` seam, injectable so tests drive presence expiry deterministically with a `ManualClock` instead of sleeping. Sweep is explicit (caller-driven) rather than a background timer so the registry stays pure/synchronous; wiring a periodic tokio sweep into the runtime is a separate follow-on. Clearing is gated on the client actually holding presence, so the common no-awareness disconnect never reads the clock (keeps the Miri-gated registry tests off the unsupported system-time call under isolation). TTL-per-entry and throttle/coalesce remain queued on the same seam.

## 2026-07-02 ┬Ę #66 ┬Ę awareness ŌĆö ephemeral LWW-per-client presence on the connection, off the op log
**Changed:** added an awareness channel to the wire: `Message::AwarenessSet { channel, key, value }` (tag 8, clientŌåÆserver) publishes/replaces this client's ephemeral entry `key` on the channel's room; the server fans it out as `Message::AwarenessUpdate { channel, actor, key, value }` (tag 9), tagged with the publisher's server-derived actor. Entries are last-writer-wins per (client, key), held in an ephemeral per-room store that is never written to the op log or a snapshot. Keys/values/actors are opaque.
**Why:** ARCHITECTURE ┬¦Awareness makes presence (cursors/selections/typing) explicitly ephemeral ŌĆö not durable, not CRDT-merged, on a lower-latency path than doc ops ŌĆö so it lives entirely outside the CRDT/convergence machinery, which is why it carries no convergence risk. Tagging fan-out with the actor (available post-auth) lets receivers attribute an entry to a human. Landed in slices: #66 wire first (this), then server ephemeral store + fan-out, then client view; TTL/throttle/reconnect-grace/auth-filter/anchors are deferred follow-ons.

## 2026-07-02 ┬Ę #63 ┬Ę handshake auth ŌĆö opaque credential in, server-derived actor out, pluggable verifier
**Changed:** the handshake gains an Auth phase between Hello and Subscribe. Wire: `Message::Auth { credential: Vec<u8> }` (tag 6, clientŌåÆserver) and `Message::AuthOk { actor: Vec<u8> }` (tag 7, serverŌåÆclient); `ErrorCode::AuthFailed` (already reserved) reports rejection. `credential` and `actor` are opaque byte strings the core never parses. Server-side, a pluggable `Verifier { verify(&[u8]) -> Option<Vec<u8>> }` maps a credential to an actor id (or rejects); the session enforces Hello ŌåÆ Auth ŌåÆ Subscribe and never trusts a client-asserted actor. Replaces the two-phase Hello ŌåÆ Subscribe with no actor.
**Why:** ARCHITECTURE makes `actor_id` mandatory from v0.1 and the credential carrier deployment-pluggable; keeping credential/actor opaque at the core keeps the engine identity-provider-agnostic (apps bring JWT/OIDC/custom), and deriving the actor server-side is the security invariant ŌĆö a client must not be able to assert who it is. Landed in slices: #63 wire messages first (this), then server verifier + session phase, then client `auth()`; the WS-upgrade fast path + anonymous-mode are a deferred follow-on.

## 2026-07-02 ┬Ę #61 ┬Ę multi-room multiplexing keyed on a client-assigned channel id, not a per-message room tag
**Changed:** a connection carries several room subscriptions at once, each on a connection-local `Channel(u32)` the client assigns at Subscribe. `Message::Subscribe`, `Ops`, and `Snapshot` gained a `channel`; a new `Message::Unsubscribe { channel }` frees one. The server session holds `channels: HashMap<Channel, RoomId>`; the registry fans a broadcast out to each peer on the channel that peer opened for the room. The Hub stays room-keyed ŌĆö the session translates channelŌåöroom. Replaces "a connection binds a single room; a second Subscribe switches it." `ClientSession` (the SDK-side driver) stays single-room for now, pinned to `Channel(0)`; its multi-room extension is a separate unit.
**Why:** ARCHITECTURE prescribes "logical channels multiplexed per (room, branch, zone)", so the routing key is a channel, not the room bytes. A compact per-connection id keeps op batches small and ŌĆö the deciding factor ŌĆö lets a subscription later widen from room to (room, branch, zone) with no wire change, whereas a literal room tag would force re-versioning the frame when branches/zones land.

## 2026-07-02 ┬Ę #60 ┬Ę blob-ref reserved as a `Scalar` variant, not a separate `Value` enum
**Changed:** the blob-ref value slot is `Scalar::BlobRef(BlobRef)` (value-codec tag 4), sitting alongside the other leaf values, rather than a distinct `Value = Scalar | BlobRef | ElementRef` payload type. `BlobRef { id: [u8;16], mime, size, inline: Option<Vec<u8>> }`. Slot-only: the blob store, dedup, presigned fetch, and GC still land v0.5. The element-ref value type is left unreserved (no v0.1 promise, shape under-specified).
**Why:** a blob ref merges as an LWW replace on assignment ŌĆö exactly a leaf value's semantics ŌĆö so it is the true analog of the `tx` slot reservation: one codec tag, near-zero churn, no refactor of `RegisterSet`/`MapSet`/`ListInsert`. ARCHITECTURE's scalar/blob-ref/element-ref taxonomy distinguishes *merge semantics*, which `Scalar` already models; it is not a mandate for a distinct Rust enum. Trade-off: `Scalar` is documented "no id" in the CRDT-identity sense ŌĆö `BlobRef`'s handle is opaque payload data (like `Bytes`), carries no CRDT identity, and does not merge or displace, so it still reads as a value, not an entity.

## 2026-07-02 ┬Ę rewrite ┬Ę core language is Rust, not C
**Changed:** the core and server are implemented in Rust ŌĆö a downward `Rc<RefCell<T>>` value graph, `#![forbid(unsafe_code)]`, Miri-gated ŌĆö not C. Export surface is unchanged: a stable C ABI (cbindgen) for native SDKs + wasm (wasm-bindgen) for the browser. Host seam is `entropy()` + `now()` only; `std`, not `no_std`.
**Why:** the C core's manual refcount/arena lifetime management was error-prone ("memory management was hell"); Rust removes the use-after-free / double-free / aliasing hazard class at compile time while keeping the same portability. Decided at the cheapest moment ŌĆö only the primitives existed.

## 2026-07-02 ┬Ę #48, #32 ┬Ę displacement retains + reinstates, not orphan-and-forget
**Changed:** a displaced container/counter is kept in a persistent per-id registry and reinstated if its slot is re-won; a displaced counter keeps accumulating. Replaces "the displaced element_id may become unreachable; core surfaces an orphan event."
**Why:** orphan-and-forget diverges ŌĆö two replicas that saw the same ops disagreed on a counter's value across displace-then-recreate. Retention is a convergence requirement, not a nicety. The orphan event still fires for the app; the state itself is retained.

## 2026-07-02 ┬Ę core op vs. wire envelope
**Changed:** the core op the CRDT engine merges is `{id, stamp, target, kind, tx}`. Authorship (`actor_id`), scope (`room`/`branch`/`zone`), `schema_version`, and wall time are wire/server-envelope concerns wrapping the core op ŌĆö not core op fields.
**Why:** keeps the CRDT engine pure and portable; the envelope is layered at the wire/server boundary, which is where those fields are actually consumed.

## 2026-07-02 ┬Ę #52ŌĆō#56 ┬Ę snapshot keyed on server seq; cold-start is snapshot-or-delta, no tail
**Changed:** a snapshot is keyed by the server sequence it covers (`base_seq`), not a lamport timestamp, and is regenerated live from the merged replica. Catch-up returns *either* an op delta (at/above the room's compaction floor) *or* a whole-replica snapshot (below it) ŌĆö never snapshot-plus-tail.
**Why:** the server sequence is what the durable log truncates on; regenerating from the live doc is always current, so a stored-snapshot-plus-tail merge is unnecessary. *Revisit* (KANBAN): O(state) CPU per below-floor cold-start ŌĆö cache the encoded snapshot if snapshots grow.

## 2026-07-02 ┬Ę #33, #40 ┬Ę one custom binary codec, shared by wire and log
**Changed:** a single deterministic little-endian, length-framed, total-decode codec ŌĆö not CBOR/MessagePack ŌĆö reused for both the durable op-log and the wire envelope. The 8-byte header (`"CRDT"` magic + version) reserves the version for future codec negotiation.
**Why:** a closed op enum doesn't benefit from a self-describing format; one codec is one code path and one test surface. Negotiation stays possible via the reserved version field.

## 2026-07-02 ┬Ę #47, #56 ┬Ę persistence is a per-room append-only file log, not SQLite
**Changed:** the store is `<room>.log` (length-framed ops) + `<room>.snap` (compaction snapshot); crash-safety is hand-rolled (append flushes before return; compaction is temp ŌåÆ fsync ŌåÆ rename ŌåÆ dir fsync ŌåÆ truncate, with dedup-on-replay). No SQLite, no relational tables.
**Why:** the op stream is pure append-only sequential ŌĆö relational storage buys nothing on the hot path and drops a dependency. *Revisit* (KANBAN): the metadata/query side (admin UI, op-log viewer, audit) may still want an embedded DB (SQLite/redb).

## 2026-07-02 ┬Ę element_id derives from (parent, key, kind)
**Changed:** the derivation tuple includes the element kind, so a type-flip on a slot yields a different element_id. (ARCHITECTURE previously wrote the tuple as `(parent_id, key)`.)
**Why:** two differently-typed elements at the same slot are genuinely different elements; the kind in the tuple drives the displacement/orphan path correctly.
