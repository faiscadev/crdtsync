# Design decisions

Log of design changes to [ARCHITECTURE.md](ARCHITECTURE.md) that implementation forced. ARCHITECTURE always reads as the current intended end-state; this file is *why* it changed. Newest first.

`/cs-implement` appends an entry when building a unit reveals a genuine design change — a forced realization (the design as written is wrong, impossible, or clearly worse than what the code must do), never a mere preference. The human appends when they revise scope. Format: **date · unit/PR · what changed · why**.

The entries below (2026-07-02) are a backfill: design changes made during the v0.1→v0.2 build that predate this log, recovered from the sessions and commit history.


## 2026-07-07 · Doc-level ACL slice 1 · the tuple CRDT set mirrors RangedElement 3a — storage-only, grantor as an explicit op field
**Changed:** realizes ARCHITECTURE §Ownership's doc-level ACL tuple as CRDT-merged document state. First slice of the Doc-level ACL-as-CRDT gate; records two cuts, no scope change.
- **The tuple set is structurally a RangedElement set with an ACL-tuple payload.** A document-level CRDT set keyed by a stamp-derived id under a fixed `crdtsync\0acl` namespace (mirroring ranged's `crdtsync\0ranged`), held on `Document` beside `ranged`, with `AclGrant`/`AclRevoke` ops (codec tags 19/20 after the ranged ops), a state-codec section id-sorted after the ranged set, and the same laws: concurrent grants union to distinct ids, a revoke tombstones (retained, delete-wins) and buffers until its grant, a local revoke of an unseen id emits nothing. The only shape difference from ranged: a tuple is **immutable** (no `set_payload`/LWW) — create + revoke are its whole lifecycle — so the entry needs no per-field stamp.
- **`grantor` is an explicit op field the caller sets, not derived from envelope authorship.** ARCHITECTURE §Ownership calls the grantor "the tuple's author … un-forgeable, since the op carries `actor_id`", but the core op deliberately carries only `{id, stamp, target, kind, tx}` (2026-07-02 · core op vs. wire envelope) — `actor_id` is a wire/server-envelope concern. So the ACL op carries `grantor: ClientId` as an explicit field the client/server sets to the authenticated actor's id; core stores it faithfully. The un-forgeability §Ownership describes is the **server's** job (it gates who may emit a grant and stamps the true grantor), not core's.
- **Slice 1 is storage + merge only — the merge is content-neutral.** Core stores any tuple that arrives and merges the set; it performs no authority check — no provenance-based revocation (who *may* revoke), no delegation/attenuation, no deny-precedence, no evaluation of a decision. Those are the evaluator/delegation slices (2–3, server-side). This keeps the CRDT engine pure: it is a grow-only set of immutable tuples + tombstones, convergent on its own, with the policy layered above at the server boundary. STATE_VERSION 7→8 (break format freely, pre-release — no migration).

## 2026-07-07 · SDK Unit 6a-ii · path façade addresses xml children — the children sequence is index-addressed off the element's path; no grandchild navigation
**Changed:** realizes ARCHITECTURE §Portable CRDT Core / §Multi-Language Support (the SDKs forward the `core::path` façade). Second slice of Unit 6, extends 6a-i. No scope change.
- **An element's/fragment's children are addressed by naming the element then an index — not by extending the path with a child key.** Unlike an attr (6a-i, addressed by a further path key into the attrs Map), a child of the children List has no map key: it is placed in a sequence and index-addressed, exactly as `list_insert(doc,path,index,val)` addresses a list item. So the new fns take the element `elem_path` plus an `index`: `xml_insert_element` / `xml_insert_text` (born empty then filled in the same transaction) / `xml_child_delete` / `xml_children_len`. Both XmlElement and XmlFragment own a children List, so all four accept either.
- **The write side resolves the element from its holding map, not from the leaf slot.** The path names the slot *holding* the element (not the element's own map, as the attrs case landed on). So the descent lands on the holding map and a new `MapCursor::xml_children(key)` reads that slot — an `Element::XmlElement` or `Element::XmlFragment` — and yields an `XmlChildrenCursor` over its children List id (`XmlElement::children_id` / `XmlFragment::children_id`). Mirrors 6a-i's `xml_attrs_id` seam: resolve the target container off the live slot rather than re-deriving it. A precheck (`slot_ok(is_xml_node)`) makes a write on a non-element/fragment target inert (empty ops), and `xml_child_delete` additionally prechecks the index is a live child so a no-op delete never materialises or re-stamps the List (the `list_delete` no-materialise contract).
- **No grandchild navigation — a fresh child has no stable path key.** An inserted child is index-addressed and its index shifts under sibling edits, so it has no path key of its own; this slice addresses the children List (insert/delete/len at the element level), not a nested child's attrs/children. In-place edit *into* an existing composite child stays deferred (5d-i established there is no in-place xml-child edit cursor). Move/marks are later slices (6a-iii/6a-iv).

## 2026-07-07 · SDK Unit 6a-i · path façade descends into XmlElement attrs — an element's attrs are addressed as a keyed sub-namespace, dead ends zero-op
**Changed:** realizes ARCHITECTURE §Portable CRDT Core / §Multi-Language Support (the SDKs forward the `core::path` façade). First slice of Unit 6. Records two forced realizations; no scope change.
- **An XmlElement's attrs are addressed by extending the path, not by a new attr op.** `core::path` resolved only nested Maps by key; an XmlElement in a slot stopped resolution. The fix teaches both the read walk (`resolve_map`) and the write descent (`MapCursor::child`/`into_child`, contextual via `xml_attrs_id`) to descend a further key into an element's attrs Map — an element's attrs *are* an ordinary keyed Map, so the existing `register`/`get_int`/`set_bytes`/… fns reach an attr by naming the element then the attr key. No attr-specific op, no path-model change — the façade stays key-framed. A fragment carries no attrs, so a key past it is unresolved (read) / a dead end (write).
- **A dead-end write must zero-op at any depth — the emptiness of the returned op vec is a caller signal.** The path fns return the ops to broadcast; a caller keys "did my write land / needs broadcast" off op-vec emptiness (the `list_delete` no-materialize precedent). So a write past a fragment must emit nothing — at ANY depth, not just when the fragment is the first parent. A single-walk descent that emits create-through `MapCreate`s as it goes and only discovers a deeper fragment mid-walk would emit phantom ancestor maps before aborting, breaking the contract (a self-review caught exactly this: the shallow `[frag,x]` case emitted zero ops but `[outer,frag,x]` emitted one). The resolution is a read-only `writable(doc, parents)` pre-check that walks the parents and refuses before any emit — the same slot-check-then-emit shape `list_delete`/`list_delete_id` already use. The pre-walk is deliberate (a deep dead-end needs look-ahead before the first emit); it is not a redundant second walk to be optimized away.

## 2026-07-07 · XmlElement Unit 5d-ii · marks value diff — a set-diff over the RangedElement annotation set, folded into `diff()` after the tree walk
**Changed:** realizes ARCHITECTURE §Schema-Aware-Diff ("attrs / marks / Map / Register / Counter show value diffs") — the annotation half of Unit 5d, completing 5d and **Unit 5**. Records the variant shape + the merge-join + the scalar-only deferral; no scope change.
- **Marks get their own path-less `Change` variants, not a reuse of the path-addressed tree ones.** A mark annotates a *sequence* by anchors (`RangeAnchor { seq, pos }`), not a tree slot — it has no path. So three new variants — `MarkAdded { id, seq, name, value }`, `MarkRemoved { id, seq, name, value }`, `MarkChanged { id, seq, name, old, new }` — each carrying the mark's stable `id`, the `seq` element its anchors resolve against, the `name`, and the scalar value(s). `diff()` folds `diff_marks` in **after** the tree walk (still one `Vec<Change>`): tree changes first, then the mark set, each deterministic on its own, no cross-dependency (a mark's `seq` may name a tree element also reported — that's fine, the consumer applies both).
- **Set-diff by stable id via a merge-join, not a hash join.** `ranged_elements()` returns entries id-sorted, so `diff_marks` walks the two id-sorted mark slices with a merge-join (`i`/`j` indices, `Ordering` match, trailing drains): `Less` (old id absent from new) = removed, `Greater` (new id absent from old) = added, `Equal` with a differing scalar = changed. Total-ordered by `id.as_bytes()`, so two replicas that merged the same ops produce the identical ordered mark-diff regardless of `ranged_elements()`'s internal iteration order. A removed mark pulls its `value`/`seq`/`name` from the **old** snapshot (it's gone from new — a `tx.ranged().delete` tombstones it out of `ranged_elements()`); a `set_payload` is an in-both value change; a re-set to the same value is silent (consistent with the register `Value` diff). The first cut of `diff_marks` used two `HashMap`s + a re-sort of already-sorted input — review-caught as over-engineered and refactored to the merge-join.
- **Scalar-payload marks only; a composite (object) mark is a correct deferral, not a dropped change.** `mark_views` filters `ranged_elements()` to `RangedElement { payload: RangedPayload::Scalar(value), name: Some(name), .. }`. A composite-payload mark (object flavor, a nested Map body) is skipped — but it is **unconstructible via the public API**: `tx.ranged().mark()` takes a `Scalar` only, and `create_map/list/text` pass `name: None` (a non-mark composite annotation). So there is no reachable object mark to diff; deep-diffing a composite mark body is the same virtual-container reachability model deferred elsewhere, and lands with the object-mark construction ergonomics (folded into Unit 6). A plain `name: None` annotation is not a mark and is correctly excluded.
- **wasm `change_to_js` must grow with the enum — the exhaustive match is the guard.** The wasm crate is workspace-excluded but CI's `wasm-sdk` job builds it; its `change_to_js` matches `Change` with no wildcard arm (deliberate — a new variant becomes a compile error, not a silent skip). The three mark variants broke that compile until `markAdd`/`markRemove`/`markChange` arms + a `set_mark_head` helper (id/seq as `Uint8Array` from `as_bytes()`, name as bytes) were added. Review-caught as a CONFIRMED break; the fix is the reason the no-wildcard match exists.

## 2026-07-07 · XmlElement Unit 5d-i · XmlElement/XmlFragment structural diff — the tree half of schema-aware diff
**Changed:** realizes ARCHITECTURE §Schema-Aware-Diff ("XmlElement subtrees produce structural diffs; attrs … show value diffs"). First slice of Unit 5d. Records the integration + a scoping realization; no scope change.
- **The xml diff is pure wiring over the existing walk — no new `Change` variant, no path-model change.** `diff_elem` gained `(XmlElement, XmlElement)` and `(XmlFragment, XmlFragment)` arms. An XmlElement's attrs are a keyed Map → `diff_map` (value diffs at the attr's key path); its children are a sequence → `diff_list` (structural `ListInsert`/`ListDelete` by stable node id, index carried in the change, `SeqItem::Composite(XmlElement|Text)`). A fragment is its children alone. This fixes a latent `unreachable!` **panic**: two XmlElements fell through `diff_elem`'s `_` arm into `scalar_of`, so diffing *any* xml-containing document panicked — no xml diff existed before.
- **A changed tag is a replace, not a field diff.** A tag is part of an element's identity (`XmlElement::node_id` is `(map_id, key, tag)`-seeded), so a different tag at the same slot is a different element. The xml arm compares tags first and emits `Removed{XmlElement}` + `Added{XmlElement}` when they differ, rather than diffing two unrelated elements' attrs/children. (A slot that swaps element↔fragment is already a kind mismatch, handled by `diff_elem`'s top guard.)
- **Sequence-composite survivor recursion (and the index-stepped path it needs) is DEFERRED — it has no drivable state under the current edit API.** The instinct was to also recurse `diff_list` into a *surviving* composite child so an in-place edit of a nested element/text surfaces at an index path (aligning the diff path with `validate::Step::Key|Index`). But the public tx/cursor API exposes **no way to edit an existing children-list child across a snapshot boundary**: `XmlChildrenCursor` only inserts/deletes (a child's id is stamp-seeded, not re-derivable by key), and a plain List holds only immutable scalars. So a surviving composite whose *contents* change is unreachable — there is nothing to diff and nothing to test. Every *reachable* xml change is either an attr value diff (the element itself survives at a map slot, already recursed via `diff_elem`) or a child structural insert/delete (`diff_list`, index in the change, no path step). Adding survivor recursion + the typed-step path now would be untested speculative code; it belongs with the unit that introduces xml-child in-place editing. Review-caught: `diff_list`'s long-standing doc-comment falsely claimed a live-in-both composite's inner content "surfaces at that item's path" — vacuous before (plain lists hold only immutable scalars) but newly misleading now that xml children (which can hold composites) route through `diff_list`; corrected the comment to state the surviving composite is not descended, with the recursion noted as the follow-up. 5d-ii (the marks half — mark value diffs over the annotation set) is the remaining 5d slice.

## 2026-07-07 · XmlElement Unit 5c-iii-b · per-type cardinality drop — an over-cap child of an allowed type drops, keeping the lamport-oldest
**Changed:** realizes ARCHITECTURE §Schema repair table ("Exclusive collision → keep lamport-oldest", `children.<T>.max`) — the enforcement half of 5c-iii, now that 5c-iii-a parses the cap. Records the integration; no scope change. Completes 5c-iii and the 5c structural-repair arc (disallowed-child drop · orphan-inline wrap · cardinality drop).
- **A cardinality excess reuses the drop path, not a bespoke repair.** For each child type `T` declared with a `max`, `queue_xml_children` counts the element's children resolving to `T`; the lamport-newest instances beyond `max` are the excess, each reported at its sequence position as new `ViolationKind::ExcessChild { max }` → `RepairKind::Dropped` + `RepairId::Drop { path }`. Structurally identical to `DisallowedChild` (a child that drops from the read), so it joins that arm in `keyed_repairs` — no new `RepairKind`, no element resolution (a drop reads nothing). An excess child is not descended, exactly like a disallowed one: a dropped subtree's own violations vanish with it.
- **Keep-oldest by stamp, the same order a sequence `max` uses.** The survivors are the `max` lamport-oldest of the covering set; the newest drop. Child stamps come from the children `List` (`node_ids`, index-aligned with `values()`), total-ordered by `(lamport, client)`, so replicas that merged the same ops keep the same instances — an exclusive type (`max` 1) deterministically collapses to one. This mirrors the sequence-`max` `survivors` logic (repair.rs), applied per child type instead of over the whole sequence; `max` 0 (allowed but no instances) drops all.
- **Excess is orthogonal to disallowed/orphan.** Only children that *resolve* to an allowed type (`resolve_child_type` = `Some`) participate in a cap; unresolved children stay `DisallowedChild`/`OrphanInline` as before. `tests/xml_schema.rs` +8 (conforming single, second reported, drop-is-by-stamp-not-position, unbounded-never-excesses, all-but-oldest-of-three, reads-dropped, zero-cap-drops-all, replica-convergence). Read-only, deterministic, Miri-clean. The `RepairId` `Step::Index` churn note (5b-i/5c-i) applies unchanged — a sequence-positioned drop, same deferred node-stamp follow-up.

## 2026-07-07 · XmlElement Unit 5c-iii-a · per-child-type `max` parse — the `children` allowlist becomes an object keyed by type
**Changed:** realizes ARCHITECTURE §Schema repair table ("Exclusive collision → keep lamport-oldest", declared by **`children.<T>.max`**) — the schema surface the exclusivity repair (5c-iii-b) reads. Parse-only, enforcement-inert; first slice of 5c-iii. No scope change (the repair table already pinned the declaration home).
- **"Exclusive" is a per-child-type cardinality cap, not a separate flag.** ARCHITECTURE declares exclusivity via `children.<T>.max` (e.g. `children.Heading.max = 1` = "at most one heading"), reusing the same keep-oldest/drop-newest mechanism as a sequence `max`. So no `exclusive` boolean is added; the `children` allowlist instead carries an optional per-type `max`.
- **`children` (xml) changes from an array to an object.** 5a parsed an xml type's `children` as `["Span", "Para"]` (a plain `Vec<String>` allowlist, via `parse_type_name_array`). To hang a per-type `max` off each entry it becomes an **object** `{ "Span": {}, "Heading": { "max": 1 } }` → `TypeDef::Xml.children: Vec<(String, Option<u64>)>` (type name → optional cap; empty object = allowed, uncapped). A new `parse_xml_children` mirrors `parse_children`/`parse_attrs` (object iteration, `reject_unknown_fields` per child value allowing only `max`, a negative `max` → `BadRange`). `marks` stays an array (no per-mark constraint). Break the schema format freely (pre-release).
- **The rename ripples to every child-name iterator, mechanically.** `resolve_child_type`/`queue_xml_children` (validate) and `validate_references` (schema closure) iterated the child type *names*; they now iterate `(name, _)` and are otherwise unchanged. The `max` is stored and closure-checked's reference target (`require(ty)` per child) is untouched; **no enforcement reads `max` yet** — the cardinality drop is 5c-iii-b. `tests/schema.rs` +3 (`max` parsed, negative rejected, unknown field under a child rejected) and the negative-child tests repurposed to the object form; all xml test schemas across `schema`/`xml_schema`/`xml_marks` converted array→object.

## 2026-07-07 · XmlElement Unit 5c-ii · orphan-inline wrap — loose inline text reads wrapped in a synthesized block with a derived id
**Changed:** realizes ARCHITECTURE §Invariant Repair ("Orphan inline → wrap in declared default block" + "Element-creating repairs use derived ids") — the one repair that introduces structure. Records the integration; no scope change.
- **The wrap is *described*, not materialized — consistent with the whole (observation-only) repair system.** Core's repair layer never materializes a repaired read: `repairs()` returns descriptions (`Clamped{value}` a number, `Truncated{keep}` the surviving indices, `Dropped` a marker) that the SDK applies. So the orphan wrap follows suit: loose inline text under a type declaring `repair.orphanInline` reads as a new `RepairKind::Wrapped { block, id }` — the declared block type name + the **derived** wrapper id — which the consumer builds. No materialized tree is produced in core (there is no repaired-read projection for *any* repair kind; that is a cross-cutting piece, not 5c-ii's). This is the key realization: the element-creating repair fits the existing path-based Violation/Repair model as a *description carrying the derived id*, exactly as `Truncated` carries `keep` indices — it does not need a bespoke materialization.
- **The wrapper id derives from the orphan's own element id, so it is position-stable.** `ElementId::derive(orphan.id(), block_bytes, XmlElement)` — the orphan Text's element id is already a stamp-seeded derived id (`xml_child_id`), position-independent, so the wrapper id is deterministic across replicas (`the_orphan_wrapper_id_is_deterministic_across_replicas`) and a later op can target it. Because the id, not the sequence position, is the identity, `RepairId::Wrap { id }` keys onRepaired by that id alone — it does **not** carry the `Step::Index`, so unlike `Drop`/`Clamp`/`Keep` it is immune to the preceding-insert index churn (the deferred node-stamp follow-up those still need).
- **Validate distinguishes orphan text from a disallowed child at the same "unresolved child" branch.** `queue_xml_children` (5c-i) reported every unresolved child `DisallowedChild`; it now splits: an unresolved **text** child under a type with `orphan_inline` declared → new `ViolationKind::OrphanInline { block }` (carries the declared block name), everything else → `DisallowedChild`. The orphan is still not descended (its own type is the synthesized wrapper's concern). `tests/xml_schema.rs` +3 (reported, reads-wrapped-in-the-derived-block, deterministic-id); the 5c-i orphan carve-out tests still hold (orphan text is not a `DisallowedChild`/`Dropped`, a disallowed *element* child under an orphan-inline type still drops). Read-only, deterministic. Exclusive-collision demote (needs a schema `exclusive` flag) is 5c-iii.

## 2026-07-07 · XmlElement Unit 5c-i · disallowed-child drop — an xml child whose tag no type allows drops from the read
**Changed:** realizes ARCHITECTURE §Invariant Repair ("Disallowed child → drop") + §Schema repair table — the first of the three structural rules, over the xml children sequence. Records the integration; no scope change.
- **The "not-descended" child of 5b-i becomes a reported, droppable violation.** 5b-i's `queue_xml_children` left a child that `resolve_child_type` couldn't place (its tag in no allowed child type) silently un-descended. It now emits a `ViolationKind::DisallowedChild` at the child's sequence position via a `Work::Report` (the same tree-order stack item attrs use), still without descending it — a dropped child's subtree is not validated, so its own nested violations vanish with it. The child resolves exactly as before when its tag *does* match; only the `None` branch changed from skip to report.
- **Structurally identical to the attr drop (5b-i), so it reuses that path whole.** `DisallowedChild` maps to `RepairKind::Dropped` + `RepairId::Drop { path }` in `keyed_repairs` — the reading (absent) is identified by location, and a children position is a `Step::Index` the existing `element_at` xml traversal already handles. No new `RepairKind`, no element resolution (a drop reads nothing). The `RepairId` `Step::Index` churn note (5b-i) applies unchanged — a sequence-positioned drop, same deferred node-stamp follow-up.
- **Shared by elements and fragments.** `queue_xml_children` serves both the XmlElement and XmlFragment arms, so a disallowed child under either is reported; emits in sequence order (LIFO over the reversed push). `tests/xml_schema.rs` +7. Read-only, deterministic. Orphan-inline wrap (element-creating) is 5c-ii; exclusive-collision demote (needs a schema `exclusive` flag) is 5c-iii.
- **Review-caught (CONFIRMED correctness):** dropping *every* unresolved child preempted 5c-ii's orphan-inline **wrap**. A type declaring `repair.orphanInline` (parse-valid since 5a) with loose inline text under it means "wrap this text in that block," i.e. **preserve** it — but 5c-i's blanket drop reported it `DisallowedChild` → `Dropped` the moment such a schema was bound, before 5c-ii lands, losing content the schema asked to keep. Fixed by carving out the orphan case: `queue_xml_children` now also takes the type's `orphan_inline`, and a child that resolves to no type is left **untouched** (not reported, as pre-5c-i) when it is inline **text** and the type declares `orphan_inline` — deferred to 5c-ii's wrap. The carve-out is text-only: a disallowed *element* child under an orphan-inline type still drops (orphan-inline is about inline text, not stray blocks). Regressions `loose_inline_text_under_an_orphan_inline_type_is_not_dropped` + `a_disallowed_element_child_under_an_orphan_inline_type_still_drops`. Also fixed `RepairId::Drop`'s doc, which still named only attributes.

## 2026-07-07 · XmlElement Unit 5b-ii · mark-on-disallowed-type filter — a mark reads active only when the containing XmlElement's type declares it
**Changed:** realizes ARCHITECTURE §Schema repair table ("mark on disallowed type → drop") — the read-time complement to 5b-i's attr drop, over the mark read model (4b). Records the integration; no scope change.
- **The filter lives in `marks_at` alone — it is a read convention, not a Violation/Repair.** A mark whose name is outside the containing element type's `marks` allowlist is dropped from the `marks_at` read; the `RangedElement` stays in state and every raw annotation read (`ranged_on`/`ranged_element`) is unfiltered. The path-based Violation/Repair model (5b-i) can't express it — a mark is a whole-`RangedElement` annotation targeting root, not an element at a tree path, so there is no location to key an `onRepaired` observation on. So no new `ViolationKind`/`RepairKind`; the drop is a `continue` in `marks_at`'s grouping loop, symmetric with the boolean-off omission already there.
- **The allowlist resolves by the *same contextual top-down walk* `validate` uses, not by the `parents` up-walk.** New `validate::marks_allowlist(doc, schema, seq) -> Option<&[String]>` descends from the root — map slot → child type, list item → `items`, xml child → tag-matched `resolve_child_type` — and returns the `marks` of the XmlElement that *directly holds* `seq`. The up-walk alternative (`parents[seq] → children-list → XmlElement`, then resolve that element's type) was rejected: recovering the element's type still needs the contextual root→element resolution (the same tag can be two types under different parents), and the map-slot hop the walk must cross doesn't store the slot key in `parents`. Reusing validate's descent (its now-`pub(crate)` `resolve_child_type`/`expected_kind`) keeps the one contextual-resolution path — the filter can't drift from what validate enforces.
- **No XmlElement container → no allowlist → keep every mark.** `marks_allowlist` returns `None` (not "drop all") when `seq` sits in a map slot, a plain list, or a tagless fragment. This is load-bearing: every existing `tests/marks.rs` body is a top-level map-slot Text, so the filter must be a no-op there. A **fragment** (`tag: None`) also returns `None` — a bare document-root container carries no mark concept, so reading its empty `marks` as "forbid everything" would silently strip all inline marks under a fragment; only an XmlElement (`tag: Some`) restricts marks. (If a later unit wants fragments to forbid marks, it makes that explicit.)
- **Resolved lazily, at most once per call.** The allowlist depends only on `seq` (the enclosing element), not on `index` or any mark, so `marks_at` resolves it on the first named mark and reuses it — a `RangedElement` set with no marks pays nothing for the tree walk, and a set with marks walks once. `tests/xml_marks.rs` (7): drop-disallowed, keep-allowed, mixed (only allowed survives), keep-on-plain-text, keep-under-fragment, keep-under-disallowed-tag-element, raw-set-unfiltered. The deferred `RepairId` node-stamp key (5b-i) stays deferred — a mark drop carries no `RepairId` at all, so it is untouched here.
- **Review dispositions (high `/code-review`):** (1) *A mark under a disallowed-tag element keeps every mark* — the type walk descends a child only when `resolve_child_type` matches its tag, so a mis-nested element (tag outside its parent's `children`) is never reached and its text has no allowlist. **Intended, and consistent with 5b-i:** the validator likewise does not descend a disallowed child (its own attrs aren't checked either); the disallowed element is structural repair's concern (5c), which drops it — its text, and the marks on it, go with it. Pinned by `a_mark_under_a_disallowed_tag_element_is_kept`. (2) *An unreachable container keeps every mark* — during a reorder window (the mark + text arrived, the parent element's op has not) or after the element's removal, the walk finds no XmlElement over `seq` and returns `None` → keep-all. **Intended:** matches 4b's accept-and-store (a mark resolves against what has arrived), and drop-all would be the worse default — it would hide the marks the type *does* allow; the state is transient and self-heals when the container op lands. (3) *Per-index `marks_at` re-walks the tree* — a per-character render calls `marks_at` N times, each an O(tree) resolve; the lazy gate above zeroes it for mark-free reads, but a mark-heavy render still pays O(N·tree). This is the same per-index-vs-batch tradeoff 4b already recorded (per-index `marks_at` is O(N·M·seq) at render); the fix is the deferred batch `marks_on(seq)` that resolves the allowlist and buckets every span once — the per-index query stays the semantic contract. (4) *`marks_allowlist` re-implements the validator's descent* — the divergence-prone rules (slot→type, tag-match, kind-map) are the *shared* `resolve_child_type`/`expected_kind`; only the ~30-line walk skeleton is duplicated. Unifying it into one contextual `id → type` resolver is deferred to 5c, which needs the same primitive (disallowed-child drop resolves a child's type the same way) — extracting it now, before its second caller exists, would guess the wrong shape.

## 2026-07-06 · XmlElement Unit 5b-i · attrs schema enforcement — the validator descends xml attrs + children, disallowed/mistyped attr drops at read

## 2026-07-05 · Auto-version Unit 4 · schedule triggers `every: <duration>` — sweep-driven, arm-then-fire, no catch-up burst
**Changed:** realizes ARCHITECTURE §Auto-Version Triggers' "`every: <duration>` is a schedule, driven by the `Clock` seam + periodic sweep already used for the awareness grace window"; no scope change. Completes the auto-version trigger vocabulary (event triggers were Unit 3/3b).

- **At most one capture per sweep — no catch-up burst.** When a schedule is due, last-fire is set to `now` (not advanced by `+= interval`), so a long gap between sweeps (a paused or slow server) yields exactly one capture on the next sweep, not one per missed interval. The fire decision **reads the last-fire map at sweep start and defers stamping** to after the loop, so two schedules sharing a key (same interval + name) do not shadow each other — both are considered and the tighter `keep` governs, exactly as the event-side name-collision rule. A **backward clock step re-arms** (`now < last` stamps `now` without firing) rather than stalling the schedule for the whole regression. The scan runs **unconditionally each sweep** (bounded by live bindings, cached schema parses) rather than behind the `is_armed` latch: arming happens *inside* `parsed_schema`, so a room upgraded to a schedule-bearing version under an injected awareness policy (which skips the other schema-resolving sweep pass) is armed only by this scan itself — gating it on `is_armed` stranded that schedule. ## 2026-07-05 · Auto-version Unit 3b · retention window `keep` — durable per-version provenance + a monotonic capture ordinal
**Changed:** realizes ARCHITECTURE §Auto-Version Triggers' "`keep: N` prunes the oldest auto-versions of that trigger (the retention-window mechanism)", the piece split off Unit 3; no scope change. Closes the retention that three heuristics failed to do correctly in Unit 3's review rounds.
- **Retention is provenance-based, not name- or memory-based — it is the only sound design.** Each `Version` gains an `origin: Option<Vec<u8>>` (the authoring trigger's identity, `None` for a manual version) and a monotonic `ordinal: u64` stamped by the hub. `Hub::retain_by_origin(room, origin, keep)` keeps the newest `keep` versions of that origin (ordered by ordinal) and deletes the rest. This fixes every heuristic's flaw at once: it never matches a manual version (origin `None`) or a differently-named trigger (distinct origin), and it orders by capture, not the wall-clock name — so a backward NTP step cannot make the newest capture sort oldest and be pruned.
- **A trigger's identity is `(event, template)`, NUL-joined.** Two triggers that render the same name under *different* events (`on: subscribe` and `on: compaction`, both `snap/${timestamp}`) keep independent windows — the Unit-3-review collision where the smaller `keep` destroyed the other's captures. Two genuinely identical triggers share one window (correct — they are the same trigger). The identity is the retention key, decoupled from the rendered name.
- **Provenance + ordinal persist, so retention survives a restart.** The `<room>.versions` store record gains a 1-byte origin present-flag (+ the origin bytes when present) and the 8-byte ordinal — a format break taken freely pre-release, no old-file migration. On load, `Hub::version_ordinal` resumes past the highest persisted ordinal, so a post-restart capture never predates a restored one. This is exactly what the abandoned in-memory tally could not do (it reset across dormancy/restart, orphaning the persisted captures) — the reason provenance had to be durable, proven by a store-reopen test.
- **`keep: 0` retains nothing → the capture is skipped (no create-then-delete churn); `keep: none` retains all (no pruning).** Retention runs on each capture; a *lowered* `keep` (or a removed trigger) takes effect on that trigger's next capture, not retroactively at a sweep — a bounded, documented lag, not unbounded growth (a trigger that stops firing stays at the `keep` it last pruned to). Manual `create_version` stays a 2-arg call (origin `None`), so its callers and every existing test are untouched; the auto path uses `create_auto_version(room, name, origin)`.
- **Renaming an auto-capture detaches it from its trigger's window.** A rename is a deliberate operator act — the version becomes curated, so `rename_version` clears its `origin`; a later capture's `keep` can never prune the preserved snapshot. Retention runs on a name collision too (`create_auto_version` returning `Ok(false)`), so the tighter `keep` still governs a shared provenance group rather than being skipped.

## 2026-07-05 · Auto-version Unit 3 · AutoVersion sink — event-driven capture (a passive recorder + post-delivery drain); `keep` retention split to its own unit
**Changed:** realizes ARCHITECTURE §Auto-Version Triggers' "auto-versioning is the first built-in sink" over the Unit 1 event bus (#191) + Unit 2 triggers (#192); no scope change. On a matching `on:` event, a room's governing schema drives `Hub::create_version` with the expanded template name. The `keep` retention window is sliced out to **Unit 3b** — it needs a store-format change (see below), so this unit is the capture mechanism.

This keeps the sink a pure observer (as the event-bus design requires) while the mutation runs at the registry's top level where `&mut Hub` is available. Trigger matching lives at the registry (not the sink) because the schema resolution it needs (`room_apps` → `parsed_schema`) is the registry's, not the sink's.

Room-scoped connect/disconnect (expand a disconnect to the conn's subscribed rooms — an autosave-on-leave) is a deferred follow-on; `every:` schedule triggers are Unit 4's (Clock + sweep), inert on the event path. The clock is read once per drain, only when a room-bearing event occurred — the op hot path emits none and never reaches the drain, so it stays clock-free.

A `draining` flag on the shared state suppresses the sink recording during the drain, so a capture's own events never re-enter the queue. A *client's* manual `VersionCreate` still fires a `version-created` trigger (it is recorded outside a drain); only auto-created versions are suppressed.

That is a coherent slice of its own (**Unit 3b**), so `keep` parses (Unit 2) but is **inert** here: every matching event captures, unbounded, like a staged/reserved feature. Shipping no retention (a documented limit) is correct; shipping a retention that silently deletes manual snapshots is not.
- **The recording sink is armed lazily in schema *resolution*, so a no-`autoVersion` deployment pays nothing per event and the arming subscribe is itself recorded.** The registry always installs the recording sink (auto-versioning is built-in, and whether a schema uses it is per-room and dynamic), but the sink records nothing until an `armed` latch is raised — set the first time `parsed_schema` resolves a schema declaring any trigger. Arming in `parsed_schema` (not, as a first cut did, in `bind_room_app`) is the fix for a subtle miss: a subscribe's schema is resolved for authorization *before* its `Subscribed` fires, so the very subscribe that arms the latch is still recorded — a room already populated at a fresh server's first subscribe (or a relay-populated room's first enforcing subscriber) captures on that first join, not only the second. Until armed, `on_event` is a cheap flag check with no allocation, honoring the event bus's "no sink, no cost."
- **A capture on a room with no state, or a persist failure, is best-effort — never aborts the delivery.** A subscribe to a not-yet-materialized room (subscribe does not create the hub room; only an ingest does) has nothing to version — `create_version` returns `Ok(false)`, skipped. A durable-store persist failure (`Err`) is logged to stderr (the operator wanted a snapshot that was not captured) but never propagates — auto-versioning is a server-internal observer with no client requester to surface an error to, unlike the client-driven `VersionCreate` path. **Deferred:** the room→governing-app binding (`room_apps`) is in-memory (rebuilt on subscribe), so like the awareness-TTL binding it does not survive a restart until a durable binding lands — an auto-version trigger resolves only for a bound room.

## 2026-07-05 · Unit 8f/#190 · cold-start snapshot migration — the op seam's key rewrites at snapshot granularity, decoupled from the current slot value
**Changed:** realizes ARCHITECTURE §Mixed-Version Sync's per-recipient translation on the *cold-start* path, closing the last untranslated seam — an admitted older joiner below the compaction floor got the materialized snapshot verbatim (undecodable at its version) while the ops seam (8c/8d) was already translated; no scope change.
- **`Document::migrate_leaf_slots(fate)` is the op seam's `OpRewrite` at snapshot granularity.** Across every map, each leaf slot (scalar / register / counter, live or tombstoned) is `Keep`t, `Drop`ped, or `Rename`d per a per-key `SlotFate`; a live container (map / list / text) is always kept unre-keyed, mirroring `Chain::translate_ops`, which carries a container-create verbatim rather than tear its keyless subtree. `translate::translate_snapshot` decodes the below-floor state, runs the fate (each key probed through the same `translate_op` the ops seam uses, so the key rewrite is identical), and re-encodes only if anything changed — a same/newer/unknown-app/unreachable joiner takes the bytes verbatim. So a snapshot-served joiner converges byte-identically with a peer served the same history as a translated op delta.
- **Counter-registry migration is decoupled from the current slot value, because a counter is retained across displacement.** A counter's tally lives in the persistent registry keyed by the id its slot key derives (`derive(map, key, Counter)`), retained there even when a scalar/register displaces the slot or the slot is tombstoned (the #48 displacement-retains design). So the migration must act on that registry entry from the *key*, independent of whether the slot still holds the counter live: a **Drop** removes the registry entry regardless of slot body (else a displaced/tombstoned tally lingers as a phantom that a later re-write of the key re-adopts, diverging from an op-served peer whose `CounterInc` was simply dropped); a **Rename** captures the tally and merges it into `derive(map, newKey, Counter)` — the shared id the renamed increments land at — regardless of slot body. An earlier revision gated this on the slot value being a live counter, which missed the displaced/tombstoned cases.
- **A renamed register re-derives its id from the new key.** A `Register` carries an id derived from its slot key, encoded into the snapshot. Moving it verbatim under a rename would keep the old-key id, diverging from an op-served peer whose renamed `RegisterSet` derives the id from the new key — so the moved register is re-keyed to `derive(map, newKey, Register)`. (A scalar carries no id; a container is never renamed; a counter re-homes through its registry — so register is the one leaf body needing an explicit id re-derivation.)
- **Rename is two-phase with an isolated tally capture, so any fate is order-independent.** Phase one takes every mover out of its slot, capturing a renamed counter's tally as an *isolated* deep-clone (not its live handle) and removing the old registry id; phase two re-homes each — a register re-keyed, a counter merged into its new id's registry entry (summing on a cross-type key collision, as the ops would), the slot pointed at the merged handle. Capturing an isolated copy keeps a chained rename (`a→c`, `c→d` in one pass) from leaking `a`'s tally through `c`'s live handle into `d`; commutative PN-merge + LWW-by-stamp at the slot make both phases independent of the traversal (HashMap) order.
- **A deleted (tombstoned) container field is carried verbatim, not migrated as a leaf — faithful container-field migration is impossible at snapshot granularity.** A create-then-delete of a container leaves a tombstone slot (`value: None`) plus the displaced container retained in the registry. The op seam carries the create verbatim (resurrecting the container at the old key) and re-keys only the delete, so an op-served joiner of a *renamed* such field holds `{oldKey: live container, newKey: tombstone}`. A snapshot cannot reproduce that: the materialized tombstone has dropped the create's stamp, so the container can't be re-instated with the right LWW stamp. Rather than mis-migrate the tombstone as a leaf (the pre-fix bug — it re-keyed the tombstone and diverged both ways), `migrate_leaf_slots` detects container identity on a tombstone (`has_container_identity` — the registry still holds the deleted container at `derive(map, key, {Map,List,Text})`) and carries the *slot body* verbatim. Crucially the **counter registry at the key's derived id still migrates** — a key can carry a deleted container (in `self.maps/…`) *and* a displaced counter (in `self.counters`, encoded in the snapshot and decoded back on the cold-start path), which are separate identities; the counter is a phantom that must prune (Drop) / re-home (Rename) regardless of the slot being carried, or a snapshot-served joiner keeps a stale tally an op-served peer (whose `CounterInc` was dropped) does not. So the container-identity skip is scoped to the slot body only; counter handling is decoupled from it (the same decoupling the fifth-review fix applied to displacement). The residual op-vs-snapshot divergence for the *container* under this create-then-delete-then-rename corner is the deferred **element-set-aware container-field migration** (the same seam 8c deferred faithful container subtree elision to); a live container was already carried verbatim and converges.
- **Known limitation (deferred, the type-scope item):** the fate is key-local and type-blind, so a key that is a container in one version and a counter in another — a counter displaced *by a container* — is skipped as a container while its shadow registry counter is left un-migrated. This is the already-deferred type-scope applicability limitation (a rename/drop applies across all map types) compounded with displacement; it needs a per-recipient type-scope filter, tracked separately.

## 2026-07-05 · Unit 8c · live fan-out translation — app-scoped, convergence-preserving, chain resolved once
**Changed:** realizes ARCHITECTURE §Mixed-Version Sync's "on fan-out, for each (op, recipient) the server composes the edge-rewrites … and sends the rewritten op" on the live broadcast seam; no scope change.
- **The source version rides the `Response`, not each broadcast op.** A broadcast is one ingest batch, so all its ops share the writer's version; `Response.broadcast_version` carries that one `Option<u32>` (set from `session.schema_version()`) rather than re-tagging every op. The fan-out reads it as the source and each recipient's `session.schema_version()` as the target. (Catch-up, whose delta is heterogeneous, reads per-op versions from the log's `StoredOp`s — that is 8d.)
- **Translation is scoped to the room's governing app on both ends.** Version numbers only compare within one app's chain. A migration chain belongs to the room's *governing* app, so the write is translated only when the *writer* speaks that app (a foreign-app connection's version is a different space), and only to recipients of that app; a foreign-app writer or recipient is left verbatim. This closes a corruption hole where a foreign-app write's version number would have driven the room's chain and inverted an op that was never authored under it.
- **A partly-translatable transaction delivers its survivors destranded — convergence over the atomic view.** A migration can drop a subset of a transaction's members (a member on a v2-only field, down to v1). The group can then never reach its `count` at that recipient, so its surviving members are **destranded**: each delivered with its `tx` tag stripped, applying standalone instead of buffering forever. Dropping them with the group (the earlier all-or-nothing choice) is a convergence bug — a representable member (a write to a field both versions model) that never reaches the recipient leaves it diverged from the correct down-projection of the writer's state, and no strand-avoidance justifies that: destranding avoids the strand *and* delivers the content. The transaction's atomic-view boundary is unavoidably lost at a recipient that cannot see the member that could not cross, but the underlying ops still merge, so state converges (the same property the core relies on — an atomic author and a non-atomic peer reach identical state). A fully-carried transaction keeps its tags and stays atomic. (Re-grouping the survivors under a fresh server-minted `count` to keep them atomic among themselves is possible but unneeded for convergence; deferred.)
- **The chain is resolved and parsed once, off the lock.** For each distinct target version among the room's same-app recipients, the migration path is resolved into a parsed `Chain` up front — the registry is locked only for that resolution, not across the fan-out — and the peer loop rewrites against the owned chains. The common case (a same-version, relay, or foreign-app recipient, or a relay write) is verbatim: no lock, no parse, and a single clone per channel.
- **A recipient the chain cannot serve drops the whole batch, fail-closed.** An unreachable (breaking-gap), gapped, or unparseable chain yields no ops for that recipient rather than a wrong op. The handshake range-check (8e) refuses an unreachable — and, since `reachable` returns an error for a gapped/unparseable chain, a broken — recipient outright before it subscribes; until then the drop is fail-closed, never a corrupt apply. **Known interim:** catch-up delivery is still untranslated (8d), so a mixed-version late-joiner's backlog and the live stream disagree until 8d lands — the immediate next slice.
- **A container-create is carried verbatim; the key-local rewrite cannot elide a subtree.** `Chain::translate_ops` never drops or key-rewrites a `MapCreate`/`ListCreate`/`TextCreate` — it emits the original op (the container-kind predicate is `OpKind::creates_container` in core, one source of truth next to the id-derivation sites; it is an *exhaustive* match with no catch-all, so a newly added container kind fails to compile until it is classified there — the predicate cannot silently drift). The rewrite is key-local, but a container's descendants (a list/text insert, a nested set) target the container's *element id* and carry no field key, so no migration step ever matches them. Dropping the create while the descendants survive strands them against a container that never arrives; rewriting the create's key repoints it away from descendants that derive their element id from the original key. Either way the subtree tears. Carrying the create as-is keeps it whole — a field the recipient's version does not model surfaces as an unknown slot its invariant repair elides, never a strand. This qualifies ARCHITECTURE §Mixed-Version Sync's "no state materialization": faithful subtree elision (dropping the whole container for a version that lacks it) needs per-recipient element-set awareness — the materialization this seam avoids — so it is deferred; the verbatim carry is the fail-safe until then.
- **A poisoned transaction's whole surviving subtree is destranded together.** Because every survivor of a poisoned group is destranded (previous bullet), a container-create *and* its in-transaction keyless descendants are all carried, each `tx`-stripped — the container arrives populated, not empty. Rescuing only the create while dropping its in-tx inserts would deliver an empty container and silently lose content the recipient can represent; delivering all survivors keeps the subtree whole. **Known interim (cross-batch):** the tear across *frames* — a container's create fanned out in one batch, an insert into it in a later batch — is out of a single batch's reach; per-op poisoning is scoped to one fan-out batch, so a container edit split across `Ops` frames is not covered until the element-set-aware refinement lands.
- **The batch is translated once per distinct target version, not once per recipient.** The rewrite depends only on the (source, target) pair, so `deliver` builds `translated_by_target: HashMap<u32, Vec<Op>>` once — one `translate_ops` per distinct target among the room's same-app recipients — and every recipient at that version reads the shared result. A same-version fleet of N peers costs one translation, not N.

## 2026-07-05 · Unit 7d/#184 · edge + chain op-rewrite composition; down folds in reverse; reachability = all-edges-back-compat
**Changed:** finalizes the composition order 7c deferred, realizing ARCHITECTURE §Mixed-Version Sync ("compose the edge-rewrites along the chain") + gate 4 (per-doc reachability); no scope change.
- **An edge composes its steps; a chain composes its edges — up forward, down in reverse.** `Migration::rewrite_up` folds the edge's steps in declaration order; `rewrite_down` is `None` for a breaking edge and otherwise folds them **last-to-first** (inverting a sequence inverts its members in reverse). `rewrite_up_along` / `rewrite_down_along` do the same one level up over a chain of edges: up walks the chain forward, down walks it in reverse. A `Drop` short-circuits at either level (a dropped op has no image downstream). An empty edge / empty chain is identity.
- **Both chain fns take the ascending, contiguous chain; direction is the function, not the argument order.** The caller (Unit 8 fan-out) supplies the edge slice between the op's creation version and the target, ascending and contiguous (`edges[i].to == edges[i+1].from`); `rewrite_up_along` walks it forward, `rewrite_down_along` in reverse. Contiguity + ascending order is a **documented precondition**, not validated — the caller resolves the path from the registry chain, so re-validating here would duplicate that and invent an error type with no caller; kept a precondition until a consumer needs otherwise.
- **Reachability = every edge back-compatible.** `reachable_down(edges)` is true iff each edge has an inverse; forward is always reachable (no `reachable_up`). It is exactly the predicate `rewrite_down_along` gates on (returns `Some` iff `reachable_down`), and the one the handshake range-check consults to refuse an unbridgeable recipient (`onUpdateRequired`) — so a down-translation only ever traverses invertible edges. `rewrite_down_along` enforces the equivalence by checking `reachable_down` **up front**, before the fold — not by relying on the reverse walk to reach the breaking edge, since a `Drop` from an upper edge would short-circuit before it does, letting a per-op result disagree with the path's reachability. This mirrors edge-level `Migration::rewrite_down`, which gates on `compat()` before threading its steps.
- **Down step-order is correct-by-construction but not yet observable.** The only back-compatible steps today (addType/addField) have inverses that commute (independent key drops), so the reverse-fold's order cannot change an outcome under the current step set; the reverse is implemented for when non-commuting back-compatible steps (e.g. value transforms) land. Tests assert the observable behavior (drops, reachability, up-order via renames on breaking-but-forward edges).

The `TypeDef` model carries no exclusivity and no disallowed-value/enum field, so those constraints are not checkable here — exclusivity stays XmlElement/marks-gated (already deferred), and **"disallowed-value drop" (listed as a Unit 2b repair) has nothing to validate against and cannot be built until a schema enum/disallowed-value field is added.** 2b is descoped to the repairs whose constraint the model expresses: scalar/counter clamp and sequence-`max` drop-newest.
- **`Violation` is the shared currency for repair.** It carries the located `path` (`Vec<Step>` of map keys / list indices) plus the offending value and the crossed bound, so 2b's read-time normalization has everything it needs without re-deriving it.
- **The walk is total on every decoded document, not just well-formed ones.** A valid tree is single-parent, so each element is reached once and every violation is reported. But a document decoded from a *crafted* snapshot can share or cycle its slot graph (decode's cycle guard only covers the separate `parents` map, not the slot-ref graph), so the walk is iterative over an explicit stack with an `ElementId` descend-once guard: an element's own constraint is checked on every reach (a shared leaf is still flagged under each slot) but its subtree is entered once, bounding work to the slot-edge count and making a cycle terminate. The tree location is an `Rc<PathNode>` parent-cons-list (extending a path is O(1); the `Vec<Step>` is materialized only when a violation is recorded) with an iterative `Drop`, so neither the walk nor the path teardown recurses off the stack at depth.

**Why:** the validator runs server-side over client-authored state, so "never panics / never overflows on any input" is a correctness requirement, not a nicety — a recursive walk or a recursive `Rc` teardown would let a deep or crafted document crash the process. Confining the checks to model-expressible constraints keeps the closure guarantee honest (every reported violation has a repair) and surfaces the missing disallowed-value field before 2b tries to build a repair with no constraint behind it.

## 2026-07-04 · Unit 3b-iv · server-side throttle is a leading-edge inbound cap, not a timer-driven trailing flush
**Changed:** no ARCHITECTURE change — the enforcement shape of `awareness.<kind>.throttle` (ARCHITECTURE §Throttle: "Client-side SDK debounces at throttle interval before sending. Server-side caps inbound — faster updates coalesce, keep latest only").
- **The server caps the outbound rate; it does not run a trailing-edge timer.** The first update, any update on an unthrottled kind, and the first update once the window has elapsed fan out at once and become the entry's stored value. An update arriving inside the window is *coalesced*: it refreshes the entry's last-seen time (so a fast stream does not TTL-expire mid-move) but does **not** replace the stored value or fan out. The stored value is therefore always what the room was last sent. **Delivering the trailing value is the client SDK's debounce, not the server's job** (§Throttle line 840 assigns debounce to the client) — so the server needs no per-entry timer, and the periodic sweep does no throttle work.
- **A joiner replays the last-broadcast value, so every peer and any joiner agree.** Because the stored value is exactly what was last fanned out (a coalesced value is dropped, not stored), a client subscribing mid-stream is replayed the same value existing peers hold — there is no window in which a joiner sees a peer's presence ahead of everyone else.
- **Fan-out excludes the originating connection (by `ConnId`), unchanged from the pre-throttle path.** An earlier revision that added a coalesced-flush path excluded the whole owning `ClientId`, which froze a second connection of the same client; removing the flush removes the divergence — a sibling connection (another tab) is a normal recipient again.

**Why:** an earlier design held the coalesced value and flushed it on the periodic sweep. But the sweep interval (default 1s) is far coarser than a cursor throttle (≈100ms), so the trailing value lagged up to a second; the held value also raced timed-TTL expiry (expire-before-flush could clear it unseen) and let a joiner replay a not-yet-broadcast value, disagreeing with existing peers. All three fall away once the server stops trying to own the trailing edge: the client debounce already delivers the final position on its next past-window send, so a server-side timer is redundant complexity. The cost — a misbehaving client that streams faster than its throttle and then stops leaves peers one window behind its true final position — is the intended defensive degradation, and is bounded by the throttle window.

## 2026-07-04 · Unit 3b-iii · a room's governing schema is a sticky per-room binding, not a per-sweep scan of live subscribers
**Changed:** no ARCHITECTURE change — the schema-resolution half of awareness timed TTL, and how a room binds to an app.
- **A room is governed by one app — the first enforcing app to bind it — for as long as the room stays live; a foreign app never seizes it.** The schema that times a room's presence out is resolved from the `{app_id, version}` bound in `room_apps`, seeded when an enforcing client subscribes and reconciled each sweep against who is present. Earlier attempts each traded one cross-app edge for another: a per-sweep scan of live subscribers left a ghost cursor once the last enforcer departed and was version-nondeterministic; a pure sticky binding let a foreign app grief-expire an incumbent; adding *successor takeover* (a present app inheriting a departed app's room) then let a foreign app seize the presence of an incumbent that had merely blipped into its grace window, and ratcheted the version to the highest ever bound. The resolved rule is the simplest safe one: **the incumbent app governs while the room is live (holds presence or has any subscriber); a foreign app subscribing is ignored** (so it can never grief-expire the incumbent, connected or grace-held). The governing version is the highest version of the incumbent seen while the room has held presence (the bound version is a floor a present higher version lifts), resetting only when the room goes dormant — so a rolling upgrade adopts the newer schema and a just-departed newer client's grace-held presence keeps its own longer TTL instead of expiring early under an older peer's. A version *downgrade* mid-presence (a rollback, or a newer version that drops a kind's TTL) is the transient mixed-version edge this trades away; per-entry version governance (each presence entry timed by the schema version that set it) is the exact fix, deferred as beyond the TTL bridge. A room with neither presence nor a subscriber is dropped, bounding the map. Binding is gated on the subscribe being *accepted* — a rejected/read-denied subscribe opens no channel and governs nothing.
- **An injected `AwarenessPolicy` is authoritative; the schema path is the default when none is injected.** The `Registry` holds `Option<Arc<dyn AwarenessPolicy>>` (was a `NoTimedTtl` default). `Some(policy)` governs expiry alone — one declaring no TTLs *suppresses* expiry — and `None` resolves TTLs from each room's schema. This makes the seam's meaning unambiguous (an injected no-TTL policy means "no expiry", not "fall through to schema") and confines the schema machinery (the room→app binding, its per-sweep reconcile and pruning) to the `None` path — so a server on an injected policy neither seeds nor grows `room_apps`.

- **A room is served by a single app; cross-app reuse is an unsupported configuration.** The above governs it *safely* — the first app wins until the room fully empties, then the next app rebinds — rather than perfectly. Perfectly governing two apps sharing one room would need per-app presence partitioning under one room, which the one-schema-per-room model does not carry; grief-resistance (a foreign or malicious app can never seize a live room's TTLs) is worth more than exactness in a config the engine does not support.
- **Parsed schemas are cached per `{app_id, version}` for the process lifetime (`schema_cache`, `Arc<Schema>`).** A registry link is immutable once locked, so a version's schema never changes — re-parsing its JSON every 1s sweep was pure waste. The cache also lets many rooms of one app share a single parsed copy (an `Arc`), not a deep clone per room.
- **The sweep skips the registry lock entirely when no room holds presence** — the idle server never contends the handshake path's mutex.

**Why:** presence lifetime is a room property, not a property of who happens to be connected this instant, so the schema that governs it must be a stable room binding — resolving it from the live connection set is both leaky (ghost cursors) and nondeterministic (version races). Caching the immutable parse keeps the per-second sweep cheap. Completes the schema→TTL bridge; unblocks 3b-iv (throttle/coalesce).

## 2026-07-03 · invariant repair (schema Unit 2) — read-time normalization; reject sequence `min` (scope, human-directed, pre-build)
**Changed:** ARCHITECTURE §Invariant Repair gains a **Mechanism** section and the **Closure** section is extended, settling the two questions the loop parked Unit 2 on:
- **Convergent-repair mechanism = read-time normalization, never a minted op.** Repair is a deterministic pure function of the merged op-set applied when materializing a read; the stored/encoded state stays the raw merged ops. The lamport order repair needs (keep-oldest, drop-newest) comes from the **stamps already in state** (Map-slot / Register / sequence-node), never the local clock — so `encode_state` is byte-identical across replicas and repaired reads converge by construction. A clamp returns the clamped value on read (stored value/stamp untouched); a disallowed value or over-`max` excess is hidden on read. No op minted → no repair-stamp to diverge. The one element-*creating* repair (orphan → wrap in default block) also mints no op — the wrapper's `element_id` is *derived* from the violating position (as Map slots derive from `(parent,key,kind)`), so every replica synthesizes the same wrapper; it ships with XmlElement/default-block. `onRepaired` fires at the **apply** boundary (the validator detects the violation there); the repaired value is produced at read. Apply-time *materialization* (rewriting stored state) is rejected — a clamp that overwrote the stored value would need a stamp and reintroduce divergence. This resolves the loop's open "derive-stamp (new `Map`/`Register` stamp accessors) vs state-normalization" choice in favor of normalization, so no new accessors are needed.
- **Reject `min` on a sequence at parse time.** A sequence *upper* bound is repairable (drop the lamport-newest excess); a *minimum count* is not — concurrent deletes underflow it and repair cannot invent items — so admitting it would break the closure invariant ("no schema admits an unrepairable runtime state"). `List.min` (the only such field: `Text` has only `max`; register/counter `min` is a numeric clamp, unaffected) is therefore rejected at schema parse. Minimum cardinality is a *semantic* constraint (structure = core, semantics = app) apps express structurally instead: a **structural floor** (required minimum as fixed Map slots + a List for the variable part above — "≥ 2 options" = `optionA`/`optionB` slots + `moreOptions` list, guaranteed by construction since a slot can't be concurrently deleted out of existence), or a **transition gate** (refuse `published = true` unless the count is met — best-effort holds at a single-actor moment). `List.min` was parsed-but-unenforced (#136), so removing it drops no shipped behavior.

**Why:** Unit 2 was gated on (1) how repair converges given it may write stamps and (2) the `min`-on-sequence closure gap (PR #139 / DECISIONS 2026-07-03). Grounding in the code resolved both: the built-primitive repairs (scalar/counter clamp, disallowed-value drop, exclusive keep-oldest, sequence-`max` drop-newest) are all pure read-time *filters* of merged state — none creates an element or rewrites a value — so "repair mints an op with the local clock and diverges" never arises; normalization is convergent by construction and needs no stamp at all. And the only unrepairable dimension, `List.min`, can't be honored under concurrency by *any* mechanism (can't invent items), so the human directed rejecting it and keeping the closure invariant literally true, having confirmed the poll-style "≥ N" use case survives via structural composition (fixed slots + list) or a publish gate. Element-creating repair and the exclusive-collision / attr / mark rules are XmlElement/marks-gated and deferred; Unit 2's built-primitive scope is validator + read-time clamp / sequence-`max` / disallowed-drop + `onRepaired`. No code yet — unblocks the Unit 2 build (and folds in the schema state/op validation moved out of 1b in #136, since the validator produces the violation set repair consumes over the same tree).


## 2026-07-03 · authorization — role membership is two-source, group subjects, provenance-bounded deny (scope, human-directed, refines #142)
**Changed:** ARCHITECTURE §Authorization (Subject Types, Role tier, Ownership tuple, Decision Flow) is refined over the model merged in #142, which framed role membership as token-only. Three additions:
- **Role membership is two-source.** A **token** role is deliberately *global* over the `app_id` (a bare token `editor` = editor of every doc — reserve token roles for genuinely app-wide authority like `admin`). A **per-doc** role is assigned in the doc-level ACL by an owner, scoped to a path ("Alice is `editor` of X"). The schema still only *defines* a role's powers; who *has* it is a token claim or a per-doc grant, never the schema. An actor's **effective roles** on a resource = token roles ∪ per-doc assignments for the actor or its groups on the resource-or-ancestor.
- **Group subjects.** The token carries group membership (`Identity { actor, groups, roles }`); a doc-level ACL tuple can target a **group**, not just an actor — assign a whole team a role/capability at once. `alice ∈ designers` (token) + `designers = editor on X` (doc-ACL) ⇒ Alice edits X. The doc-ACL tuple subject is `Actor | Group | Authenticated | Anonymous | Anyone` and its grant is `Capability(read|write|publish_awareness|own) | Role(name)`.
- **Provenance-bounded deny.** A `deny` **always** beats a static default (schema role-grant / global token role) — so an owner's `deny read alice` on doc X overrides Alice's app-wide `viewer` role. But **between doc-level grants** a deny is provenance-bounded: it overrides an allow/ownership only from the deny author's own subtree (superior carve-out), never a *peer's* or *superior's* grant. Otherwise deny would be a backdoor around provenance-based revocation (a co-owner could `deny` a peer it cannot `revoke`). A co-owner can no more deny a peer than revoke one.

**Why:** flushing the token-only membership from #142 against real use showed a bare token role is global — "Alice is editor" would make her editor of *every* document, which is wrong for the common case. The human directed: token roles stay global-on-purpose (for app-wide roles), and doc-scoped roles come from per-doc doc-ACL assignment — with **groups** as the membership indirection (token carries group membership; the doc-ACL assigns roles to groups), matching how Notion/Google-Docs/IAM scope a role to a resource and assign teams. The deny question ("can a deny override an app-wide role?") then exposed that a literal deny-wins-global would let a co-owner deny a peer, defeating the provenance-revocation guarantee (#1) — resolved by bounding a deny's reach by provenance for doc-level grants while keeping deny-beats-static for the app-wide/schema case. No code — refines the design-of-record; builds in the doc-level ACL-as-CRDT gate (KANBAN), whose tuple now carries subject ∈ {actor, group, class}, grant ∈ {capability, role}, effect, path, grantor.

- **Three authority tiers, distinct mechanisms:** **App admin** — the schema-registry authority (app owner / CI), a **superuser** over every doc in the `app_id`, above the document; a credential class (the registration key), not a role, not an owner, never in `@auth`. **Owner** — a dynamic, recursive, path-scoped capability (doc-level ACL state); creator auto-owns `/`; multiple owners per path. **Role** — static schema vocabulary + grants, membership from claims.
- **`@auth` DSL (Unit 4)** — `auth.roles` (vocabulary) + `auth.grants` (`{allow|deny, to: role/subject/${template}, on: path}`), static, authored by the app admin at CI. **Ownership is *not* in the schema** — it is entirely runtime doc-level ACL state, so the admin never writes it in stone.
- **Ownership model (dynamic capability):** a doc-level ACL tuple is `{ subject, capability ∈ {read, write, publish_awareness, own}, path, grantor }`. **Attenuated delegation** — an owner of P may write a tuple on P or any subpath (co-owner of P, owner of a subpath who re-delegates recursively, or a leaf grant); uniform rule: *may write a tuple on Q iff owns Q or an ancestor, or is app admin.* **`own` is delegable authority, leaf grants are not** (a read/write grantee cannot re-grant). **Provenance-based revocation** — a tuple is removable only by its grantor (the tuple's op author, un-forgeable) or above it in the grant chain, NOT by whoever owns an ancestor path; so co-owners of a common superior cannot revoke each other, and a superior-imposed carve-out cannot be removed by the subordinate. **Downstream deny + carve-outs** — grants and denies inherit downward, deny-wins globally (no re-open below an ancestor deny); a superior places `deny own` on a subpath to carve a subordinate's authority, and provenance-removal makes it stick.
- **Decision flow (updated):** 0 app-admin superuser → 1 explicit DENY (deny-wins) → 2 owns resource-or-ancestor → 3 explicit ALLOW leaf → 4 schema `@auth` role-grant → 5 default-deny. Permission state versioned in lamport time, so concurrent grant/revoke resolves at the op's lamport position (existing §Hard Problems mechanism).
- **Registration (Unit 5):** an HTTP **admin endpoint** (control plane, separate from the sync WS), gated by `register_schema` on `App(app_id)` with a registration credential → admin `Identity`. **Hash-lock is SHA-256 in the *server* crate**, not core — core stays dep-minimal and clients never hash-verify (they trust the server); only the un-embedded server takes `sha2`. Resolves the loop's "crypto dep into dep-minimal core" concern by placement.
- **`Action`/`Resource` extensions:** `+register_schema` action, `+App(app_id)` resource, `Room`→ path-carrying resource (with the resource's `author` for `${author_id}`). The `Authorizer` trait is untouched (already generic over `Action`/`Resource`).

**Why:** the loop (PR #139) shipped every cleanly-buildable schema slice and parked on Units 4/5 because the grant DSL, the meta-auth credential model, the `Action`/`Resource` trust extensions, and the hash-lock crypto were unspecified and security-consequential — "not mine to force." The design was settled in discussion around these constraints: (1) the schema drives authorization, so a **client-supplied schema body can't be trusted** — enforcement rests on the admin-registered, hash-locked schema (client asserts only a version key); (2) "admin" is ambiguous and conflating scopes is a hole, so it splits into **app admin** (registration superuser, above docs) vs **owner** (per-instance, inside docs) — with the human directing that ownership be **fully dynamic** (not a schema-declared allowlist), **path-scoped**, **multi-owner**, and **recursively delegable with attenuation** (an owner delegates its own path or any subpath, incl. co-ownership); (3) revocation needs to express "owner A can't revoke owner B," solved by **provenance-based revocation** (revoke what you granted, not what you sit above) — which also makes **superior carve-outs** (`deny own` on a subpath) enforceable, since the subordinate can't delete a constraint it didn't place; (4) downstream read/write deny and owner carve-outs both fall out of **path-inherit + deny-wins-global + provenance-removal** with no new primitive; (5) crypto placement in the server crate keeps core dep-minimal. No code yet — this unblocks the schema build slices in KANBAN: Unit 4 (`@auth` parse + evaluator role-grant tier), Unit 5 (Verifier→Identity claims, HTTP registration + meta-auth + hash-lock, `Action`/`Resource` extensions), then the gated 3b/6-enforcement. ## 2026-07-03 · schema layer — distribution / trust / versioning model (scope, human-directed, pre-build)
**Changed:** ARCHITECTURE §Schema is rewritten and §Schema Migration extended to pin down the schema layer's distribution, trust, and versioning model (the design gate that was blocking the awareness-TTL, `@auth`, invariant-repair, and migration units). The design is now:
- **Opt-in feature tier.** Schema-less documents are first-class (converge / persist / fan-out / snapshot / offline / room-ACL, zero ceremony); a schema is adopted only to unlock producer validation, invariant repair, migration, fine-grained `@auth`, typed API, awareness TTL/throttle, and marks/attrs/structural constraints. Adoption is never a prerequisite.
- **Schema is code, not document state.** Authored JSON, versioned, CI-gated, checked into the app repo. The document carries only the `schema_version` envelope tag, never the schema body. Schema-as-doc-state was rejected (it makes the schema concurrently mergeable → kills migration determinism, defeats the CI drift/verify gates + boot hash-lock, and creates a bootstrap cycle).
- **Distribution.** Client gets the schema via a build-time **bundle** (typed API + cold-offline) and/or a **handshake advertisement** (server → client, cached) — so bundling is optional (typed client bundles; dynamic client fetches at handshake and adopts the served version). Server gets it via **registration** (the app owner's CI pushes `{app_id, version, schema, migrations}` over an admin API), never deploy-time config — so a generic engine binary serves any tenant and a SaaS server is a per-`app_id` registry.
- **Two server tiers.** **Relay** (no registered schema) stores/dedups/fans-out/persists/snapshots + room-ACL only; clients still validate/repair locally. **Enforcing** (registered schema) adds ingress validation, authoritative repair, in-flight migration, schema `@auth`. The split exists because merge needs no schema.
- **Trust boundary.** A client-supplied schema **body is never trusted for enforcement** — the enforcing server enforces only its **registered** schema; the client asserts a version *number* used purely as a registry lookup key (unknown → reject, never fabricate). Registration is meta-authed (app-owner CI credential) + hash-locked. A client's own schema is advisory; the server re-validates every op. (Repair `f(state,schema,lamport)` converges across replicas only when they share the schema; the registered server is the arbiter.)
- **Versioning: the server is the compatibility layer; a client speaks a single version.** A typed client is generated for its build version, a dynamic client adopts the served one, and each declares on handshake the version it wants ops delivered at (normally a "range of one"; a multi-version-codegen client, rare, accepts a small range). Each migration edge is classed **back-compatible** (a bidirectional op-rewrite exists) or **breaking** (forward-only, no inverse), by the CI gates. The server **composes the per-edge op-rewrites along the chain, per recipient, on the fan-out seam** (the same seam as redaction #83) over an **immutable heterogeneous log** (each op stored at its creation version, never rewritten); cold-start migrates materialized state into the snapshot instead. The **handshake range-check guards** every down-path (a recipient across an unbridgeable breaking gap is refused before it subscribes), so fan-out only ever traverses invertible edges. Only an unbridgeable breaking gap rejects (`onUpdateRequired`). The relay/enforcing tier is decided **per `app_id`** (schema registration), not per document.
- **Three ceremony tiers, app-chosen per change:** no-schema (no migration concept) / lazy-breaking (break freely, `onUpdateRequired` handles stranded clients) / zero-downtime (expand→migrate→contract with version ranges). Same machinery; the only difference is whether an edge is made back-compatible.
- **Lifecycle hooks.** `onRepaired` / `onOpsRejected` / `onUpdateRequired` — engine detects, app decides UX (observation, not override), extending the existing repair-event + rejected-ops doctrine.

**Why:** the schema layer was the highest-leverage blocked gate — awareness timed-TTL/throttle, `@auth` defaults, the whole invariant-repair section, and schema migration were all waiting on "where does the schema live and how is it trusted," which ARCHITECTURE described only at the intent level. The forcing constraints, surfaced in design discussion, were: (1) the server must run as **multi-tenant SaaS** hosting documents from apps whose developers do not operate the server — so the operator cannot bake schemas into the deploy; registration by the tenant's CI is the only distribution that keeps a generic binary; (2) the schema drives **authorization / validation / repair / migration**, so a client-supplied body cannot be trusted — the trust anchor must be the admin-registered, hash-locked schema, with the client asserting only a version key; (3) adoption must be **frictionless** or it becomes an adoption nightmare — hence opt-in with a fully-functional schema-less relay tier; (4) real deployments need **zero-downtime rolling schema changes** (the update-to-both → change → update-to-new dance), which a scalar "reject on version mismatch" cannot express — hence client-declared version ranges + per-edge compatibility classes + per-recipient in-flight translation, with a hard reject only across a genuinely breaking gap. No code yet — this is the design-of-record that unblocks the schema build slices in KANBAN (validation engine → awareness TTL + scalar clamp + `@auth` first, since those consume only built primitives; marks/attrs/structural clauses parse but stay enforcement-inert until XmlElement/marks land).

## 2026-07-02 · tombstone GC → tombstone compression (design pivot, human-directed)
**Changed:** ARCHITECTURE §Tombstone GC (both copies) + the *Implementation Status* note flip from "remove tombstones below a `min(last-acked seq)` watermark" to "**never remove tombstones — compress them.**" A contiguous deleted run (consecutive char/list ids forming one insert's parent-chain, all tombstoned) encodes as a single range record, and a tombstone's dead value is dropped, at state-encoding time (snapshots, cold-start catch-up, the durable `.snap`). §Op Acknowledgement's `Ack { channel, seq }` frame is demoted to a **reserved, no-consumer** wire slot: its only job was to feed the GC watermark, which no longer exists. The `Accepted` frame + `ClientSession` outbox (the offline queue) are unaffected and stay built. The retention-horizon eviction policy and the load-bearing floor-coupling invariant (both watermark-GC machinery) are deleted.
**Why:** watermark-based *removal* is the wrong target. Building it surfaced that removing a positional node is only safe when no replica still references it — but clients never compact (only the server does), so a live always-connected client holds every tombstone forever and can always author an insert anchored to one. The strict "provably unreferenced" condition therefore essentially **never fires** for live clients (it degenerates to the never-GC case). Making it fire forces one of two bad mechanisms: **reparent** (removal must become a *replicated GC op* carrying a forward pointer, because peers that kept the tombstone must reparent identically — a whole new op kind + forwarding table) or **reject** (a late insert into a removed cold region is dropped and surfaced — rare, but real user-facing data loss). The human rejected both — reject for the data loss, reparent for the complexity — and directed a step back.

The reframe: the goal was never "remove tombstones," it is "**tombstones don't bloat.**" Removal is what drags in watermarks, acks, horizons, clocks, and orphan handling; compression achieves the goal with none of it. Deleted content in real editing is contiguous (words/lines/paragraphs = consecutive same-client ids on a parent-chain), so run-length range-encoding collapses a deleted region from O(deleted items) to O(runs), and a tombstone's value is never read again so it is dropped. It is a **pure encoding optimization** — identical logical state, fewer bytes — so convergence is preserved by construction (the harness's materialized-read invariant is untouched), and there is no distributed decision, no removal, and therefore no orphan problem to solve. This is precisely what shipped CRDTs do (Yjs compresses deleted items into range structs; Automerge columnar-RLEs the log); the watermark-removal path was reinventing a problem the field already routed around. First build: encoding-level run compression in the state codec (bounds snapshot/wire/disk/replay). Carrying the range representation in-memory (bounding RAM too) is a later step, deferred until RAM bloat is actually measured.

## 2026-07-02 · op-acknowledgement gate — the ack protocol behind offline queue + tombstone GC (scope, pre-build)
**Changed:** ARCHITECTURE gains §Op Acknowledgement under §Realtime Synchronization, and §Tombstone GC (both copies), §Offline-First, and the *Implementation Status* GC note now reference it. Two wire frames, one per direction, both meaning "my durable commit frontier on this channel is X": **`Accepted { channel, through }`** (server → client), the highest per-client op sequence (`OpId.seq`) of the author's own ops the server has durably logged — it drains the client outbox; **`Ack { channel, seq }`** (client → server), the server sequence the client has applied — it feeds the tombstone-GC watermark. The `ClientSession` outbox retains authored ops, `edit` appends, `Accepted` prunes `id.seq <= through`, a reconnect re-emits the unpruned tail. The GC floor is `min(last-acked seq)` over clients within a configurable **retention horizon**; a client absent past the horizon is dropped and re-synced by Snapshot if it later returns below the compaction floor. No code yet — this is the design-of-record that unblocks the two gated units.
**Why:** both features were in ARCHITECTURE at the *intent* level (§Offline-First "offline op queues"; §Tombstone GC "min watermark") but neither had a *mechanism*, and the built protocol has no acknowledgement path at all — the server never echoes an author its own ops, so today a client cannot learn a local write landed (no outbox drain) and the server learns a client's caught-up position only once, at Subscribe (no live watermark). One concept — acknowledge progress along the per-channel sequence — supplies both, in opposite directions, so it is one gate.

Three sub-decisions, each with its rejected alternative:
- **Author-ack keyed by the per-client op sequence (`OpId.seq`), not the server sequence.** The op identity `(client_id, seq)` is what idempotent ingest already dedups on and is stable across reconnect, so a resent op re-acks to the same `through` and the outbox prune is idempotent. Server-sequence correlation was rejected: a resent op is assigned a *new* server sequence on ingest, so a server-seq-keyed ack would fail to match the outbox entry after any reconnect — exactly when the outbox matters most. (Note: this is `OpId.seq`, the per-client dedup counter, not the causal Lamport `Stamp`.)
- **Retention-horizon eviction** for the GC watermark, over two alternatives. *Live-subscribers-only* (drop a client from the floor the instant it disconnects) was rejected because it guts the very offline queue this gate builds — it GC's a client's tombstone context the moment it goes offline, so a returning offline edit resurrects concurrently-deleted elements at maximum rate. *Never-GC-while-any-known-client-is-behind* was rejected because one client that dies forever stalls GC for the whole room — unbounded tombstone growth. The horizon is the bounded middle: an offline client within the window resolves its edits against still-present tombstones; past the window it re-syncs by Snapshot. None of the three causes divergence given the invariant below; the horizon is the only one that both bounds growth and preserves the offline case.
- **Floor-coupling is the correctness guard, not the eviction policy.** Load-bearing invariant: GC never removes a tombstone at or above the compaction floor, and a client reconnecting below the floor always receives a Snapshot, never a delta. This keeps every eviction policy convergent — a stale replica adopts server truth via snapshot instead of replaying past a missing tombstone. The residual cost of GC'ing a tombstone "early" is *semantic*, not consistency: an offline edit to a concurrently-deleted element resurrects it (convergently) instead of losing to the delete; the horizon bounds how often that window is hit.

## 2026-07-02 · #115 · schema-aware diff — the wasm surface + the cross-language change representation
**Changed:** `WasmDocument.diff(oldState, newState)` decodes two snapshot buffers, computes `core::diff`, and returns a JS array of plain change objects: each has `op` (string tag), `path` (Uint8Array), and its variant's fields — `kind` (string) for add/remove, `old`/`new` for value/counter, `index`+`items` for list runs, `index`+`text` for text runs. A scalar is a **tagged `{ t, v }` object** (`t` ∈ `null`/`bool`/`int`/`bytes`/`blobref`; `int` is a BigInt, `bytes`/`blobref` a Uint8Array); a list item is `{ scalar }` or `{ kind }`. It throws (a `JsError`) on a malformed snapshot. wasm calls `core::diff` directly rather than the FFI, so no ABI is involved.
**Why:** this is the first language binding to surface the diff, so it sets the **cross-language change representation** the Python/Go decoders will mirror. Objects (not the raw change-list bytes) because a JS consumer wants to switch on `op` and read fields, not re-decode a binary format — the whole point of an SDK layer. The scalar is *tagged* rather than mapped to a bare JS value because `Bytes` and a `BlobRef` are both binary and an `Int` needs full 64-bit range (a JS `number` loses precision past 2^53), so a discriminated `{ t, v }` keeps every scalar distinguishable and lossless — a bare mapping would conflate `Bytes`/`BlobRef` and silently truncate large ints. Indices cross as JS numbers (a document is far below 2^53 elements) while scalar/counter integers cross as BigInt (they are user data of full range). wasm bypasses the FFI because it is Rust and can call the core diff and encode nothing — the FFI `crdtsync_diff` (#113) exists for the C-ABI bindings (Python/Go), which decode the change-list bytes into this same object shape.

## 2026-07-02 · #113 · schema-aware diff — the FFI entry point `crdtsync_diff`
**Changed:** the C ABI gains `crdtsync_diff(old, old_len, new, new_len) -> CrdtBuf`: it borrows two snapshot buffers, decodes each to a `Document`, computes `diff`, and returns `encode_changes` of the result as an owned `CrdtBuf` the caller frees with `crdtsync_buf_free`. Malformed input or a snapshot that fails to decode yields an empty buffer; a panic is caught at the frame like every other entry point. The generated header asserts the new symbol.
**Why:** this makes the core diff (#109–#112) reachable from any C-ABI binding. It is a **stateless free function** — it takes two state blobs, not a live `CrdtDoc` handle — because the inputs a caller diffs are snapshots it already holds as bytes: a named version's `version_state`, an exported room, or a `crdtsync_doc_encode_state`; requiring a decoded handle would force the embedder to reconstruct two documents just to hand them back. It returns the *encoded* change list (not a handle to walk) so the whole diff crosses in one buffer the binding decodes with the change-list codec, matching how snapshots and op batches already cross as bytes rather than as accessor-walked handles. Empty-on-error mirrors the awareness/version framers (a bad input is an empty buffer, never a partial or a trap), keeping the boundary total. The Python/Go decoders layer over this entry point; wasm will call `core::diff` directly (it is Rust), so no ABI widening is needed there.

## 2026-07-02 · #112 · schema-aware diff — a change-list codec for the SDK boundary
**Changed:** `core::diff` gains `encode_changes(&[Change]) -> Vec<u8>` and `decode_changes(&[u8]) -> Result<Vec<Change>, DecodeError>`: a `u32` count then each change, tag-led, reusing the op-codec primitives (`put_scalar` / `put_bytes` / `Cursor`). Decoding is total — a bad tag, a short buffer, or trailing bytes is a `DecodeError`, never a panic. `ElementKind::from_tag` is added as the decode inverse of `kind as u8`.
**Why:** with the core diff complete across every built primitive (#109–#111), the next step is making it reachable from the language SDKs, and the C ABI has no way to hand back a `Vec<Change>` of mixed-payload variants directly — so the change list crosses as *one serialized buffer* the binding decodes, the same shape every other list-shaped result uses when a count+indexed-accessor would be awkward (the diff record is a tagged union, not a flat row). It deliberately is **not** a durable/versioned format (no version byte, unlike the state codec): a diff is a transient computed result recomputed on demand, never persisted, so there is nothing to migrate across releases — keeping it version-free avoids implying a stability contract it doesn't need. The FFI `crdtsync_diff` entry point and the Python/Go/wasm decoders are the follow-on slices that consume this format.

## 2026-07-02 · #111 · schema-aware diff — List element-level runs by node id
**Changed:** the List branch of `core::diff` upgrades from the coarse `Sequence` marker to element-level detail, mirroring the Text char-level work (#110): `Change` gains `ListInsert { index, items }` / `ListDelete { index, items }` where `items: Vec<SeqItem>` and `SeqItem` is `Scalar(Scalar)` (an inline value or a register's scalar) or `Composite(ElementKind)`. `diff_list` reports them by comparing the two snapshots' node ids (`Stamp`) — a node live in one and not the other is an exact insert/delete — coalescing consecutive same-op items into runs, deletes (old index) before inserts (new index). The `Sequence` variant is removed (it had no remaining producer once both sequence types diff in detail); no new read surface — `List::node_at` + `get` + `len` already give the ordered live nodes with ids.
**Why:** #109 left List coarse and #110 did Text; this closes the sequence pair so the core diff now spans every built primitive (map / register / counter / list / text). It reuses the same **stable-id, no-heuristic** principle as Text: list nodes carry Fugue `Stamp` ids, so "live in new, not in old" is precisely an insert with no alignment guess. A list item is modeled as `SeqItem` rather than the raw `Element` because `Element` is a live handle (not `PartialEq`/`Debug`, and holding an `Rc` in a value-comparable change list is wrong) — an inline scalar carries its value (what a renderer shows), a composite carries only its kind because the composite's *own* contents diff at that composite's path once it is reachable, not inside the parent list's change. XmlElement / marks / attrs diffs stay blocked on those unbuilt primitives; the SDK-facing marshalling of the change list is the next slice.

## 2026-07-02 · #110 · schema-aware diff — Text char-level runs by char id

## 2026-07-02 · #107 · snapshot export / import — the identity-preserving move; id-namespaced clone deferred

## 2026-07-02 · #102 · atomic transactions — count-tagged members, buffered until whole, applied together
**Changed:** the reserved op-envelope `tx` field, previously an unused `Option<TxId>`, becomes `Option<Tx>` where `Tx { id, count }` carries both the shared transaction id and the group size; the op codec round-trips the added count. `Document::atomic_transact(f)` mints a per-replica `TxId` (namespaced by the replica's client, so `(client, tx)` is globally unique) and tags every op the closure emits with the group's `Tx`. On the receive side, an atomic member is always buffered; a transaction commits only once every one of its `count` members is present *and* the group's external dependencies resolve, at which point all members apply together in op-seq order. A single-member atomic transaction commits immediately. The author applies its own edits eagerly, as with any local edit.
**Why:** ARCHITECTURE §Transactions specifies opt-in atomic transactions as an all-or-nothing *view* boundary ("receivers buffer member ops until commit marker arrives; on commit, all apply atomically"). The wire envelope already reserved `tx` (round-tripped since the op codec shipped) but nothing set it and no atomic behavior existed. The "commit marker" is realized as a **per-member count** rather than a separate terminator op: it makes each member self-describing, so completeness is detectable regardless of delivery order or splitting across messages, with no distinct marker to lose or order. Atomicity is enforced purely as a *buffering gate* — the same ops still merge, so convergence is unchanged by construction (an atomic author and a non-atomic peer reach identical state; verified against the convergence property harness), and the only observable difference is that no peer sees a partial group. The group commits in op-seq order so a member targeting a container an earlier member creates lands after it; the readiness gate treats intra-group creates/inserts as satisfied (they will exist once the in-order apply reaches them) and only blocks on dependencies *external* to the transaction, so a self-contained tx (create-then-populate, insert-then-delete) never deadlocks. Buffered partial transactions ride the existing op-buffer, so they persist in a snapshot and complete after a restart for free. `next_tx` is not persisted: tx ids scope only the buffering of *remote* partials (keyed by the author's client), and a restored replica mints under its own client with fresh op ids, so restarting the counter at 0 cannot collide with anything buffered. This is the core slice; the wire/server/client drive path and the SDK `atomic_transact` surface are the follow-on slices. Atomic-tx *undo* (ARCHITECTURE §Transactions "a transaction is naturally an undo intention") and scope constraints tied to branches/zones/schema remain gated on those layers.

## 2026-07-02 · #83 · per-recipient read redaction — the fan-out re-checks read on every op and awareness send
**Changed:** the registry now consults the `Authorizer` on **each** broadcast, not only at Subscribe: before queuing a room's ops (or an awareness update) to a peer it checks `authorize(peer_actor, Read, Room)`, and skips any peer that no longer passes. So a peer whose read is revoked mid-session stops receiving the room's ops and presence at once, without waiting for it to resubscribe or reconnect. Seeing a peer's awareness is treated as a read of the room, so the same gate covers both fan-outs. The write still ingests for the room's permitted readers; only the redacted peer's send is dropped.
**Why:** ARCHITECTURE §Authorization requires per-recipient filtering on every op send ("unauthorized bytes never leave the server"), which the Subscribe-time check (#82) alone does not give: authorization is dynamic (a deployment's policy can revoke access at any moment), and a check made only at join leaks every subsequent op to a since-revoked reader until it happens to resubscribe. Re-checking at the fan-out is the enforcement point ARCHITECTURE names, and it is where finer resource checks (element/zone/path) slot in later — the resource is `Room` today because that is all the wire addresses, but the hook is per-send from here on. The check is the *recipient's* read right, not the writer's; the writer's write right is already enforced at Ops ingest. Doc-level ACL-as-CRDT, cold-start snapshot redaction at finer-than-room grain, and the audit log remain the larger follow-on slices atop this hook.

## 2026-07-02 · #82 · authorization decision seam — a pluggable `Authorizer` enforced at room-level access points, `Forbidden` non-closing
**Changed:** the server gains an `authz::Authorizer` trait — the read/write analog of the credential `Verifier` — that decides whether an authenticated actor may take an `Action` (`Read`/`Write`/`PublishAwareness`) on a `Resource` (`Room` today; the enum widens to path/element/branch later). `step` consults it at Subscribe (read), Ops (write), and AwarenessSet (publish); a denial replies `ErrorCode::Forbidden` (new wire tag 5) and — unlike a protocol violation — **keeps the connection open**, since a denied request is well-formed. Injectable via `Registry::set_authorizer`; the dev-mode `PermitAll` default allows everything, so existing behavior is unchanged. The contract is default-deny (an authorizer returns `true` only for what it explicitly permits).
**Why:** ARCHITECTURE §Authorization makes authz first-class with a single policy consulted at every enforcement point, default-deny, deny-wins. The full two-tier model (schema-`@auth` defaults + doc-level CRDT ACLs), per-recipient wire redaction, zones, and the audit log are gated on the unbuilt schema layer, zones, and an ACL CRDT — so this slice lands the *seam and the room-level enforcement points that exist today*, the foundation those layers refine. `Forbidden` is non-closing (distinct from `AuthFailed`, which closes) because authorization is a per-request verdict a client should survive — it may still act within what it is allowed. Room is the only `Resource` variant now because rooms are the only resource the wire currently addresses; finer resources arrive with paths/zones. Awareness per-recipient auth-filtering stays queued: it needs the richer policy (a `see` check per recipient), not just the publish gate this adds.

## 2026-07-02 · #76 · fast-path credential carriers beyond the header, with a precedence order; mTLS out of scope
**Changed:** the upgrade fast path now reads a credential from four carriers, in precedence order — `Authorization` header, then the `crdtsync.auth.<value>` WebSocket subprotocol, then the `crdtsync_credential=<value>` cookie, then the `?credential=<value>` query param. When the client offers the plain `crdtsync` subprotocol, the server echoes it in the handshake response so the client's subprotocol negotiation succeeds. mTLS is **not** implemented — the server terminates plain TCP, with no TLS layer to expose a client certificate; it stays queued for when TLS termination lands.
**Why:** ARCHITECTURE lists these carriers as deployment-pluggable, but the forcing reality is the browser: a browser `WebSocket` cannot set the `Authorization` header, so the header-only fast path (#74) was unreachable from the web — the subprotocol and query carriers are the only ones a browser controls (cookies ride along same-origin). The precedence order is a required disambiguation the spec leaves open; header-first keeps the explicit, non-browser credential authoritative when several are present. The query carrier is supported but noted logs-leak-prone (URLs land in access logs). mTLS is deferred not by choice but by dependency: there is no TLS layer yet.

## 2026-07-02 · #75 · a real `Verifier` is injectable into the runtime, not just the dev `AllowAll`
**Changed:** `serve_with_verifier(listener, server, store, config, Box<dyn Verifier + Send>)` lets a deployment plug in its identity seam (JWT/OIDC/API key); `serve`/`serve_with` keep their signatures and default to the dev `AllowAll`. The registry actor sets the verifier on its `Registry` before serving, and it derives the actor for both the in-band Auth phase and the upgrade fast path. Until now `serve` hardcoded `AllowAll`, so a real credential check was reachable only in unit tests through `Registry::set_verifier`.
**Why:** ARCHITECTURE makes the credential verifier deployment-pluggable (the engine ships no identity provider), but the runtime gave no way to supply one — the seam existed on `Registry` yet the only entry points bypassed it. The `+ Send` bound is forced: the `Registry` lives on a dedicated actor thread, so the verifier must cross the thread boundary; `Box<dyn Verifier + Send>` upcasts to the `Registry`'s `Box<dyn Verifier>` once on the far side. Additional credential carriers (cookie, WS subprotocol, mTLS, query param) remain a follow-on; the `Authorization` header carrier plus this injection make real end-to-end auth (map + reject) exercisable.

## 2026-07-02 · #73, #74 · auth fast path — credential in the `Authorization` header, unsolicited `AuthOk`, transport-minted anonymous actor
**Changed:** the handshake gains the ARCHITECTURE fast path. A credential presented at the WebSocket upgrade is verified during accept (via `accept_hdr_async` reading the `Authorization` header — the first carrier implemented of the several ARCHITECTURE lists), and the connection opens already authenticated: `Session::authenticated` / `Registry::connect_authenticated` pre-set the actor so the client skips the in-band Auth phase. The server sends an **unsolicited `AuthOk`** (after version negotiation, before the message loop) so a fast-path or anonymous client still learns its server-derived actor without having sent Auth. Anonymous mode (`ServeConfig::anonymous`, off by default) admits a credential-less connection by minting `actor = anon:<128 random bits, hex>` from system entropy. A credential presented but refused closes the connection with `AuthFailed` before the loop. `serve` keeps its signature; policy rides on `ServeConfig`.
**Why:** ARCHITECTURE §Handshake specifies the fast path ("credentials present at upgrade → validate during accept → skip Phase 2, one round trip saved") and anonymous mode (`actor = "anon:<random>"`). The unsolicited `AuthOk` is the forced detail ARCHITECTURE leaves implicit: the client never asserts its actor, so on the fast path — where it sends no Auth — the server must volunteer the actor or the client would never know its identity. Entropy is kept at the transport layer (a `getrandom` dep on the server, not the core) to preserve the core's deliberate no-entropy stance (`Host` seam) and its Miri-gated purity; the registry stays entropy-free so its logic tests remain Miri-clean under isolation. Carriers beyond the header (cookie, subprotocol, mTLS, query param) and injecting a non-`AllowAll` verifier into `serve` remain follow-ons.

## 2026-07-02 · #72 · periodic awareness sweep wired into the runtime; timed-TTL + throttle deferred as schema-gated
**Changed:** the tokio runtime now drives `Registry::sweep` on an interval, so the reconnect-grace expiry (the session-TTL half of awareness) fires in production without an external caller — previously `sweep` was reachable only from tests. `serve` keeps its signature (5s grace, 1s sweep by default via `ServeConfig::default`); a new `serve_with(listener, server, store, ServeConfig)` exposes the grace + sweep cadence, and the registry actor selects over inbound commands and the sweep tick, flushing outboxes after each. The registry runtime enables tokio time. **Deferred, not built:** per-entry *timed* TTL (auto-expire after silence) and server-side throttle/coalesce — ARCHITECTURE §Awareness makes both **schema-declared** (an entry's type carries its TTL and throttle interval), and the schema layer is unbuilt, so their trigger values have no home yet. Building them now would either invent a wire field ARCHITECTURE doesn't sanction or hardcode a policy the schema is meant to own.
**Why:** the grace window is only useful if something actually sweeps; the runtime is where wall time and the connection set both live. Splitting `serve`/`serve_with` keeps the binary + existing callers untouched while letting tests drive expiry on a sub-second cadence instead of waiting the 5s default. Recognizing timed-TTL/throttle as schema-gated (rather than clock-gated) is the real finding: the clock seam (#71) unblocked *session* TTL, but the remaining two await schema — recorded so they are not attempted prematurely.

## 2026-07-02 · #70, #71 · awareness reconnect grace via an injected server `Clock` + explicit sweep, not clear-on-disconnect
**Changed:** a disconnect no longer clears the client's ephemeral presence at once. The server gains a `Clock` seam (`clock::{Clock, SystemClock, ManualClock}`, wall-millis) — the analog of the core's `Host::now`, which the server runtime lacked. On disconnect a client with live presence is marked *stale* with a `now + grace` deadline (default 5s); a reconnect (Hello) within the window cancels it; `Registry::sweep` clears every past-deadline client and fans a new `Message::AwarenessClear { channel, actor }` (wire tag 10, server→client) to each affected room's remaining subscribers on their own channel. Replaces #69's synchronous `Hub::clear_client_awareness` on disconnect; that method now *returns* the cleared `(room, actor)` pairs so the sweep can address the fan-out.
**Why:** ARCHITECTURE §Awareness prescribes a reconnect grace window so a brief drop (tab reload, flaky network) doesn't flicker a user's cursor out and back for every peer. That needs wall time on the server, which only the core had — hence the `Clock` seam, injectable so tests drive presence expiry deterministically with a `ManualClock` instead of sleeping. Sweep is explicit (caller-driven) rather than a background timer so the registry stays pure/synchronous; wiring a periodic tokio sweep into the runtime is a separate follow-on. Clearing is gated on the client actually holding presence, so the common no-awareness disconnect never reads the clock (keeps the Miri-gated registry tests off the unsupported system-time call under isolation). TTL-per-entry and throttle/coalesce remain queued on the same seam.

## 2026-07-02 · #63 · handshake auth — opaque credential in, server-derived actor out, pluggable verifier
**Changed:** the handshake gains an Auth phase between Hello and Subscribe. Wire: `Message::Auth { credential: Vec<u8> }` (tag 6, client→server) and `Message::AuthOk { actor: Vec<u8> }` (tag 7, server→client); `ErrorCode::AuthFailed` (already reserved) reports rejection. `credential` and `actor` are opaque byte strings the core never parses. ## 2026-07-02 · #61 · multi-room multiplexing keyed on a client-assigned channel id, not a per-message room tag
**Changed:** a connection carries several room subscriptions at once, each on a connection-local `Channel(u32)` the client assigns at Subscribe. `Message::Subscribe`, `Ops`, and `Snapshot` gained a `channel`; a new `Message::Unsubscribe { channel }` frees one. The server session holds `channels: HashMap<Channel, RoomId>`; the registry fans a broadcast out to each peer on the channel that peer opened for the room. The Hub stays room-keyed — the session translates channel↔room. Replaces "a connection binds a single room; a second Subscribe switches it." `ClientSession` (the SDK-side driver) stays single-room for now, pinned to `Channel(0)`; its multi-room extension is a separate unit.
**Why:** ARCHITECTURE prescribes "logical channels multiplexed per (room, branch, zone)", so the routing key is a channel, not the room bytes. A compact per-connection id keeps op batches small and — the deciding factor — lets a subscription later widen from room to (room, branch, zone) with no wire change, whereas a literal room tag would force re-versioning the frame when branches/zones land.

## 2026-07-02 · #60 · blob-ref reserved as a `Scalar` variant, not a separate `Value` enum
**Changed:** the blob-ref value slot is `Scalar::BlobRef(BlobRef)` (value-codec tag 4), sitting alongside the other leaf values, rather than a distinct `Value = Scalar | BlobRef | ElementRef` payload type. `BlobRef { id: [u8;16], mime, size, inline: Option<Vec<u8>> }`. Slot-only: the blob store, dedup, presigned fetch, and GC still land v0.5. The element-ref value type is left unreserved (no v0.1 promise, shape under-specified).
**Why:** a blob ref merges as an LWW replace on assignment — exactly a leaf value's semantics — so it is the true analog of the `tx` slot reservation: one codec tag, near-zero churn, no refactor of `RegisterSet`/`MapSet`/`ListInsert`. ARCHITECTURE's scalar/blob-ref/element-ref taxonomy distinguishes *merge semantics*, which `Scalar` already models; it is not a mandate for a distinct Rust enum. Trade-off: `Scalar` is documented "no id" in the CRDT-identity sense — `BlobRef`'s handle is opaque payload data (like `Bytes`), carries no CRDT identity, and does not merge or displace, so it still reads as a value, not an entity.

## 2026-07-02 · rewrite · core language is Rust, not C
**Changed:** the core and server are implemented in Rust — a downward `Rc<RefCell<T>>` value graph, `#![forbid(unsafe_code)]`, Miri-gated — not C. Export surface is unchanged: a stable C ABI (cbindgen) for native SDKs + wasm (wasm-bindgen) for the browser. Host seam is `entropy()` + `now()` only; `std`, not `no_std`.
**Why:** the C core's manual refcount/arena lifetime management was error-prone ("memory management was hell"); Rust removes the use-after-free / double-free / aliasing hazard class at compile time while keeping the same portability. Decided at the cheapest moment — only the primitives existed.

## 2026-07-02 · #48, #32 · displacement retains + reinstates, not orphan-and-forget
**Changed:** a displaced container/counter is kept in a persistent per-id registry and reinstated if its slot is re-won; a displaced counter keeps accumulating. Replaces "the displaced element_id may become unreachable; core surfaces an orphan event."
**Why:** orphan-and-forget diverges — two replicas that saw the same ops disagreed on a counter's value across displace-then-recreate. Retention is a convergence requirement, not a nicety. The orphan event still fires for the app; the state itself is retained.

## 2026-07-02 · core op vs. wire envelope
**Changed:** the core op the CRDT engine merges is `{id, stamp, target, kind, tx}`. Authorship (`actor_id`), scope (`room`/`branch`/`zone`), `schema_version`, and wall time are wire/server-envelope concerns wrapping the core op — not core op fields.
**Why:** keeps the CRDT engine pure and portable; the envelope is layered at the wire/server boundary, which is where those fields are actually consumed.

## 2026-07-02 · #52–#56 · snapshot keyed on server seq; cold-start is snapshot-or-delta, no tail
**Changed:** a snapshot is keyed by the server sequence it covers (`base_seq`), not a lamport timestamp, and is regenerated live from the merged replica. Catch-up returns *either* an op delta (at/above the room's compaction floor) *or* a whole-replica snapshot (below it) — never snapshot-plus-tail.
**Why:** the server sequence is what the durable log truncates on; regenerating from the live doc is always current, so a stored-snapshot-plus-tail merge is unnecessary. *Revisit* (KANBAN): O(state) CPU per below-floor cold-start — cache the encoded snapshot if snapshots grow.

## 2026-07-02 · #33, #40 · one custom binary codec, shared by wire and log
**Changed:** a single deterministic little-endian, length-framed, total-decode codec — not CBOR/MessagePack — reused for both the durable op-log and the wire envelope. The 8-byte header (`"CRDT"` magic + version) reserves the version for future codec negotiation.
**Why:** a closed op enum doesn't benefit from a self-describing format; one codec is one code path and one test surface. Negotiation stays possible via the reserved version field.

## 2026-07-02 · #47, #56 · persistence is a per-room append-only file log, not SQLite
**Changed:** the store is `<room>.log` (length-framed ops) + `<room>.snap` (compaction snapshot); crash-safety is hand-rolled (append flushes before return; compaction is temp → fsync → rename → dir fsync → truncate, with dedup-on-replay). No SQLite, no relational tables.
**Why:** the op stream is pure append-only sequential — relational storage buys nothing on the hot path and drops a dependency. *Revisit* (KANBAN): the metadata/query side (admin UI, op-log viewer, audit) may still want an embedded DB (SQLite/redb).

## 2026-07-02 · element_id derives from (parent, key, kind)
**Changed:** the derivation tuple includes the element kind, so a type-flip on a slot yields a different element_id. (ARCHITECTURE previously wrote the tuple as `(parent_id, key)`.)
**Why:** two differently-typed elements at the same slot are genuinely different elements; the kind in the tuple drives the displacement/orphan path correctly.
