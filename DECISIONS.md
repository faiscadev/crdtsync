# Design decisions

Log of design changes to [ARCHITECTURE.md](ARCHITECTURE.md) that implementation forced. ARCHITECTURE always reads as the current intended end-state; this file is *why* it changed. Newest first.

`/cs-implement` appends an entry when building a unit reveals a genuine design change — a forced realization (the design as written is wrong, impossible, or clearly worse than what the code must do), never a mere preference. The human appends when they revise scope. Format: **date · unit/PR · what changed · why**.

The entries below (2026-07-02) are a backfill: design changes made during the v0.1→v0.2 build that predate this log, recovered from the sessions and commit history.



## 2026-07-04 · Unit 5b-ii-2b/#PENDING · admin plane is a sequential control-plane listener; serve seam unified to Send + Sync
**Changed:** no ARCHITECTURE change — implementation decisions for the registration transport.
- **The admin plane serves requests sequentially, no per-connection spawn.** Registration is a rare control-plane operation (an app owner's CI on release), so `serve_admin` handles one request at a time and the `SchemaRegistry` is a plain `&mut` — no `Arc`/`Mutex`. A 30 s per-request timeout keeps a slow or stalled client from wedging the loop; an I/O error or timeout drops that one connection and the plane keeps serving.
- **The registry is admin-plane-owned for now.** 5b-ii-2b only needs registration to *land and be retained* — proven by the response state machine over a socket (a re-`POST` of the same version is `200`, a changed body under a locked version is `409`), with no read route. Sharing the registry into the data-plane handshake is 5c's job (where the WS handshake resolves `{app_id, version}`), so the ownership/`Arc` question is deferred to the unit that actually needs the read side.
- **Unified the data- and admin-plane trust seams to `Box<dyn Verifier + Send + Sync>` / `Box<dyn Authorizer + Send + Sync>`.** `serve_admin`'s future holds the verifier/authorizer across `.await`s, so it is `Send`-spawnable only if they are `Sync`; every impl (`StaticTokens`/`AllowAll`/`Acl`/`PermitAll`) already is, so adding `+ Sync` across the serve chain is a no-op at runtime and lets both planes share one construction. `main.rs` builds a second verifier/authorizer for the admin plane from the same env config (`CRDTSYNC_ADMIN_ADDR`, optional — unset → relay-only, no admin plane) and runs both listeners with `try_join!`.
- **Credential carrier = the `Authorization` header verbatim**, matching the data plane's primary carrier (no `Bearer` stripping); the schema is the request body; the one route is `POST /apps/{app_id}/schemas/{version}`, `app_id` taken as a literal URL-path token.

**Why:** an admin control plane trades throughput for simplicity — sequential handling removes all shared-state locking and the concurrency bugs that come with it, at no real cost for a low-frequency endpoint. The `Send + Sync` unification is the honest fix for a spawnable server (rather than making `serve_admin` un-spawnable), and it sets up 5c to share the registry across both planes without re-touching signatures. Completes the registration write path (5b); 5c adds the handshake read path that resolves against it.

## 2026-07-04 · Unit 5b-ii-2a/#156 · admin HTTP is hand-rolled and strict; request parser split from the socket transport
**Changed:** no ARCHITECTURE change — an implementation + build-slicing decision recorded so the registration transport is legible.
- **The admin HTTP endpoint is hand-rolled, not a framework (`hyper`/`axum`) nor a parser crate (`httparse`).** It matches the engine's "one deployable unit, minimal deps, hand-rolled codec/durability" stance, and the surface is tiny and fully under our control: one `POST` route, `Content-Length` framing only, strict fail-loud rejection of everything else. `core` stays the dep-minimal boundary; the server crate already carries `tokio`/`tokio-tungstenite`/`sha2`, and this adds no HTTP dependency.
- **Strictness is the security posture.** Only `HTTP/1.1`, only `Content-Length` (any `Transfer-Encoding` is rejected — supporting both is the classic request-smuggling vector), a 16 KiB head cap (memory guard, mirroring the codec's decompression-bomb guards), and a total-decode parser: any bytes yield a parsed head, `Ok(None)` for "read more", or a typed `HeadError` — never a panic.
- **Split 5b-ii-2 into the pure parser (2a) and the socket transport (2b).** Untrusted HTTP request parsing is the risky part and is pure/Miri-testable, exactly like `core::json` and the binary codec — so it lands with its own edge-case spec (malformed request line, unsupported version, bad/duplicate `Content-Length`, chunked, over-large head, arbitrary-bytes-never-panic) before any `TcpListener`, route dispatch, or response formatting exists.

**Why:** an admin endpoint may be network-exposed, so its request parser is a real attack surface; isolating it as a strict, spec'd, panic-free pure unit de-risks the transport, and hand-rolling keeps the deploy story ("no external infrastructure") honest without pulling a general HTTP stack in for a single route. Unblocks 5b-ii-2b (listener + route + response wiring).

## 2026-07-04 · Unit 5b-ii-1/#155 · admin registration sliced into a pure handler + an HTTP transport; meta-auth order pinned
**Changed:** no ARCHITECTURE change — a build-slicing + semantics-pinning decision recorded so the registration arc stays legible.
- **Sliced 5b-ii into 5b-ii-1 (pure meta-auth handler) and 5b-ii-2 (HTTP/1.1 transport + listener),** mirroring the existing `session`/`runtime` split — the security-critical authenticate→authorize→register decision is a pure function testable without sockets, and the HTTP wire is a thin shell that decodes a request into it and maps the outcome to a status. The decision flow is where a bug would let an unauthenticated or unpermitted caller mutate the schema registry, so it is isolated and exhaustively spec'd first.
- **Pinned the enforcement order: authenticate → authorize → register.** `admin::register_schema` returns `Unauthenticated` for an absent/unknown credential, `Forbidden` for an authenticated identity lacking `RegisterSchema` on the request's `App(app_id)`, and only then reaches the registry (`Accepted`/`Rejected`). Matches every data-plane enforcement point (verify actor, then authorize, then act) — so a forbidden or unauthenticated request never performs a chain write, even one that would also be a hash-lock refusal.
- **Reuses the data-plane trust seams, not a parallel admin auth.** Registration authenticates through the same `Verifier` (a `StaticTokens`-style admin key → admin `Identity`) and authorizes through the same `Authorizer`, differing only in the action (`RegisterSchema`) and resource (`App`) — the app-admin surface ARCHITECTURE §Registration pins, with no second credential system.

**Why:** the registration endpoint is the schema tier's write path; getting its auth ordering wrong is a privilege-escalation bug, so it lands as a pure, Miri-clean unit with its own spec before any HTTP parsing exists. The HTTP transport (5b-ii-2) then only has to decode bytes and format a status — the trust decision is already proven. Unblocks 5b-ii-2 (the wire) and, with 5b-i, the whole registration path 5c resolves against.

## 2026-07-04 · Unit 5b-i/#154 · schema registry sliced from its HTTP transport; hash-lock semantics pinned
**Changed:** no ARCHITECTURE change — a build-slicing + semantics-pinning decision recorded so the registration arc stays legible.
- **Sliced 5b into 5b-i (pure registry + hash-lock) and 5b-ii (HTTP admin endpoint).** The hash-lock is the security-critical core the handshake resolves against; it is fully testable without a transport, so it lands first, exhaustively spec'd, and the HTTP route becomes a thin authenticated writer over it. Same rationale as 5a: the registry is an in-memory structure (not wire), so no forward-compat cost to deferring the transport.
- **Pinned the three gate-3 failure modes to concrete registration semantics.** ARCHITECTURE §Schema Migration gate 3 names "gap / out-of-sequence / hash mismatch"; the registry realizes them as: a chain is contiguous from version 1 and appends only at `head+1` (skip-ahead → `Gap`); a backward or zero version → `OutOfSequence` (a chain only moves forward — a superseded version is never re-registered); an already-locked version re-registered with different content → `HashMismatch` (links are immutable). An **identical** re-push of the head is an idempotent `Unchanged`, so a CI retry after an unacked response is safe.
- **Hash binds `version‖schema‖migration`.** The schema and migration are length-framed so no boundary shift collides two links; the fixed-width version prefix binds the position so identical bytes at two versions lock differently. Matches the content-addressable blob store (keyed by SHA-256 of content).
- **`sha2` is a server-crate dependency, never core.** ARCHITECTURE already pins this ("the crypto lives in the server, not core … only the server takes the `sha2` dependency"); recorded here as the first code to consume it. Core stays `#![forbid(unsafe_code)]`, `uuid`-only, wasm-embeddable; a client never hash-verifies the server it already trusts.

**Why:** the hash-lock is the trust root of the schema tier — a bug here lets a client slip a different schema body under a known version. Isolating it as a pure, Miri-clean unit with its own spec suite de-risks the larger 5b before any HTTP surface exists, and pins the otherwise-underspecified "gap / out-of-sequence" prose to testable behaviour. Unblocks 5b-ii (HTTP writer), then 5c (handshake resolve).

## 2026-07-04 · auto-version triggers — a general engine event bus, schema-declared triggers (scope, human-directed, pre-build)
**Changed:** ARCHITECTURE §Auto-Version Triggers and §Schema (`autoVersion` key) are extended to pin the design:
- **General engine event bus.** The engine emits typed `EngineEvent`s at lifecycle points, dispatched to pluggable `EventSink`s — generalizing the audit `AccessLog` sink pattern (#85). Auto-versioning is the first built-in sink; the bus is the substrate for future external integrations (webhooks) and can subsume the audit sink later. Chosen over a narrow trigger→version-only mechanism (which would be re-invented when webhooks land).
- **Schema-declared triggers.** An `autoVersion` block in the schema (app-level declarative policy that ships with app code + is version-controlled, like `@auth`/`zones`) — chosen over server config. Each entry is `{ on: <event> | every: <duration>, name: <template>, keep: <N> }`. `on` fires on an `EngineEvent`; `every` is a schedule driven by the existing `Clock` seam + periodic sweep (#71/#72); `name` templates (`${timestamp}`/`${event}`); `keep: N` prunes the oldest auto-versions of that trigger (the retention-window mechanism).
- **Staged event vocabulary.** The full `EngineEvent` enum is defined now; only *available* events fire (version created/deleted, connect, subscribe, snapshot/compaction). The branch/migration events (`before-publish`, `after-restore`, `before-migration`) are declarable but fire once those ops exist (gated on the branch/migration layers) — a trigger on an unavailable event parses and waits, never errors.

**Why:** the last remaining design gate (auto-version triggers needed the engine-event hook API, which ARCHITECTURE named only at intent level: "declaratively in response to engine events or schedules"). The human's calls: a **general event bus** (auto-version is one sink; same machinery serves webhooks + could fold in audit — one event system) over a narrow mechanism; **schema-declared** triggers (declarative app policy, ships with the app, version-controlled — matching `@auth`/`zones`) over server config; a **template + keep-N** naming/retention (which also delivers the queued retention-window item); and a **staged vocabulary** so the mechanism ships now against available events while the branch/migration events wait on their layers. The design reuses two existing seams — the `Clock` + periodic sweep (schedules) and the pluggable-sink pattern (event dispatch) — so it is mostly composition. No code — unblocks the KANBAN auto-version arc: `EngineEvent` + `EventSink` bus → the `AutoVersion` sink over `create_version` (#86) with schema-declared triggers + `keep`-N retention → schedule triggers via the Clock sweep. **This closes the design-gate backlog — the board is fully design-ungated.**



## 2026-07-04 · element-ref value slot + snapshot-clone — two small gates (scope, human-directed, pre-build)
**Changed:** ARCHITECTURE §Internal Data Model and §Export/Import, settling the two remaining small design gates:
- **Element-ref value slot = `Scalar::ElementRef(ElementId)`, a forward-compat reservation.** The missing payload value type (scalars / blob refs / element refs) is a `Scalar` leaf naming another element (mentions, links, foreign keys), mirroring the `Scalar::BlobRef` precedent (#60): a plain LWW value, no substructure, does not merge; a dangling target is an app concern. It carries a **bare `ElementId`** — references are same-room (a room is the sync-isolation unit, no cross-room refs), so no room qualifier; a `kind` hint can be added later if schema validation wants it (rejected adding it now — keep the reservation minimal). Reserved in the codec, round-tripped, no producer/consumer yet — like the blob-ref slot.
- **Snapshot-clone = `clone_room(src,dst)` over export+import; the feared id-namespacing is unnecessary.** The KANBAN item assumed a clone needs id-rewrite / prefix-namespacing + a clock-bump. Tracing the actual collision paths showed **room-scoping already prevents every collision**: server sequences renumber per-room on import (#107); `OpId (client_id, client_seq)` never collides because `client_seq` is monotonic per-client-global (editing a clone of one's own past work still mints fresh seqs, never reusing the cloned `1..M`); a client subscribed to both origin and clone holds *separate per-room replicas*, so a shared `element_id` names distinct objects in distinct documents; and the clock-bump past the imported lamport rides the existing snapshot-adoption high-water (#126). So a clone is just `export_room(src)` + `import_room(dst,…)` under a fresh room id — a thin convenience. Prefix-namespacing would be needed **only** if cross-room id references or cross-room merge ever existed (they don't), so it is deferred until such a feature appears. ARCHITECTURE §Export/Import's stale "element/client IDs are namespaced so no identity conflict" is corrected to the room-scoping reality.

**Why:** the last two small "needs design" gates. Element-ref: the shape was the only open question (BlobRef gave the template) — bare `ElementId`, minimal, same-room, deferring the `kind` hint. Snapshot-clone: investigating the code dissolved the gate — the "two live copies collide on shared ids" premise is false under the engine's per-room replica scoping and per-client-global sequence counter, so the design is "document why it's already safe + add a thin `clone_room`," not a namespacing scheme. No code — unblocks the two KANBAN slices: reserve `Scalar::ElementRef` in the op codec (forward-compat, like #60), and add `Hub::clone_room` over the existing export/import.



## 2026-07-04 · zones (coarse auth partition) — static path-rooted, sealed-handle cross-zone token deferred (scope, human-directed, pre-build)
**Changed:** ARCHITECTURE §Zones and §Schema (`zones` key) are extended to pin the zone model:
- **Zone vs. ACL distinction made explicit.** ACL redaction filters *within one replication stream* (structure/counts/activity still leak); a **zone is a separately replicated stream** with its own lamport clock and op-log partition — an unauthorized client receives nothing (ops, snapshot, structure, existence, size all withheld; per-zone lamport blocks activity inference). Zones = coarse subtree-aligned strong isolation; ACL = fine within-stream.
- **Static, path-rooted, schema-declared.** A zone is a schema `zones` entry (name → subtree root path); membership is positional (everything under the path). Per-element *dynamic* zoning was rejected: it scatters a zone across the tree (defeating subtree=stream isolation, breaking per-zone lamport, and forcing cross-zone "moves" on re-tag) and merely duplicates the doc-ACL layer, which is exactly the fine-grained dynamic tool. Causal independence is enforced (cross-zone tree moves + anchors forbidden), so the per-zone lamport clocks never need cross-zone ordering. Zone access reuses the authz seam (`Resource::Zone`, subscribe-gated); the `Channel` widens to `(room, branch, zone)`.
- **Cross-zone references: opt-in, deferred, sealed-handle when built.** Default is forbidden (schema validation rejects a cross-zone anchor). The opt-in is a **per-recipient redaction, not merged state**: the authoring client (authorized for both zones) writes the real anchor, the server stores it, and only at fan-out to a recipient lacking the target zone does the server swap the real `(zone, element_id, position)` for an **AEAD-sealed opaque token** (server key; deterministic sealing → stable token; associated data binds it to the room). Stateless (token = the sealed data, no mapping table/GC), reusing the server-crate crypto precedent (schema hash-lock). The random-id + server-table alternative (blob-ref-style) was set aside for the stateless sealed handle. Deferred to a follow-on; the first cut ships cross-zone anchors forbidden.

**Why:** zones were the meatiest remaining design gate (unblocks authz zones + channel-mux `(room,branch,zone)`), opened once XmlElement — the subtree substrate — was pinned. The human's calls: **static path-rooted** (the isolation is only cheap when a zone is a contiguous subtree = one stream; dynamic per-element zoning blurs zones into a worse ACL), and **defer the cross-zone opaque token** (cross-zone refs are the advanced case; ship the isolation value first), using an **AEAD-sealed handle** when it lands (stateless + stable + tamper-proof, no server table, reuses the hash-lock crypto precedent) over the random-id+table variant. No code — unblocks the KANBAN zones arc: schema `zones` parse + validation (cross-zone move/anchor rejection) → envelope `zone` + per-zone lamport → zone-scoped subscription + `Resource::Zone` authz + wire redaction (unauthorized zone entirely absent) → (deferred: sealed-handle cross-zone token). Depends on the tree primitives (§CRDT Model) for meaningful subtrees and the authz seam (§Authorization, done in design).



## 2026-07-03 · XmlElement / XmlFragment / RangedElement / marks — the v0.5 primitive gate (scope, human-directed, pre-build)
**Changed:** ARCHITECTURE §CRDT Model and §Marks are extended to pin how the last unbuilt primitives map onto crdtsync and how they slice. The algorithms were already fixed (Kleppmann-2021 tree-move, Fugue children, Peritext marks); this settles the representation + scope the loop would otherwise park on:
- **XmlElement = composition + one hard algorithm.** `XmlElement { tag: String, attrs: Map, children: Fugue<XmlElement|Text> }` reuses the **Map** primitive for attrs (attrs hold CRDT values — decided **full Map**, not scalar-only) and the **Fugue** sequence engine (List/Text) for children; `XmlFragment` is a tagless/attr-less children sequence (tree root). The only new algorithm is the **tree move**, which is **deferred to its own slice** after the structural build (create/edit/delete children) — it is the hardest, most bug-prone part (Kleppmann undo-and-replay), so structure lands first.
- **RangedElement is first-class and fully generic** (decided over Text-local-only). `RangedElement { start, end, payload }` where each endpoint is an anchor `(element_id, RelativePosition)`, so a range may **span elements** (a comment across paragraphs), `start.element ≠ end.element`. RangedElements live in a **document/fragment-level annotation set** (CRDT set keyed by RangedElement id), not inside the annotated Text; "marks on this Text" is a query over the set filtered by `element_id`.
- **Marks = convention over RangedElement.** Payload carries name + value; `start`/`end` are `RelativePosition`s. **Anchor expansion needs no new mechanism** — it is the `RelativePosition` **gravity** (`Before`/`After`, #134) of the boundary anchor. Three schema-declared merge flavors: **boolean** (presence; concurrent add+remove resolves **LWW by stamp** — decided, consistent with Register), **value** (LWW), **object** (independent instances — comments). Per-character active-mark state is a **read-time computation** over the RangedElement set (deterministic function of merged state → convergent by construction), no per-character mark storage.

**Why:** XmlElement/marks was the last big primitive and the highest-cascade remaining gate (rich text, doc trees, attrs, the deferred invariant-repair rules — orphan-wrap, exclusive-collision — and XmlElement/marks/attrs diff detail all wait on it), and the human chose to open it. ARCHITECTURE already pinned the *algorithms*, so the open decisions were representation + scope, and the loop would park on the correctness-critical ones (tree-move integration, Peritext concurrent-mark resolution). The human's four calls: **defer tree-move** (de-risk — ship structure first), **full generic RangedElement** (one primitive serves marks + comments + suggestions, ranges span elements), **LWW-by-stamp** for boolean-mark add/remove (matches the engine's existing LWW), **attrs = full Map** (matches "attributes that can themselves hold CRDTs"). The representation falls out of the built machinery — Map (attrs), Fugue (children), RelativePosition (anchors, gravity = expansion) — so the primitives are mostly *composition*, and marks-as-read-time-computation mirrors the invariant-repair normalization decision (2026-07-03): derive per-character state from the merged set, store nothing extra, converge by construction. No code — unblocks the KANBAN XmlElement/marks arc: structure → tree-move → RangedElement → marks → (then the deferred repair rules, attrs, and diff detail become buildable).


## 2026-07-04 · schema Unit 5a · Action/Resource extensions — RegisterSchema + App now, Room path/author deferred (implementation, PR #153)
**Changed:** No ARCHITECTURE change — implements the registration-arc primitives (ARCHITECTURE §Registration / §Actions / §Resources). `Action::RegisterSchema` + `Resource::App(app_id)` land, with `ResourceMatch::App` and policy tokens `register_schema` / `app:<id>`; room and app scopes are **disjoint** (a `room:`/`*` rule never covers an app resource, nor the reverse — the data plane and the control plane don't cross). One concrete slicing decision:
- **Room→path-carrying + `author` is deferred to its consumer (4c / doc-ACL), not done in 5a.** The KANBAN 5a line bundled it, but `Resource` is an **in-memory enum, not a wire format**, so adding a `path`/`author` to `Resource::Room` later is a pure refactor with zero compatibility cost (unlike the element-ref wire reservation, which had a real forward-compat rationale). Adding those fields now — with no evaluator reading them until 4c — would be speculative code that ripples to all six `authorize` callsites for nothing. So 5a ships exactly what the registration endpoint (5b) needs (`RegisterSchema` on `App`), and the path/author Resource shape lands when 4c/doc-ACL actually consume it.

**Why:** mirrors the 4a capture-not-consume slice — build the piece the next unit needs, don't speculatively churn callsites for a consumer several units away. Disjoint room/app scoping is the safe default (an app-admin `register_schema` grant must not accidentally widen into a room grant, and a room wildcard must not confer schema-registration authority).

## 2026-07-04 · schema Unit 4a · claims-model credentials-file grammar + capture-not-consume slice (implementation, PR #152)
**Changed:** No ARCHITECTURE change — implements the pinned claims model (`Verifier → Identity { actor, roles, groups }`, ARCHITECTURE §Authorization / #142) and fixes two concrete choices:
- **`StaticTokens` credentials-file grammar.** A row grows from `<credential> <actor>` to `<credential> <actor> [roles] [groups]`: `roles`/`groups` are comma-separated name lists (`editor,viewer`), a lone `-` is an empty list (so a row can assert groups but no roles), arity is 2–4, and an empty name (stray comma) is a new `CredentialsErrorKind::EmptyName`. Two-field rows keep their exact old meaning (no roles/groups), so existing files are unaffected.
- **4a captures the claims; 4c consumes them.** `Verifier`/`StaticTokens`/`Session`/`Registry`/anon all thread and store the full `Identity`, and `Session::identity()` exposes it — but the policy evaluator is left untouched: `Session::actor()` still derives the actor bytes, so the `Authorizer::authorize(&[u8])` seam and all its callsites are unchanged. The signature change to `authorize(&Identity)` plus `Subject::Role`/`Group` matching is Unit 4c, where it is coupled with actually reading roles/groups.

**Why:** the model is pinned; the file grammar is `cs-next`'s to fix (mirrors the 4b `@auth` JSON-encoding decision) — a comma-list + `-` sentinel keeps the line whitespace-tokenized and back-compatible. Slicing 4a as capture-only keeps its blast radius to the credential→identity path and leaves the whole authz/acl/policy test surface green; 4c is then a purely additive evaluator change. Roles/groups are `Vec<String>` (names, matched against the schema `@auth` `Subject::Role(String)` later); the actor stays `Vec<u8>` (an opaque server-derived id).

## 2026-07-03 · schema Unit 4b · `@auth` JSON encoding + `own`-not-schema-grantable (implementation, PR 145)
**Changed:** No ARCHITECTURE change — this fixes the concrete JSON encoding of the pinned `@auth` DSL (ARCHITECTURE §Schema File / §Authorization describe it in prose), which `core::schema` now parses. Two decisions worth recording:
- **Effect + action are fused into the `allow`/`deny` key.** A grant is `{ "allow"|"deny": "<action>", "to": "<subject>", "on": "<path>" }` — exactly one of `allow`/`deny` present, its *value* the action. Matches the human's `{allow|deny, to, on}` shorthand (KANBAN Unit 4) and yields a clean closure rule (neither/both → rejected). `to` is a declared role, a subject class (`authenticated`/`anonymous`/`anyone`), or an ownership template (`${actor_id,author_id,room_id,branch_id}`) — **not** a bare actor id (per-actor grants are dynamic doc-ACL, never schema). `on` is a well-formed absolute path (`/` or `/seg/seg…`, no empty segment).
- **`own` (and the meta-auth actions) are not schema-grantable.** Schema-grantable actions are exactly `read`/`write`/`publish_awareness`; `own` in a schema grant is rejected (`UnknownAction`). ARCHITECTURE already states ownership is dynamic doc-level ACL state "never declared in the schema" — this makes the parser enforce it, so a schema cannot smuggle a static ownership grant.

**Why:** ARCHITECTURE pins the *model* but not the byte-level JSON; `cs-next` owns the encoding. Fusing effect+action follows the pinned shorthand and keeps the grant a flat 3-key object; excluding `own` at parse time is the closure-of-violations discipline (the schema tier can only express what the schema tier owns). Value-shape enforcement of grants at check time is Unit 4c (server evaluator).


## 2026-07-03 · invariant repair (schema Unit 2) — read-time normalization; reject sequence `min` (scope, human-directed, pre-build)
**Changed:** ARCHITECTURE §Invariant Repair gains a **Mechanism** section and the **Closure** section is extended, settling the two questions the loop parked Unit 2 on:
- **Convergent-repair mechanism = read-time normalization, never a minted op.** Repair is a deterministic pure function of the merged op-set applied when materializing a read; the stored/encoded state stays the raw merged ops. The lamport order repair needs (keep-oldest, drop-newest) comes from the **stamps already in state** (Map-slot / Register / sequence-node), never the local clock — so `encode_state` is byte-identical across replicas and repaired reads converge by construction. A clamp returns the clamped value on read (stored value/stamp untouched); a disallowed value or over-`max` excess is hidden on read. No op minted → no repair-stamp to diverge. The one element-*creating* repair (orphan → wrap in default block) also mints no op — the wrapper's `element_id` is *derived* from the violating position (as Map slots derive from `(parent,key,kind)`), so every replica synthesizes the same wrapper; it ships with XmlElement/default-block. `onRepaired` fires at the **apply** boundary (the validator detects the violation there); the repaired value is produced at read. Apply-time *materialization* (rewriting stored state) is rejected — a clamp that overwrote the stored value would need a stamp and reintroduce divergence. This resolves the loop's open "derive-stamp (new `Map`/`Register` stamp accessors) vs state-normalization" choice in favor of normalization, so no new accessors are needed.
- **Reject `min` on a sequence at parse time.** A sequence *upper* bound is repairable (drop the lamport-newest excess); a *minimum count* is not — concurrent deletes underflow it and repair cannot invent items — so admitting it would break the closure invariant ("no schema admits an unrepairable runtime state"). `List.min` (the only such field: `Text` has only `max`; register/counter `min` is a numeric clamp, unaffected) is therefore rejected at schema parse. Minimum cardinality is a *semantic* constraint (structure = core, semantics = app) apps express structurally instead: a **structural floor** (required minimum as fixed Map slots + a List for the variable part above — "≥ 2 options" = `optionA`/`optionB` slots + `moreOptions` list, guaranteed by construction since a slot can't be concurrently deleted out of existence), or a **transition gate** (refuse `published = true` unless the count is met — best-effort holds at a single-actor moment). `List.min` was parsed-but-unenforced (#136), so removing it drops no shipped behavior.

**Why:** Unit 2 was gated on (1) how repair converges given it may write stamps and (2) the `min`-on-sequence closure gap (PR #139 / DECISIONS 2026-07-03). Grounding in the code resolved both: the built-primitive repairs (scalar/counter clamp, disallowed-value drop, exclusive keep-oldest, sequence-`max` drop-newest) are all pure read-time *filters* of merged state — none creates an element or rewrites a value — so "repair mints an op with the local clock and diverges" never arises; normalization is convergent by construction and needs no stamp at all. ## 2026-07-03 · authorization — role membership is two-source, group subjects, provenance-bounded deny (scope, human-directed, refines #142)
**Changed:** ARCHITECTURE §Authorization (Subject Types, Role tier, Ownership tuple, Decision Flow) is refined over the model merged in #142, which framed role membership as token-only. Three additions:
- **Role membership is two-source.** A **token** role is deliberately *global* over the `app_id` (a bare token `editor` = editor of every doc — reserve token roles for genuinely app-wide authority like `admin`). A **per-doc** role is assigned in the doc-level ACL by an owner, scoped to a path ("Alice is `editor` of X"). The schema still only *defines* a role's powers; who *has* it is a token claim or a per-doc grant, never the schema. An actor's **effective roles** on a resource = token roles ∪ per-doc assignments for the actor or its groups on the resource-or-ancestor.
- **Group subjects.** The token carries group membership (`Identity { actor, groups, roles }`); a doc-level ACL tuple can target a **group**, not just an actor — assign a whole team a role/capability at once. `alice ∈ designers` (token) + `designers = editor on X` (doc-ACL) ⇒ Alice edits X. The doc-ACL tuple subject is `Actor | Group | Authenticated | Anonymous | Anyone` and its grant is `Capability(read|write|publish_awareness|own) | Role(name)`.
- **Provenance-bounded deny.** A `deny` **always** beats a static default (schema role-grant / global token role) — so an owner's `deny read alice` on doc X overrides Alice's app-wide `viewer` role. But **between doc-level grants** a deny is provenance-bounded: it overrides an allow/ownership only from the deny author's own subtree (superior carve-out), never a *peer's* or *superior's* grant. Otherwise deny would be a backdoor around provenance-based revocation (a co-owner could `deny` a peer it cannot `revoke`). A co-owner can no more deny a peer than revoke one.

**Why:** flushing the token-only membership from #142 against real use showed a bare token role is global — "Alice is editor" would make her editor of *every* document, which is wrong for the common case. The human directed: token roles stay global-on-purpose (for app-wide roles), and doc-scoped roles come from per-doc doc-ACL assignment — with **groups** as the membership indirection (token carries group membership; the doc-ACL assigns roles to groups), matching how Notion/Google-Docs/IAM scope a role to a resource and assign teams. The deny question ("can a deny override an app-wide role?") then exposed that a literal deny-wins-global would let a co-owner deny a peer, defeating the provenance-revocation guarantee (#1) — resolved by bounding a deny's reach by provenance for doc-level grants while keeping deny-beats-static for the app-wide/schema case. No code — refines the design-of-record; builds in the doc-level ACL-as-CRDT gate (KANBAN), whose tuple now carries subject ∈ {actor, group, class}, grant ∈ {capability, role}, effect, path, grantor.

## 2026-07-03 · authorization / trust / ownership model — the schema Unit 4+5 design gate (scope, human-directed, pre-build)
**Changed:** ARCHITECTURE §Authorization is rewritten and §Schema (`@auth`, Registration) extended to pin the authorization, trust, and ownership model the loop parked on (schema Units 4 `@auth` + 5 registration/trust, DECISIONS 2026-07-03 / PR #139). The model:
- **Claims model** — the `Verifier` maps a credential to `Identity { actor, roles, groups }` (was `actor` only). Membership comes from the token (the app's IdP issues it); the engine reads claims, never decides membership. This is the single extension that makes role subjects — and thus the ARCHITECTURE decision flow — buildable.
- **Three authority tiers, distinct mechanisms:** **App admin** — the schema-registry authority (app owner / CI), a **superuser** over every doc in the `app_id`, above the document; a credential class (the registration key), not a role, not an owner, never in `@auth`. **Owner** — a dynamic, recursive, path-scoped capability (doc-level ACL state); creator auto-owns `/`; multiple owners per path. **Role** — static schema vocabulary + grants, membership from claims.
- **`@auth` DSL (Unit 4)** — `auth.roles` (vocabulary) + `auth.grants` (`{allow|deny, to: role/subject/${template}, on: path}`), static, authored by the app admin at CI. **Ownership is *not* in the schema** — it is entirely runtime doc-level ACL state, so the admin never writes it in stone.
- **Ownership model (dynamic capability):** a doc-level ACL tuple is `{ subject, capability ∈ {read, write, publish_awareness, own}, path, grantor }`. **Attenuated delegation** — an owner of P may write a tuple on P or any subpath (co-owner of P, owner of a subpath who re-delegates recursively, or a leaf grant); uniform rule: *may write a tuple on Q iff owns Q or an ancestor, or is app admin.* **`own` is delegable authority, leaf grants are not** (a read/write grantee cannot re-grant). **Provenance-based revocation** — a tuple is removable only by its grantor (the tuple's op author, un-forgeable) or above it in the grant chain, NOT by whoever owns an ancestor path; so co-owners of a common superior cannot revoke each other, and a superior-imposed carve-out cannot be removed by the subordinate. **Downstream deny + carve-outs** — grants and denies inherit downward, deny-wins globally (no re-open below an ancestor deny); a superior places `deny own` on a subpath to carve a subordinate's authority, and provenance-removal makes it stick.
- **Decision flow (updated):** 0 app-admin superuser → 1 explicit DENY (deny-wins) → 2 owns resource-or-ancestor → 3 explicit ALLOW leaf → 4 schema `@auth` role-grant → 5 default-deny. Permission state versioned in lamport time, so concurrent grant/revoke resolves at the op's lamport position (existing §Hard Problems mechanism).
- **Registration (Unit 5):** an HTTP **admin endpoint** (control plane, separate from the sync WS), gated by `register_schema` on `App(app_id)` with a registration credential → admin `Identity`. **Hash-lock is SHA-256 in the *server* crate**, not core — core stays dep-minimal and clients never hash-verify (they trust the server); only the un-embedded server takes `sha2`. Resolves the loop's "crypto dep into dep-minimal core" concern by placement.
- **`Action`/`Resource` extensions:** `+register_schema` action, `+App(app_id)` resource, `Room`→ path-carrying resource (with the resource's `author` for `${author_id}`). The `Authorizer` trait is untouched (already generic over `Action`/`Resource`).

**Why:** the loop (PR #139) shipped every cleanly-buildable schema slice and parked on Units 4/5 because the grant DSL, the meta-auth credential model, the `Action`/`Resource` trust extensions, and the hash-lock crypto were unspecified and security-consequential — "not mine to force." The design was settled in discussion around these constraints: (1) the schema drives authorization, so a **client-supplied schema body can't be trusted** — enforcement rests on the admin-registered, hash-locked schema (client asserts only a version key); (2) "admin" is ambiguous and conflating scopes is a hole, so it splits into **app admin** (registration superuser, above docs) vs **owner** (per-instance, inside docs) — with the human directing that ownership be **fully dynamic** (not a schema-declared allowlist), **path-scoped**, **multi-owner**, and **recursively delegable with attenuation** (an owner delegates its own path or any subpath, incl. co-ownership); (3) revocation needs to express "owner A can't revoke owner B," solved by **provenance-based revocation** (revoke what you granted, not what you sit above) — which also makes **superior carve-outs** (`deny own` on a subpath) enforceable, since the subordinate can't delete a constraint it didn't place; (4) downstream read/write deny and owner carve-outs both fall out of **path-inherit + deny-wins-global + provenance-removal** with no new primitive; (5) crypto placement in the server crate keeps core dep-minimal. No code yet — this unblocks the schema build slices in KANBAN: Unit 4 (`@auth` parse + evaluator role-grant tier), Unit 5 (Verifier→Identity claims, HTTP registration + meta-auth + hash-lock, `Action`/`Resource` extensions), then the gated 3b/6-enforcement. The **dynamic ownership / doc-level ACL tuples** (owner/grant/deny CRDT state + the walk-up-the-path evaluator + `AclGrant`/`AclRevoke` + creator-auto-owns op + grantor recording) build in the separate **doc-level ACL-as-CRDT** gate; this pass pins their design and records the tuple's `grantor` field as the substrate for provenance revocation. **Invariant repair (Unit 2)** stays a separate gate (CRDT repair determinism + the `min`-on-sequence closure gap), not folded in.

## 2026-07-03 · schema layer — distribution / trust / versioning model (scope, human-directed, pre-build)
**Changed:** ARCHITECTURE §Schema is rewritten and §Schema Migration extended to pin down the schema layer's distribution, trust, and versioning model (the design gate that was blocking the awareness-TTL, `@auth`, invariant-repair, and migration units). The design is now:
- **Opt-in feature tier.** Schema-less documents are first-class (converge / persist / fan-out / snapshot / offline / room-ACL, zero ceremony); a schema is adopted only to unlock producer validation, invariant repair, migration, fine-grained `@auth`, typed API, awareness TTL/throttle, and marks/attrs/structural constraints. Adoption is never a prerequisite.
- **Schema is code, not document state.** Authored JSON, versioned, CI-gated, checked into the app repo. The document carries only the `schema_version` envelope tag, never the schema body. Schema-as-doc-state was rejected (it makes the schema concurrently mergeable → kills migration determinism, defeats the CI drift/verify gates + boot hash-lock, and creates a bootstrap cycle).
- **Distribution.** Client gets the schema via a build-time **bundle** (typed API + cold-offline) and/or a **handshake advertisement** (server → client, cached) — so bundling is optional (typed client bundles; dynamic client fetches at handshake and adopts the served version). Server gets it via **registration** (the app owner's CI pushes `{app_id, version, schema, migrations}` over an admin API), never deploy-time config — so a generic engine binary serves any tenant and a SaaS server is a per-`app_id` registry.
- **Two server tiers.** **Relay** (no registered schema) stores/dedups/fans-out/persists/snapshots + room-ACL only; clients still validate/repair locally. **Enforcing** (registered schema) adds ingress validation, authoritative repair, in-flight migration, schema `@auth`. The split exists because merge needs no schema.
- **Trust boundary.** A client-supplied schema **body is never trusted for enforcement** — the enforcing server enforces only its **registered** schema; the client asserts a version *number* used purely as a registry lookup key (unknown → reject, never fabricate). Registration is meta-authed (app-owner CI credential) + hash-locked. A client's own schema is advisory; the server re-validates every op. (Repair `f(state,schema,lamport)` converges across replicas only when they share the schema; the registered server is the arbiter.)
- **Versioning: the server is the compatibility layer; a client speaks a single version.** A typed client is generated for its build version, a dynamic client adopts the served one, and each declares on handshake the version it wants ops delivered at (normally a "range of one"; a multi-version-codegen client, rare, accepts a small range). Each migration edge is classed **back-compatible** (a bidirectional op-rewrite exists) or **breaking** (forward-only, no inverse), by the CI gates. The server **composes the per-edge op-rewrites along the chain, per recipient, on the fan-out seam** (the same seam as redaction #83) over an **immutable heterogeneous log** (each op stored at its creation version, never rewritten); cold-start migrates materialized state into the snapshot instead. The **handshake range-check guards** every down-path (a recipient across an unbridgeable breaking gap is refused before it subscribes), so fan-out only ever traverses invertible edges. Only an unbridgeable breaking gap rejects (`onUpdateRequired`). The relay/enforcing tier is decided **per `app_id`** (schema registration), not per document.
- **Three ceremony tiers, app-chosen per change:** no-schema (no migration concept) / lazy-breaking (break freely, `onUpdateRequired` handles stranded clients) / zero-downtime (expand→migrate→contract with version ranges). Same machinery; the only difference is whether an edge is made back-compatible.
- **Lifecycle hooks.** `onRepaired` / `onOpsRejected` / `onUpdateRequired` — engine detects, app decides UX (observation, not override), extending the existing repair-event + rejected-ops doctrine.

**Why:** the schema layer was the highest-leverage blocked gate — awareness timed-TTL/throttle, `@auth` defaults, the whole invariant-repair section, and schema migration were all waiting on "where does the schema live and how is it trusted," which ARCHITECTURE described only at the intent level. The forcing constraints, surfaced in design discussion, were: (1) the server must run as **multi-tenant SaaS** hosting documents from apps whose developers do not operate the server — so the operator cannot bake schemas into the deploy; registration by the tenant's CI is the only distribution that keeps a generic binary; (2) the schema drives **authorization / validation / repair / migration**, so a client-supplied body cannot be trusted — the trust anchor must be the admin-registered, hash-locked schema, with the client asserting only a version key; (3) adoption must be **frictionless** or it becomes an adoption nightmare — hence opt-in with a fully-functional schema-less relay tier; (4) real deployments need **zero-downtime rolling schema changes** (the update-to-both → change → update-to-new dance), which a scalar "reject on version mismatch" cannot express — hence client-declared version ranges + per-edge compatibility classes + per-recipient in-flight translation, with a hard reject only across a genuinely breaking gap. No code yet — this is the design-of-record that unblocks the schema build slices in KANBAN (validation engine → awareness TTL + scalar clamp + `@auth` first, since those consume only built primitives; marks/attrs/structural clauses parse but stay enforcement-inert until XmlElement/marks land).

## 2026-07-02 · tombstone GC → tombstone compression (design pivot, human-directed)
**Changed:** ARCHITECTURE §Tombstone GC (both copies) + the *Implementation Status* note flip from "remove tombstones below a `min(last-acked seq)` watermark" to "**never remove tombstones — compress them.**" A contiguous deleted run (consecutive char/list ids forming one insert's parent-chain, all tombstoned) encodes as a single range record, and a tombstone's dead value is dropped, at state-encoding time (snapshots, cold-start catch-up, the durable `.snap`). §Op Acknowledgement's `Ack { channel, seq }` frame is demoted to a **reserved, no-consumer** wire slot: its only job was to feed the GC watermark, which no longer exists. The `Accepted` frame + `ClientSession` outbox (the offline queue) are unaffected and stay built. The retention-horizon eviction policy and the load-bearing floor-coupling invariant (both watermark-GC machinery) are deleted.
**Why:** watermark-based *removal* is the wrong target. Building it surfaced that removing a positional node is only safe when no replica still references it — but clients never compact (only the server does), so a live always-connected client holds every tombstone forever and can always author an insert anchored to one. The strict "provably unreferenced" condition therefore essentially **never fires** for live clients (it degenerates to the never-GC case). Making it fire forces one of two bad mechanisms: **reparent** (removal must become a *replicated GC op* carrying a forward pointer, because peers that kept the tombstone must reparent identically — a whole new op kind + forwarding table) or **reject** (a late insert into a removed cold region is dropped and surfaced — rare, but real user-facing data loss). The human rejected both — reject for the data loss, reparent for the complexity — and directed a step back.

The reframe: the goal was never "remove tombstones," it is "**tombstones don't bloat.**" Removal is what drags in watermarks, acks, horizons, clocks, and orphan handling; compression achieves the goal with none of it. Deleted content in real editing is contiguous (words/lines/paragraphs = consecutive same-client ids on a parent-chain), so run-length range-encoding collapses a deleted region from O(deleted items) to O(runs), and a tombstone's value is never read again so it is dropped. It is a **pure encoding optimization** — identical logical state, fewer bytes — so convergence is preserved by construction (the harness's materialized-read invariant is untouched), and there is no distributed decision, no removal, and therefore no orphan problem to solve. This is precisely what shipped CRDTs do (Yjs compresses deleted items into range structs; Automerge columnar-RLEs the log); the watermark-removal path was reinventing a problem the field already routed around. First build: encoding-level run compression in the state codec (bounds snapshot/wire/disk/replay). Carrying the range representation in-memory (bounding RAM too) is a later step, deferred until RAM bloat is actually measured.

## 2026-07-02 · op-acknowledgement gate — the ack protocol behind offline queue + tombstone GC (scope, pre-build)
**Changed:** ARCHITECTURE gains §Op Acknowledgement under §Realtime Synchronization, and §Tombstone GC (both copies), §Offline-First, and the *Implementation Status* GC note now reference it. Two wire frames, one per direction, both meaning "my durable commit frontier on this channel is X": **`Accepted { channel, through }`** (server → client), the highest per-client op sequence (`OpId.seq`) of the author's own ops the server has durably logged — it drains the client outbox; **`Ack { channel, seq }`** (client → server), the server sequence the client has applied — it feeds the tombstone-GC watermark. The `ClientSession` outbox retains authored ops, `edit` appends, `Accepted` prunes `id.seq <= through`, a reconnect re-emits the unpruned tail. The GC floor is `min(last-acked seq)` over clients within a configurable **retention horizon**; a client absent past the horizon is dropped and re-synced by Snapshot if it later returns below the compaction floor. No code yet — this is the design-of-record that unblocks the two gated units.
**Why:** both features were in ARCHITECTURE at the *intent* level (§Offline-First "offline op queues"; §Tombstone GC "min watermark") but neither had a *mechanism*, and the built protocol has no acknowledgement path at all — the server never echoes an author its own ops, so today a client cannot learn a local write landed (no outbox drain) and the server learns a client's caught-up position only once, at Subscribe (no live watermark). One concept — acknowledge progress along the per-channel sequence — supplies both, in opposite directions, so it is one gate.

Three sub-decisions, each with its rejected alternative:
- **Author-ack keyed by the per-client op sequence (`OpId.seq`), not the server sequence.** The op identity `(client_id, seq)` is what idempotent ingest already dedups on and is stable across reconnect, so a resent op re-acks to the same `through` and the outbox prune is idempotent. Server-sequence correlation was rejected: a resent op is assigned a *new* server sequence on ingest, so a server-seq-keyed ack would fail to match the outbox entry after any reconnect — exactly when the outbox matters most. (Note: this is `OpId.seq`, the per-client dedup counter, not the causal Lamport `Stamp`.)
- **Retention-horizon eviction** for the GC watermark, over two alternatives. *Live-subscribers-only* (drop a client from the floor the instant it disconnects) was rejected because it guts the very offline queue this gate builds — it GC's a client's tombstone context the moment it goes offline, so a returning offline edit resurrects concurrently-deleted elements at maximum rate. *Never-GC-while-any-known-client-is-behind* was rejected because one client that dies forever stalls GC for the whole room — unbounded tombstone growth. The horizon is the bounded middle: an offline client within the window resolves its edits against still-present tombstones; past the window it re-syncs by Snapshot. None of the three causes divergence given the invariant below; the horizon is the only one that both bounds growth and preserves the offline case.
- **Floor-coupling is the correctness guard, not the eviction policy.** Load-bearing invariant: GC never removes a tombstone at or above the compaction floor, and a client reconnecting below the floor always receives a Snapshot, never a delta. This keeps every eviction policy convergent — a stale replica adopts server truth via snapshot instead of replaying past a missing tombstone. The residual cost of GC'ing a tombstone "early" is *semantic*, not consistency: an offline edit to a concurrently-deleted element resurrects it (convergently) instead of losing to the delete; the horizon bounds how often that window is hit.

## 2026-07-02 · #115 · schema-aware diff — the wasm surface + the cross-language change representation
**Changed:** `WasmDocument.diff(oldState, newState)` decodes two snapshot buffers, computes `core::diff`, and returns a JS array of plain change objects: each has `op` (string tag), `path` (Uint8Array), and its variant's fields — `kind` (string) for add/remove, `old`/`new` for value/counter, `index`+`items` for list runs, `index`+`text` for text runs. A scalar is a **tagged `{ t, v }` object** (`t` ∈ `null`/`bool`/`int`/`bytes`/`blobref`; `int` is a BigInt, `bytes`/`blobref` a Uint8Array); a list item is `{ scalar }` or `{ kind }`. ## 2026-07-02 · #102 · atomic transactions — count-tagged members, buffered until whole, applied together
**Changed:** the reserved op-envelope `tx` field, previously an unused `Option<TxId>`, becomes `Option<Tx>` where `Tx { id, count }` carries both the shared transaction id and the group size; the op codec round-trips the added count. `Document::atomic_transact(f)` mints a per-replica `TxId` (namespaced by the replica's client, so `(client, tx)` is globally unique) and tags every op the closure emits with the group's `Tx`. On the receive side, an atomic member is always buffered; a transaction commits only once every one of its `count` members is present *and* the group's external dependencies resolve, at which point all members apply together in op-seq order. A single-member atomic transaction commits immediately. The author applies its own edits eagerly, as with any local edit.
**Why:** ARCHITECTURE §Transactions specifies opt-in atomic transactions as an all-or-nothing *view* boundary ("receivers buffer member ops until commit marker arrives; on commit, all apply atomically"). The wire envelope already reserved `tx` (round-tripped since the op codec shipped) but nothing set it and no atomic behavior existed. The "commit marker" is realized as a **per-member count** rather than a separate terminator op: it makes each member self-describing, so completeness is detectable regardless of delivery order or splitting across messages, with no distinct marker to lose or order. Atomicity is enforced purely as a *buffering gate* — the same ops still merge, so convergence is unchanged by construction (an atomic author and a non-atomic peer reach identical state; verified against the convergence property harness), and the only observable difference is that no peer sees a partial group. The group commits in op-seq order so a member targeting a container an earlier member creates lands after it; the readiness gate treats intra-group creates/inserts as satisfied (they will exist once the in-order apply reaches them) and only blocks on dependencies *external* to the transaction, so a self-contained tx (create-then-populate, insert-then-delete) never deadlocks. Buffered partial transactions ride the existing op-buffer, so they persist in a snapshot and complete after a restart for free. `next_tx` is not persisted: tx ids scope only the buffering of *remote* partials (keyed by the author's client), and a restored replica mints under its own client with fresh op ids, so restarting the counter at 0 cannot collide with anything buffered. This is the core slice; the wire/server/client drive path and the SDK `atomic_transact` surface are the follow-on slices. Atomic-tx *undo* (ARCHITECTURE §Transactions "a transaction is naturally an undo intention") and scope constraints tied to branches/zones/schema remain gated on those layers.

**Why:** ARCHITECTURE §Named-Versions specifies "snapshot + entry in a versions index" with list/paginate/rename/delete first-class — all of which the shipped state codec (#54) and snapshot machinery (#55–#58) already make buildable, so this lands the index layer directly on them. Two parts of the versioning section are deliberately *not* built, each dependency-gated: **restore** is restore-as-branch (ARCHITECTURE §Restore-as-Branch forks a new branch from the chosen snapshot and switches HEAD — it explicitly does *not* rewrite live state), so it needs the unbuilt branch layer; the API therefore exposes reading a version's state but no mutating restore. And **durable persistence** of the index (versions are "retained until the app deletes") plus **auto-version triggers** (engine-event/schedule-driven creation) are follow-ons: the `Store` persists the op log + compaction snapshot but has no versions record yet, so the index is in-memory for now — the same hub-first / store-later split the compaction arc used. Wire messages + SDK surface for versions are the next slices once the index shape is proven. `BTreeMap` keys the index so listing is sorted for pagination without a separate sort at the query surface.

## 2026-07-02 · #85 · access log — an `Audited` authorizer decorator records every authorization decision at the seam
**Changed:** the server gains `audit::{AccessLog, AccessRecord, Audited, Decision}`. `AccessLog` is a pluggable sink (with a blanket closure impl) a deployment supplies; `Audited` wraps an inner `Box<dyn Authorizer>`, forwards its verdict unchanged, and hands each decision to the sink as an `AccessRecord { actor, action, resource, decision }`. Compose it around a real policy — `set_authorizer(Box::new(Audited::new(Box::new(acl), Box::new(sink))))` — and every enforcement-point check (Subscribe read, Ops write, AwarenessSet publish) is logged at the instant it is enforced. A record carries the server-derived actor id, never the credential; and for an awareness publish it carries only the decision, never the entry's key or value.
**Why:** ARCHITECTURE §Authorization requires an audit log covering read-only accesses (connect, snapshot export, subscribe) that generate no op, alongside the writes the op log already records with actor+lamport+timestamp — and §Decision-Flow calls authorization the "single source of truth used at every enforcement point." That single seam is therefore the one place every access decision provably passes through, so a decorator on it captures the whole trail without threading a log parameter through `step`/registry (which would grow every call site and risk an enforcement point that checks but forgets to log). Logging inside `authorize` also makes drift impossible: the recorded verdict is the enforced verdict, by construction. Two invariants shaped the record: credential bytes are never logged (the actor id is the derived identity, safe and the point of the audit), and awareness stays off any log — so `AccessRecord` carries no awareness key/value, only that a publish was decided. Enforcement points that need a *distinct* action to audit — `Connect`, snapshot export, branch/version ops — are deferred with those actions/resources themselves (no `Action::Connect` or branch layer yet); the query surface over the trail (an admin/CLI concern) is a separate follow-on. The op log remains the authoritative write record; this adds the access-decision half.

## 2026-07-02 · #84 · ACL decision flow — a concrete tuple-walking `Authorizer` with explicit-deny-wins over default-deny
**Changed:** the server gains `acl::Acl`, a concrete [`Authorizer`] built from allow/deny ACL tuples and plugged in via `Registry::set_authorizer` — the first real (non-`PermitAll`) policy on the seam. A check walks every matching `(subject, action, resource)` rule: any explicit `Deny` wins outright, else any `Allow` grants, else the request is denied. A `Subject` is `Actor(id)` / `Authenticated` / `Anonymous` / `Anyone`; a rule's action is `Option<Action>` (`None` = any) and resource is `ResourceMatch::{AnyRoom, Room(name)}`. Rules are order-independent (deny-wins makes the outcome the same however they were added).
**Why:** ARCHITECTURE §Authorization/Decision-Flow specifies the exact IAM semantics — walk matching tuples, explicit deny wins, explicit allow grants, absence of a grant denies (default-deny). This slice implements those explicit-tuple steps as a standalone, doc-independent policy engine on the #82 seam; the doc-level ACL-as-CRDT layer will feed its tuples into the same evaluation once it lands. Two forced scope cuts, both dependency-gated: (1) **role/group subjects are omitted** — ARCHITECTURE has the engine read role membership from token claims ("never decides role membership itself"), but claims are not threaded through to an enforcement point yet (an actor is opaque bytes), so only subjects decidable from the actor id are built; (2) **decision-flow step 4 (schema `@auth` role grants) is skipped** — the schema layer is unbuilt. The `Authenticated`/`Anonymous` split keys on the `anon:` actor prefix #74 already mints — the only in-band marker distinguishing an anonymous connection from a credentialed one without claims. `ResourceMatch` stays an enum (not `Option<Vec<u8>>`) so it widens to path/element variants as `Resource` grows, and `ResourceMatch::matches` irrefutably destructures `Resource::Room` so a new `Resource` variant breaks the build rather than silently mismatching.

## 2026-07-02 · #83 · per-recipient read redaction — the fan-out re-checks read on every op and awareness send
**Changed:** the registry now consults the `Authorizer` on **each** broadcast, not only at Subscribe: before queuing a room's ops (or an awareness update) to a peer it checks `authorize(peer_actor, Read, Room)`, and skips any peer that no longer passes. So a peer whose read is revoked mid-session stops receiving the room's ops and presence at once, without waiting for it to resubscribe or reconnect. Seeing a peer's awareness is treated as a read of the room, so the same gate covers both fan-outs. The write still ingests for the room's permitted readers; only the redacted peer's send is dropped.
**Why:** ARCHITECTURE §Authorization requires per-recipient filtering on every op send ("unauthorized bytes never leave the server"), which the Subscribe-time check (#82) alone does not give: authorization is dynamic (a deployment's policy can revoke access at any moment), and a check made only at join leaks every subsequent op to a since-revoked reader until it happens to resubscribe. Re-checking at the fan-out is the enforcement point ARCHITECTURE names, and it is where finer resource checks (element/zone/path) slot in later — the resource is `Room` today because that is all the wire addresses, but the hook is per-send from here on. The check is the *recipient's* read right, not the writer's; the writer's write right is already enforced at Ops ingest. Doc-level ACL-as-CRDT, cold-start snapshot redaction at finer-than-room grain, and the audit log remain the larger follow-on slices atop this hook.

## 2026-07-02 · #82 · authorization decision seam — a pluggable `Authorizer` enforced at room-level access points, `Forbidden` non-closing
**Changed:** the server gains an `authz::Authorizer` trait — the read/write analog of the credential `Verifier` — that decides whether an authenticated actor may take an `Action` (`Read`/`Write`/`PublishAwareness`) on a `Resource` (`Room` today; the enum widens to path/element/branch later). `step` consults it at Subscribe (read), Ops (write), and AwarenessSet (publish); a denial replies `ErrorCode::Forbidden` (new wire tag 5) and — unlike a protocol violation — **keeps the connection open**, since a denied request is well-formed. Injectable via `Registry::set_authorizer`; the dev-mode `PermitAll` default allows everything, so existing behavior is unchanged. The contract is default-deny (an authorizer returns `true` only for what it explicitly permits).
**Why:** ARCHITECTURE §Authorization makes authz first-class with a single policy consulted at every enforcement point, default-deny, deny-wins. The full two-tier model (schema-`@auth` defaults + doc-level CRDT ACLs), per-recipient wire redaction, zones, and the audit log are gated on the unbuilt schema layer, zones, and an ACL CRDT — so this slice lands the *seam and the room-level enforcement points that exist today*, the foundation those layers refine. `Forbidden` is non-closing (distinct from `AuthFailed`, which closes) because authorization is a per-request verdict a client should survive — it may still act within what it is allowed. Room is the only `Resource` variant now because rooms are the only resource the wire currently addresses; finer resources arrive with paths/zones. Awareness per-recipient auth-filtering stays queued: it needs the richer policy (a `see` check per recipient), not just the publish gate this adds.

## 2026-07-02 · #76 · fast-path credential carriers beyond the header, with a precedence order; mTLS out of scope
**Changed:** the upgrade fast path now reads a credential from four carriers, in precedence order — `Authorization` header, then the `crdtsync.auth.<value>` WebSocket subprotocol, then the `crdtsync_credential=<value>` cookie, then the `?credential=<value>` query param. When the client offers the plain `crdtsync` subprotocol, the server echoes it in the handshake response so the client's subprotocol negotiation succeeds. mTLS is **not** implemented — the server terminates plain TCP, with no TLS layer to expose a client certificate; it stays queued for when TLS termination lands.
**Why:** ARCHITECTURE lists these carriers as deployment-pluggable, but the forcing reality is the browser: a browser `WebSocket` cannot set the `Authorization` header, so the header-only fast path (#74) was unreachable from the web — the subprotocol and query carriers are the only ones a browser controls (cookies ride along same-origin). The precedence order is a required disambiguation the spec leaves open; header-first keeps the explicit, non-browser credential authoritative when several are present. The query carrier is supported but noted logs-leak-prone (URLs land in access logs). mTLS is deferred not by choice but by dependency: there is no TLS layer yet.

## 2026-07-02 · #75 · a real `Verifier` is injectable into the runtime, not just the dev `AllowAll`
**Changed:** `serve_with_verifier(listener, server, store, config, Box<dyn Verifier + Send>)` lets a deployment plug in its identity seam (JWT/OIDC/API key); `serve`/`serve_with` keep their signatures and default to the dev `AllowAll`. The registry actor sets the verifier on its `Registry` before serving, and it derives the actor for both the in-band Auth phase and the upgrade fast path. Until now `serve` hardcoded `AllowAll`, so a real credential check was reachable only in unit tests through `Registry::set_verifier`.
**Why:** ARCHITECTURE makes the credential verifier deployment-pluggable (the engine ships no identity provider), but the runtime gave no way to supply one — the seam existed on `Registry` yet the only entry points bypassed it. The `+ Send` bound is forced: the `Registry` lives on a dedicated actor thread, so the verifier must cross the thread boundary; `Box<dyn Verifier + Send>` upcasts to the `Registry`'s `Box<dyn Verifier>` once on the far side. Additional credential carriers (cookie, WS subprotocol, mTLS, query param) remain a follow-on; the `Authorization` header carrier plus this injection make real end-to-end auth (map + reject) exercisable.

## 2026-07-02 · #73, #74 · auth fast path — credential in the `Authorization` header, unsolicited `AuthOk`, transport-minted anonymous actor
**Changed:** the handshake gains the ARCHITECTURE fast path. A credential presented at the WebSocket upgrade is verified during accept (via `accept_hdr_async` reading the `Authorization` header — the first carrier implemented of the several ARCHITECTURE lists), and the connection opens already authenticated: `Session::authenticated` / `Registry::connect_authenticated` pre-set the actor so the client skips the in-band Auth phase. A credential presented but refused closes the connection with `AuthFailed` before the loop. `serve` keeps its signature; policy rides on `ServeConfig`.
**Why:** ARCHITECTURE §Handshake specifies the fast path ("credentials present at upgrade → validate during accept → skip Phase 2, one round trip saved") and anonymous mode (`actor = "anon:<random>"`). Carriers beyond the header (cookie, subprotocol, mTLS, query param) and injecting a non-`AllowAll` verifier into `serve` remain follow-ons.

## 2026-07-02 · #72 · periodic awareness sweep wired into the runtime; timed-TTL + throttle deferred as schema-gated
**Changed:** the tokio runtime now drives `Registry::sweep` on an interval, so the reconnect-grace expiry (the session-TTL half of awareness) fires in production without an external caller — previously `sweep` was reachable only from tests. `serve` keeps its signature (5s grace, 1s sweep by default via `ServeConfig::default`); a new `serve_with(listener, server, store, ServeConfig)` exposes the grace + sweep cadence, and the registry actor selects over inbound commands and the sweep tick, flushing outboxes after each. **Why:** the grace window is only useful if something actually sweeps; the runtime is where wall time and the connection set both live. ## 2026-07-02 · #70, #71 · awareness reconnect grace via an injected server `Clock` + explicit sweep, not clear-on-disconnect
**Changed:** a disconnect no longer clears the client's ephemeral presence at once. On disconnect a client with live presence is marked *stale* with a `now + grace` deadline (default 5s); a reconnect (Hello) within the window cancels it; `Registry::sweep` clears every past-deadline client and fans a new `Message::AwarenessClear { channel, actor }` (wire tag 10, server→client) to each affected room's remaining subscribers on their own channel. **Why:** ARCHITECTURE §Awareness prescribes a reconnect grace window so a brief drop (tab reload, flaky network) doesn't flicker a user's cursor out and back for every peer. That needs wall time on the server, which only the core had — hence the `Clock` seam, injectable so tests drive presence expiry deterministically with a `ManualClock` instead of sleeping. Sweep is explicit (caller-driven) rather than a background timer so the registry stays pure/synchronous; wiring a periodic tokio sweep into the runtime is a separate follow-on. Clearing is gated on the client actually holding presence, so the common no-awareness disconnect never reads the clock (keeps the Miri-gated registry tests off the unsupported system-time call under isolation). TTL-per-entry and throttle/coalesce remain queued on the same seam.

## 2026-07-02 · rewrite · core language is Rust, not C
**Changed:** the core and server are implemented in Rust — a downward `Rc<RefCell<T>>` value graph, `#![forbid(unsafe_code)]`, Miri-gated — not C. ## 2026-07-02 · #48, #32 · displacement retains + reinstates, not orphan-and-forget
**Changed:** a displaced container/counter is kept in a persistent per-id registry and reinstated if its slot is re-won; a displaced counter keeps accumulating. Replaces "the displaced element_id may become unreachable; core surfaces an orphan event."

**Changed:** the core op the CRDT engine merges is `{id, stamp, target, kind, tx}`. Authorship (`actor_id`), scope (`room`/`branch`/`zone`), `schema_version`, and wall time are wire/server-envelope concerns wrapping the core op — not core op fields.

**Changed:** a snapshot is keyed by the server sequence it covers (`base_seq`), not a lamport timestamp, and is regenerated live from the merged replica. Catch-up returns *either* an op delta (at/above the room's compaction floor) *or* a whole-replica snapshot (below it) — never snapshot-plus-tail.
