# Design decisions

Log of design changes to [ARCHITECTURE.md](ARCHITECTURE.md) that implementation forced. ARCHITECTURE always reads as the current intended end-state; this file is *why* it changed. Newest first.

`/cs-implement` appends an entry when building a unit reveals a genuine design change — a forced realization (the design as written is wrong, impossible, or clearly worse than what the code must do), never a mere preference. The human appends when they revise scope. Format: **date · unit/PR · what changed · why**.

The entries below (2026-07-02) are a backfill: design changes made during the v0.1→v0.2 build that predate this log, recovered from the sessions and commit history.

## 2026-07-02 · #108 · error envelope — reserve the opaque `details` byte string on the wire (no producer yet)
**Changed:** `Message::Error` gains a `details: Vec<u8>` field beside `code` + `message`; the codec appends it as a length-framed byte string (tag 3 unchanged, existing fields unchanged) and decodes it totally. Every construction site sets `details: Vec::new()` — nothing populates it yet — and the core client ignores it when surfacing a server error (`ClientError::Server` stays code + message). ARCHITECTURE §Error-Envelope's as-built note is updated from "the `details` field is not yet implemented" to "reserved (round-tripped, empty)".
**Why:** ARCHITECTURE §Error-Envelope specifies the envelope as *code + human message + opaque details*, and rationalizes the closed-enum-plus-details shape for uniform cross-language error handling — but only code + message were built. Adding a wire field is cheap before adoption and a breaking change after (once SDKs decode `Error` in the wild, a new trailing field shifts the layout), so it lands now as a forward-compat reservation, the same discipline as the blob-ref value slot (#60): reserve the slot in the envelope + codec + round-trip tests, defer the producer and the SDK-surface adoption until a use exists. `details` is opaque bytes the core never parses (a client interprets it by `code`), matching how the protocol treats every other app-controlled payload (awareness key/value, version names/state). No behavior change — every error still carries the same code + message; the field rides along empty.

## 2026-07-02 · #107 · snapshot export / import — the identity-preserving move; id-namespaced clone deferred
**Changed:** the `Hub` gains `export_room(room) -> Option<Vec<u8>>` (the room's `encode_state` bytes, the portable snapshot) and `import_room(room, state) -> io::Result<bool>` (decode the state, install it as a fresh room). Import is create-only — it returns `Ok(false)` for a room that already exists rather than clobbering live state — and maps a malformed blob to `InvalidData`, never a panic. The imported room's dedup set comes back from the state (`doc.seen()`), and its compaction floor is set to the imported op count so a fresh subscriber lands below the floor and is caught up with the snapshot rather than an empty delta. With a store attached the snapshot is persisted (via the existing `store.compact` path) before the room commits, so the import survives a restart. ARCHITECTURE §Export/Import's "import bumps clocks past imported lamport; element / client IDs are namespaced" is **not** built here.
**Why:** ARCHITECTURE §Export/Import lists four use cases — backup, cloning rooms (templates), cross-server moves, debug repro. Three of them (backup, cross-server move, debug repro) are an *identity-preserving* move: the room keeps its id and every element/op/client identity, and the shipped state codec (#54) plus snapshot storage (#55–#58) already make it a thin layer — export is the same `encode_state` a cold-start subscriber is served (asserted equal in the suite), import is the same `decode_state` + install `from_rooms` already does on reload. Only **cloning under a new id** needs the clock-bump + id-namespacing step, and that is genuinely harder and unspecified (how ids are namespaced — prefixing? rewriting the encoded blob? tagging at the room boundary? — and the clock-bump semantics), and it touches the state codec to rewrite identities. So the move subset ships now and the clone is queued as design-gated, rather than blocking the whole section on the namespacing design. Import renumbers sequences from the imported op count because a server sequence is server-local (the origin's numbering has no meaning on the destination) and starting at the op count keeps `head` meaningful and monotonic while guaranteeing a below-floor subscriber is served the snapshot. Import is create-only and persist-before-commit for the same reasons the rest of the hub is: never clobber state the operator did not explicitly clear, and never advertise a room the disk has not accepted.

## 2026-07-02 · #102 · atomic transactions — count-tagged members, buffered until whole, applied together
**Changed:** the reserved op-envelope `tx` field, previously an unused `Option<TxId>`, becomes `Option<Tx>` where `Tx { id, count }` carries both the shared transaction id and the group size; the op codec round-trips the added count. `Document::atomic_transact(f)` mints a per-replica `TxId` (namespaced by the replica's client, so `(client, tx)` is globally unique) and tags every op the closure emits with the group's `Tx`. On the receive side, an atomic member is always buffered; a transaction commits only once every one of its `count` members is present *and* the group's external dependencies resolve, at which point all members apply together in op-seq order. A single-member atomic transaction commits immediately. The author applies its own edits eagerly, as with any local edit.
**Why:** ARCHITECTURE §Transactions specifies opt-in atomic transactions as an all-or-nothing *view* boundary ("receivers buffer member ops until commit marker arrives; on commit, all apply atomically"). The wire envelope already reserved `tx` (round-tripped since the op codec shipped) but nothing set it and no atomic behavior existed. The "commit marker" is realized as a **per-member count** rather than a separate terminator op: it makes each member self-describing, so completeness is detectable regardless of delivery order or splitting across messages, with no distinct marker to lose or order. Atomicity is enforced purely as a *buffering gate* — the same ops still merge, so convergence is unchanged by construction (an atomic author and a non-atomic peer reach identical state; verified against the convergence property harness), and the only observable difference is that no peer sees a partial group. The group commits in op-seq order so a member targeting a container an earlier member creates lands after it; the readiness gate treats intra-group creates/inserts as satisfied (they will exist once the in-order apply reaches them) and only blocks on dependencies *external* to the transaction, so a self-contained tx (create-then-populate, insert-then-delete) never deadlocks. Buffered partial transactions ride the existing op-buffer, so they persist in a snapshot and complete after a restart for free. `next_tx` is not persisted: tx ids scope only the buffering of *remote* partials (keyed by the author's client), and a restored replica mints under its own client with fresh op ids, so restarting the counter at 0 cannot collide with anything buffered. This is the core slice; the wire/server/client drive path and the SDK `atomic_transact` surface are the follow-on slices. Atomic-tx *undo* (ARCHITECTURE §Transactions "a transaction is naturally an undo intention") and scope constraints tied to branches/zones/schema remain gated on those layers.

## 2026-07-02 · #92 · named versions in the language SDKs — Python, Go, and wasm wrap the version surface
**Changed:** the Python (`Client`), Go (`Client`), and wasm (`WasmClient`) SDKs gain the versioning surface, closing the arc: `create_version` / `rename_version` / `delete_version` / `list_versions` / `fetch_version` return the request frame to send, and `versions(channel)` (the name list) + `version_state(channel, name)` (a fetched state) expose the view. Python and Go bind the C ABI added in #91 (Python reads the name list via `version_count` + indexed `version_name`; Go the same); wasm wraps `ClientSession` directly and returns the name list as `Vec<Uint8Array>`.
**Why:** the closing layer of the versioning feature — every SDK can now drive named versions, not just the local `Document` — mirroring how the awareness surface reached all four bindings (#79–#81). The per-language view accessor matches each ABI's idiom: over the C ABI (Python/Go) the list is count + indexed accessor (no variable-length-list convention in C); over wasm-bindgen the list is a native `Vec<Uint8Array>`. No new design — a straight marshal of #90/#91. The SDK tests assert the requests marshal and the view is empty before any server reply (a server round-trip is exercised in the Rust suites #89/#90), since the SDKs test the client seat in isolation. This completes named versions end to end (index → durable → wire → server → client → SDKs); restore-as-branch and auto-version triggers remain the versioning follow-ons, both dependency-gated.

## 2026-07-02 · #91 · FFI named-version surface — the `CrdtClient` C ABI exposes the version issue methods and view
**Changed:** the `CrdtClient` C ABI gains the versioning surface over `ClientSession` (#90): `crdtsync_client_{create,rename,delete,list,fetch}_version` frame the request as a `CrdtBuf` to send (empty on a bad handle / input / unheld channel, like the awareness framers), and the view is read with `crdtsync_client_version_count` + `crdtsync_client_version_name(index)` for the name list and `crdtsync_client_version_state(name)` for a fetched state (1 present / 0 absent / -1 bad handle, like the awareness readers). The generated header and its symbol assertions cover the eight new entry points.
**Why:** the SDK half of versioning, letting an embedder drive it end-to-end, mirroring how the awareness surface (#77–#81) crossed the boundary. The name list crosses as **count + indexed name** rather than one packed buffer because the C ABI has no variable-length-list convention — the same shape as `awareness_len` plus an accessor, so every language binding reads it with a simple loop rather than a bespoke length-prefixed decoder. A fetched state crosses verbatim as a `CrdtBuf` (the client caches raw bytes; only the embedder decodes it via the doc state codec). Python / Go / wasm layer over these entry points next, closing the versioning arc.

## 2026-07-02 · #90 · client version view — `ClientSession` frames version requests and folds the replies into a per-room view
**Changed:** `core::client::ClientSession` gains the client seat of versioning: `create_version` / `rename_version` / `delete_version` / `list_versions` / `fetch_version` return the request frame for a held channel (`None` otherwise, like `set_awareness`), and `receive` folds the replies into a per-room view — a `Versions` list replaces the room's known names, a `VersionState` caches the fetched bytes under its name. Getters `versions(channel)` and `version_state(channel, name)` expose the view; a version reply for an unheld channel is refused with `UnknownChannel`.
**Why:** the client mirror of the server handling (#89), completing the drive-it-from-a-replica path over the wire (#88). The `Versions` list *replaces* rather than merges because the server's reply is the authoritative post-state — the same reason the server sends the whole list after every mutation (#88 decision); merging would let a locally-stale name linger after another client deleted it. Fetched states are cached by name (not auto-evicted) because a fetch is an explicit pull for export/diff and the embedder decides its lifetime; the view holds the last bytes fetched. Issue methods return the frame for the caller to send rather than mutating anything locally — versioning is server-authoritative, so the client holds no optimistic version state. The FFI/Python/Go/wasm surface over these methods is the closing slice of the arc.

## 2026-07-02 · #89 · server handling of version requests — create/rename/delete gated on write, list/fetch on read
**Changed:** `step` now serves the five version requests (#88) on the channel's subscribed room: `VersionCreate`/`VersionRename`/`VersionDelete` run the matching `Hub` mutation and reply with the fresh `Versions` name list; `VersionList` replies the same; `VersionFetch` replies `VersionState` when the version exists and falls back to the `Versions` list when it does not. Mutations are gated on `Action::Write`, reads on `Action::Read`, both against `Resource::Room` — a denial is a non-closing `Forbidden`, an unbound channel or unauthenticated connection is a closing violation, and a persist failure is a closing `Internal`. The replies go only to the requester (no fan-out), which the registry already routes.
**Why:** this is the server half of the versioning sub-protocol whose wire landed in #88; it reuses the existing authorization seam rather than adding version-specific actions — creating/renaming/deleting a version is a write to the room and listing/fetching is a read, which the room-level `Action` set already expresses. ARCHITECTURE §Authorization enumerates *version create / restore / delete* as their own enforcement points, so finer version-scoped actions (and per-version ACLs) are a later refinement once the resource model widens beyond `Room`; mapping to Write/Read now keeps versioning behind the same default-deny policy every other operation obeys, with no gap. A failed persist closes with `Internal` (mirroring the Ops ingest path) rather than replying a stale list, so the client never sees a post-state the disk did not accept. The client `ClientSession` view + issue methods and the language SDK surface remain the closing slices of the arc.

## 2026-07-02 · #88 · named-version wire messages — a request/response sub-protocol (codec only; handling to follow)
**Changed:** the wire protocol gains seven messages for versioning (tags 11–17, existing tags unchanged): client→server `VersionCreate{channel,name}`, `VersionRename{channel,from,to}`, `VersionDelete{channel,name}`, `VersionList{channel}`, `VersionFetch{channel,name}`; server→client `Versions{channel,names}` and `VersionState{channel,name,seq,state}`. Names and state are opaque byte strings the core never parses; decoding stays total. This slice ships the wire + codec + round-trip suite only — `step`/`receive` gain arms that keep them compiling (the server treats a version request as not-yet-served and a version *response* from a client as a violation; the core client treats version responses as not-yet-handled), with the server handling and SDK surface in the following slices.
**Why:** ARCHITECTURE §Named-Versions makes list/rename/delete/fetch first-class but does not pin their wire shape, so the sub-protocol is a design choice. It is modeled request/response over the channel's already-subscribed room (versions are per-room, no fan-out). The **acknowledgement is state-based, not an RPC ok/err code**: a mutation's reply is the fresh `Versions` name list (the authoritative post-state), matching the rest of the protocol where the server answers with state (Ops/Snapshot) rather than status codes — a refused create (name taken) simply yields the unchanged list, and there is no `VersionState` for a fetch that misses (the caller gets the list, learning the name is absent). The `Versions` decoder grows its vector as records are read rather than pre-sizing from the wire count, so a bogus count fails on the missing bytes instead of over-allocating — the same total-decode discipline the op codec follows. Wire-first mirrors how awareness landed (#66): codec as an isolated, low-ripple slice before the server and client behavior.

## 2026-07-02 · #87 · durable named versions — the versions index persists to its own per-room store file, crash-atomically
**Changed:** the versions index (#86) is now durable. Each room's versions live in a `<room>.versions` store file — a sequence of `(name, covered seq, state)` records (length-prefixed name, 8-byte LE sequence, length-prefixed state) — and `RoomLog` gains a `versions` field the loader populates, which `Hub::install_room` folds back into the in-memory index on reopen. `create_version` / `rename_version` / `delete_version` now return `io::Result<bool>`: with a store attached they rewrite the whole index file before committing the change in memory, rolling the in-memory mutation back if the write fails, so a reopened hub reproduces exactly the versions the writer held. Deleting the last version removes the file.
**Why:** ARCHITECTURE §Named-Versions says versions are "retained until the app deletes" — durability is part of the contract, and #86 explicitly left it as the follow-on. The index is rewritten in full on every change (rather than appended) because, unlike the op log, the *set* of versions is mutable (rename/delete), so an append-only encoding would need tombstones and compaction of its own; a whole-file atomic rewrite (temp → fsync → rename → dir-fsync, the same crash-safe dance compaction already uses) is simpler and correct, and versions change rarely enough that rewriting the index is cheap. The methods became `io::Result<bool>` (mirroring `ingest`'s persist-before-commit contract) with in-memory rollback on IO failure, so the index never reflects a version the disk rejected. The versions file is parsed strictly — any framing shortfall is corruption, not a tolerable torn tail — because the atomic rewrite means it is never left half-written (the op log tolerates a torn tail only because it is append-only). Restore-as-branch, auto-triggers, and the wire/SDK surface remain the outstanding versioning slices.

## 2026-07-02 · #86 · named versions — an in-memory versions index over the snapshot primitive; restore-as-branch and durability deferred
**Changed:** the `Hub` gains a named-versions layer over the existing snapshot storage primitive: `create_version(room, name)` captures the room's current whole-replica `encode_state` and the server sequence it covered under an app-chosen name; `version_seq` / `version_state` read it back (for read / export / diff); `version_names` lists them sorted (a `BTreeMap` per room) for pagination; `rename_version` / `delete_version` complete the first-class index operations. A version is immutable and point-in-time — later edits never touch it, a taken name is refused (no silent overwrite), and rename refuses an absent source or a taken target. Versioning an unknown room captures nothing.
**Why:** ARCHITECTURE §Named-Versions specifies "snapshot + entry in a versions index" with list/paginate/rename/delete first-class — all of which the shipped state codec (#54) and snapshot machinery (#55–#58) already make buildable, so this lands the index layer directly on them. Two parts of the versioning section are deliberately *not* built, each dependency-gated: **restore** is restore-as-branch (ARCHITECTURE §Restore-as-Branch forks a new branch from the chosen snapshot and switches HEAD — it explicitly does *not* rewrite live state), so it needs the unbuilt branch layer; the API therefore exposes reading a version's state but no mutating restore. And **durable persistence** of the index (versions are "retained until the app deletes") plus **auto-version triggers** (engine-event/schedule-driven creation) are follow-ons: the `Store` persists the op log + compaction snapshot but has no versions record yet, so the index is in-memory for now — the same hub-first / store-later split the compaction arc used. Wire messages + SDK surface for versions are the next slices once the index shape is proven. `BTreeMap` keys the index so listing is sorted for pagination without a separate sort at the query surface.

## 2026-07-02 · #85 · access log — an `Audited` authorizer decorator records every authorization decision at the seam
**Changed:** the server gains `audit::{AccessLog, AccessRecord, Audited, Decision}`. `AccessLog` is a pluggable sink (with a blanket closure impl) a deployment supplies; `Audited` wraps an inner `Box<dyn Authorizer>`, forwards its verdict unchanged, and hands each decision to the sink as an `AccessRecord { actor, action, resource, decision }`. Compose it around a real policy — `set_authorizer(Box::new(Audited::new(Box::new(acl), Box::new(sink))))` — and every enforcement-point check (Subscribe read, Ops write, AwarenessSet publish) is logged at the instant it is enforced. A record carries the server-derived actor id, never the credential; and for an awareness publish it carries only the decision, never the entry's key or value.
**Why:** ARCHITECTURE §Authorization requires an audit log covering read-only accesses (connect, snapshot export, subscribe) that generate no op, alongside the writes the op log already records with actor+lamport+timestamp — and §Decision-Flow calls authorization the "single source of truth used at every enforcement point." That single seam is therefore the one place every access decision provably passes through, so a decorator on it captures the whole trail without threading a log parameter through `step`/registry (which would grow every call site and risk an enforcement point that checks but forgets to log). Logging inside `authorize` also makes drift impossible: the recorded verdict is the enforced verdict, by construction. Two invariants shaped the record: credential bytes are never logged (the actor id is the derived identity, safe and the point of the audit), and awareness stays off any log — so `AccessRecord` carries no awareness key/value, only that a publish was decided. Enforcement points that need a *distinct* action to audit — `Connect`, snapshot export, branch/version ops — are deferred with those actions/resources themselves (no `Action::Connect` or branch layer yet); the query surface over the trail (an admin/CLI concern) is a separate follow-on. The op log remains the authoritative write record; this adds the access-decision half.

## 2026-07-02 · #84 · ACL decision flow — a concrete tuple-walking `Authorizer` with explicit-deny-wins over default-deny
**Changed:** the server gains `acl::Acl`, a concrete [`Authorizer`] built from allow/deny ACL tuples and plugged in via `Registry::set_authorizer` — the first real (non-`PermitAll`) policy on the seam. A check walks every matching `(subject, action, resource)` rule: any explicit `Deny` wins outright, else any `Allow` grants, else the request is denied. A `Subject` is `Actor(id)` / `Authenticated` / `Anonymous` / `Anyone`; a rule's action is `Option<Action>` (`None` = any) and resource is `ResourceMatch::{AnyRoom, Room(name)}`. Rules are order-independent (deny-wins makes the outcome the same however they were added).
**Why:** ARCHITECTURE §Authorization/Decision-Flow specifies the exact IAM semantics — walk matching tuples, explicit deny wins, explicit allow grants, absence of a grant denies (default-deny). This slice implements those explicit-tuple steps as a standalone, doc-independent policy engine on the #82 seam; the doc-level ACL-as-CRDT layer will feed its tuples into the same evaluation once it lands. Two forced scope cuts, both dependency-gated: (1) **role/group subjects are omitted** — ARCHITECTURE has the engine read role membership from token claims ("never decides role membership itself"), but claims are not threaded through to an enforcement point yet (an actor is opaque bytes), so only subjects decidable from the actor id are built; (2) **decision-flow step 4 (schema `@auth` role grants) is skipped** — the schema layer is unbuilt. The `Authenticated`/`Anonymous` split keys on the `anon:` actor prefix #74 already mints — the only in-band marker distinguishing an anonymous connection from a credentialed one without claims. `ResourceMatch` stays an enum (not `Option<Vec<u8>>`) so it widens to path/element variants as `Resource` grows, and `ResourceMatch::matches` irrefutably destructures `Resource::Room` so a new `Resource` variant breaks the build rather than silently mismatching.

## 2026-07-02 · #83 · per-recipient read redaction — the fan-out re-checks read on every op and awareness send
**Changed:** the registry now consults the `Authorizer` on **each** broadcast, not only at Subscribe: before queuing a room's ops (or an awareness update) to a peer it checks `authorize(peer_actor, Read, Room)`, and skips any peer that no longer passes. So a peer whose read is revoked mid-session stops receiving the room's ops and presence at once, without waiting for it to resubscribe or reconnect. Seeing a peer's awareness is treated as a read of the room, so the same gate covers both fan-outs. The write still ingests for the room's permitted readers; only the redacted peer's send is dropped.
**Why:** ARCHITECTURE §Authorization requires per-recipient filtering on every op send ("unauthorized bytes never leave the server"), which the Subscribe-time check (#82) alone does not give: authorization is dynamic (a deployment's policy can revoke access at any moment), and a check made only at join leaks every subsequent op to a since-revoked reader until it happens to resubscribe. Re-checking at the fan-out is the enforcement point ARCHITECTURE names, and it is where finer resource checks (element/zone/path) slot in later — the resource is `Room` today because that is all the wire addresses, but the hook is per-send from here on. The check is the *recipient's* read right, not the writer's; the writer's write right is already enforced at Ops ingest. Doc-level ACL-as-CRDT, cold-start snapshot redaction at finer-than-room grain, and the audit log remain the larger follow-on slices atop this hook.

## 2026-07-02 · #82 · authorization decision seam — a pluggable `Authorizer` enforced at room-level access points, `Forbidden` non-closing
**Changed:** the server gains an `authz::Authorizer` trait — the read/write analog of the credential `Verifier` — that decides whether an authenticated actor may take an `Action` (`Read`/`Write`/`PublishAwareness`) on a `Resource` (`Room` today; the enum widens to path/element/branch later). `step` consults it at Subscribe (read), Ops (write), and AwarenessSet (publish); a denial replies `ErrorCode::Forbidden` (new wire tag 5) and — unlike a protocol violation — **keeps the connection open**, since a denied request is well-formed. Injectable via `Registry::set_authorizer`; the dev-mode `PermitAll` default allows everything, so existing behavior is unchanged. The contract is default-deny (an authorizer returns `true` only for what it explicitly permits).
**Why:** ARCHITECTURE §Authorization makes authz first-class with a single policy consulted at every enforcement point, default-deny, deny-wins. The full two-tier model (schema-`@auth` defaults + doc-level CRDT ACLs), per-recipient wire redaction, zones, and the audit log are gated on the unbuilt schema layer, zones, and an ACL CRDT — so this slice lands the *seam and the room-level enforcement points that exist today*, the foundation those layers refine. `Forbidden` is non-closing (distinct from `AuthFailed`, which closes) because authorization is a per-request verdict a client should survive — it may still act within what it is allowed. Room is the only `Resource` variant now because rooms are the only resource the wire currently addresses; finer resources arrive with paths/zones. Awareness per-recipient auth-filtering stays queued: it needs the richer policy (a `see` check per recipient), not just the publish gate this adds.

## 2026-07-02 · #76 · fast-path credential carriers beyond the header, with a precedence order; mTLS out of scope
**Changed:** the upgrade fast path now reads a credential from four carriers, in precedence order — `Authorization` header, then the `crdtsync.auth.<value>` WebSocket subprotocol, then the `crdtsync_credential=<value>` cookie, then the `?credential=<value>` query param. When the client offers the plain `crdtsync` subprotocol, the server echoes it in the handshake response so the client's subprotocol negotiation succeeds. mTLS is **not** implemented — the server terminates plain TCP, with no TLS layer to expose a client certificate; it stays queued for when TLS termination lands.
**Why:** ARCHITECTURE lists these carriers as deployment-pluggable, but the forcing reality is the browser: a browser `WebSocket` cannot set the `Authorization` header, so the header-only fast path (#74) was unreachable from the web — the subprotocol and query carriers are the only ones a browser controls (cookies ride along same-origin). The precedence order is a required disambiguation the spec leaves open; header-first keeps the explicit, non-browser credential authoritative when several are present. The query carrier is supported but noted logs-leak-prone (URLs land in access logs). mTLS is deferred not by choice but by dependency: there is no TLS layer yet.

## 2026-07-02 · #75 · a real `Verifier` is injectable into the runtime, not just the dev `AllowAll`
**Changed:** `serve_with_verifier(listener, server, store, config, Box<dyn Verifier + Send>)` lets a deployment plug in its identity seam (JWT/OIDC/API key); `serve`/`serve_with` keep their signatures and default to the dev `AllowAll`. The registry actor sets the verifier on its `Registry` before serving, and it derives the actor for both the in-band Auth phase and the upgrade fast path. Until now `serve` hardcoded `AllowAll`, so a real credential check was reachable only in unit tests through `Registry::set_verifier`.
**Why:** ARCHITECTURE makes the credential verifier deployment-pluggable (the engine ships no identity provider), but the runtime gave no way to supply one — the seam existed on `Registry` yet the only entry points bypassed it. The `+ Send` bound is forced: the `Registry` lives on a dedicated actor thread, so the verifier must cross the thread boundary; `Box<dyn Verifier + Send>` upcasts to the `Registry`'s `Box<dyn Verifier>` once on the far side. Additional credential carriers (cookie, WS subprotocol, mTLS, query param) remain a follow-on; the `Authorization` header carrier plus this injection make real end-to-end auth (map + reject) exercisable.

## 2026-07-02 · #73, #74 · auth fast path — credential in the `Authorization` header, unsolicited `AuthOk`, transport-minted anonymous actor
**Changed:** the handshake gains the ARCHITECTURE fast path. A credential presented at the WebSocket upgrade is verified during accept (via `accept_hdr_async` reading the `Authorization` header — the first carrier implemented of the several ARCHITECTURE lists), and the connection opens already authenticated: `Session::authenticated` / `Registry::connect_authenticated` pre-set the actor so the client skips the in-band Auth phase. The server sends an **unsolicited `AuthOk`** (after version negotiation, before the message loop) so a fast-path or anonymous client still learns its server-derived actor without having sent Auth. Anonymous mode (`ServeConfig::anonymous`, off by default) admits a credential-less connection by minting `actor = anon:<128 random bits, hex>` from system entropy. A credential presented but refused closes the connection with `AuthFailed` before the loop. `serve` keeps its signature; policy rides on `ServeConfig`.
**Why:** ARCHITECTURE §Handshake specifies the fast path ("credentials present at upgrade → validate during accept → skip Phase 2, one round trip saved") and anonymous mode (`actor = "anon:<random>"`). The unsolicited `AuthOk` is the forced detail ARCHITECTURE leaves implicit: the client never asserts its actor, so on the fast path — where it sends no Auth — the server must volunteer the actor or the client would never know its identity. Entropy is kept at the transport layer (a `getrandom` dep on the server, not the core) to preserve the core's deliberate no-entropy stance (`Host` seam) and its Miri-gated purity; the registry stays entropy-free so its logic tests remain Miri-clean under isolation. Carriers beyond the header (cookie, subprotocol, mTLS, query param) and injecting a non-`AllowAll` verifier into `serve` remain follow-ons.

## 2026-07-02 · #72 · periodic awareness sweep wired into the runtime; timed-TTL + throttle deferred as schema-gated
**Changed:** the tokio runtime now drives `Registry::sweep` on an interval, so the reconnect-grace expiry (the session-TTL half of awareness) fires in production without an external caller — previously `sweep` was reachable only from tests. `serve` keeps its signature (5s grace, 1s sweep by default via `ServeConfig::default`); a new `serve_with(listener, server, store, ServeConfig)` exposes the grace + sweep cadence, and the registry actor selects over inbound commands and the sweep tick, flushing outboxes after each. The registry runtime enables tokio time. **Deferred, not built:** per-entry *timed* TTL (auto-expire after silence) and server-side throttle/coalesce — ARCHITECTURE §Awareness makes both **schema-declared** (an entry's type carries its TTL and throttle interval), and the schema layer is unbuilt, so their trigger values have no home yet. Building them now would either invent a wire field ARCHITECTURE doesn't sanction or hardcode a policy the schema is meant to own.
**Why:** the grace window is only useful if something actually sweeps; the runtime is where wall time and the connection set both live. Splitting `serve`/`serve_with` keeps the binary + existing callers untouched while letting tests drive expiry on a sub-second cadence instead of waiting the 5s default. Recognizing timed-TTL/throttle as schema-gated (rather than clock-gated) is the real finding: the clock seam (#71) unblocked *session* TTL, but the remaining two await schema — recorded so they are not attempted prematurely.

## 2026-07-02 · #70, #71 · awareness reconnect grace via an injected server `Clock` + explicit sweep, not clear-on-disconnect
**Changed:** a disconnect no longer clears the client's ephemeral presence at once. The server gains a `Clock` seam (`clock::{Clock, SystemClock, ManualClock}`, wall-millis) — the analog of the core's `Host::now`, which the server runtime lacked. On disconnect a client with live presence is marked *stale* with a `now + grace` deadline (default 5s); a reconnect (Hello) within the window cancels it; `Registry::sweep` clears every past-deadline client and fans a new `Message::AwarenessClear { channel, actor }` (wire tag 10, server→client) to each affected room's remaining subscribers on their own channel. Replaces #69's synchronous `Hub::clear_client_awareness` on disconnect; that method now *returns* the cleared `(room, actor)` pairs so the sweep can address the fan-out.
**Why:** ARCHITECTURE §Awareness prescribes a reconnect grace window so a brief drop (tab reload, flaky network) doesn't flicker a user's cursor out and back for every peer. That needs wall time on the server, which only the core had — hence the `Clock` seam, injectable so tests drive presence expiry deterministically with a `ManualClock` instead of sleeping. Sweep is explicit (caller-driven) rather than a background timer so the registry stays pure/synchronous; wiring a periodic tokio sweep into the runtime is a separate follow-on. Clearing is gated on the client actually holding presence, so the common no-awareness disconnect never reads the clock (keeps the Miri-gated registry tests off the unsupported system-time call under isolation). TTL-per-entry and throttle/coalesce remain queued on the same seam.

## 2026-07-02 · #63 · handshake auth — opaque credential in, server-derived actor out, pluggable verifier
**Changed:** the handshake gains an Auth phase between Hello and Subscribe. Wire: `Message::Auth { credential: Vec<u8> }` (tag 6, client→server) and `Message::AuthOk { actor: Vec<u8> }` (tag 7, server→client); `ErrorCode::AuthFailed` (already reserved) reports rejection. `credential` and `actor` are opaque byte strings the core never parses. Server-side, a pluggable `Verifier { verify(&[u8]) -> Option<Vec<u8>> }` maps a credential to an actor id (or rejects); the session enforces Hello → Auth → Subscribe and never trusts a client-asserted actor. **Why:** ARCHITECTURE makes `actor_id` mandatory from v0.1 and the credential carrier deployment-pluggable; keeping credential/actor opaque at the core keeps the engine identity-provider-agnostic (apps bring JWT/OIDC/custom), and deriving the actor server-side is the security invariant — a client must not be able to assert who it is. ## 2026-07-02 · #61 · multi-room multiplexing keyed on a client-assigned channel id, not a per-message room tag
**Changed:** a connection carries several room subscriptions at once, each on a connection-local `Channel(u32)` the client assigns at Subscribe. `Message::Subscribe`, `Ops`, and `Snapshot` gained a `channel`; a new `Message::Unsubscribe { channel }` frees one. The server session holds `channels: HashMap<Channel, RoomId>`; the registry fans a broadcast out to each peer on the channel that peer opened for the room. The Hub stays room-keyed — the session translates channel↔room. Replaces "a connection binds a single room; a second Subscribe switches it." `ClientSession` (the SDK-side driver) stays single-room for now, pinned to `Channel(0)`; its multi-room extension is a separate unit.
**Why:** ARCHITECTURE prescribes "logical channels multiplexed per (room, branch, zone)", so the routing key is a channel, not the room bytes. A compact per-connection id keeps op batches small and — the deciding factor — lets a subscription later widen from room to (room, branch, zone) with no wire change, whereas a literal room tag would force re-versioning the frame when branches/zones land.

## 2026-07-02 · #60 · blob-ref reserved as a `Scalar` variant, not a separate `Value` enum
**Changed:** the blob-ref value slot is `Scalar::BlobRef(BlobRef)` (value-codec tag 4), sitting alongside the other leaf values, rather than a distinct `Value = Scalar | BlobRef | ElementRef` payload type. `BlobRef { id: [u8;16], mime, size, inline: Option<Vec<u8>> }`. Slot-only: the blob store, dedup, presigned fetch, and GC still land v0.5. The element-ref value type is left unreserved (no v0.1 promise, shape under-specified).
**Why:** a blob ref merges as an LWW replace on assignment — exactly a leaf value's semantics — so it is the true analog of the `tx` slot reservation: one codec tag, near-zero churn, no refactor of `RegisterSet`/`MapSet`/`ListInsert`. ARCHITECTURE's scalar/blob-ref/element-ref taxonomy distinguishes *merge semantics*, which `Scalar` already models; it is not a mandate for a distinct Rust enum. Trade-off: `Scalar` is documented "no id" in the CRDT-identity sense — `BlobRef`'s handle is opaque payload data (like `Bytes`), carries no CRDT identity, and does not merge or displace, so it still reads as a value, not an entity.

## 2026-07-02 · rewrite · core language is Rust, not C
**Changed:** the core and server are implemented in Rust — a downward `Rc<RefCell<T>>` value graph, `#![forbid(unsafe_code)]`, Miri-gated — not C. Export surface is unchanged: a stable C ABI (cbindgen) for native SDKs + wasm (wasm-bindgen) for the browser. Host seam is `entropy()` + `now()` only; `std`, not `no_std`.
**Why:** the C core's manual refcount/arena lifetime management was error-prone ("memory management was hell"); Rust removes the use-after-free / double-free / aliasing hazard class at compile time while keeping the same portability. Decided at the cheapest moment — only the primitives existed.

## 2026-07-02 · #48, #32 · displacement retains + reinstates, not orphan-and-forget
**Changed:** a displaced container/counter is kept in a persistent per-id registry and reinstated if its slot is re-won; a displaced counter keeps accumulating. Replaces "the displaced element_id may become unreachable; core surfaces an orphan event."
**Why:** orphan-and-forget diverges — two replicas that saw the same ops disagreed on a counter's value across displace-then-recreate. Retention is a convergence requirement, not a nicety. The orphan event still fires for the app; the state itself is retained.

## 2026-07-02 · core op vs. wire envelope
**Changed:** the core op the CRDT engine merges is `{id, stamp, target, kind, tx}`. Authorship (`actor_id`), scope (`room`/`branch`/`zone`), `schema_version`, and wall time are wire/server-envelope concerns wrapping the core op — not core op fields.
**Why:** keeps the CRDT engine pure and portable; the envelope is layered at the wire/server boundary, which is where those fields are actually consumed.

## 2026-07-02 · #52–#56 · snapshot keyed on server seq; cold-start is snapshot-or-delta, no tail
**Changed:** a snapshot is keyed by the server sequence it covers (`base_seq`), not a lamport timestamp, and is regenerated live from the merged replica. Catch-up returns *either* an op delta (at/above the room's compaction floor) *or* a whole-replica snapshot (below it) — never snapshot-plus-tail.
**Why:** the server sequence is what the durable log truncates on; regenerating from the live doc is always current, so a stored-snapshot-plus-tail merge is unnecessary. *Revisit* (KANBAN): O(state) CPU per below-floor cold-start — cache the encoded snapshot if snapshots grow.

## 2026-07-02 · #33, #40 · one custom binary codec, shared by wire and log
**Changed:** a single deterministic little-endian, length-framed, total-decode codec — not CBOR/MessagePack — reused for both the durable op-log and the wire envelope. The 8-byte header (`"CRDT"` magic + version) reserves the version for future codec negotiation.
**Why:** a closed op enum doesn't benefit from a self-describing format; one codec is one code path and one test surface. Negotiation stays possible via the reserved version field.

## 2026-07-02 · #47, #56 · persistence is a per-room append-only file log, not SQLite
**Changed:** the store is `<room>.log` (length-framed ops) + `<room>.snap` (compaction snapshot); crash-safety is hand-rolled (append flushes before return; compaction is temp → fsync → rename → dir fsync → truncate, with dedup-on-replay). No SQLite, no relational tables.
**Why:** the op stream is pure append-only sequential — relational storage buys nothing on the hot path and drops a dependency. *Revisit* (KANBAN): the metadata/query side (admin UI, op-log viewer, audit) may still want an embedded DB (SQLite/redb).

## 2026-07-02 · element_id derives from (parent, key, kind)
**Changed:** the derivation tuple includes the element kind, so a type-flip on a slot yields a different element_id. (ARCHITECTURE previously wrote the tuple as `(parent_id, key)`.)
**Why:** two differently-typed elements at the same slot are genuinely different elements; the kind in the tuple drives the displacement/orphan path correctly.
