# crdtsync — worklist

**Derived from [ARCHITECTURE.md](ARCHITECTURE.md).** ARCHITECTURE is the end-state — the full scope + design, everything meant to be built *eventually*. This board is the **prioritized breakdown of what's not built yet**: a rolling queue `cs-next` cuts from ARCHITECTURE and **refills as it drains**. It is regenerable — if the board and the code disagree, the code wins. Status lives here + in the code, never in ARCHITECTURE; design changes that implementation forced are logged in [DECISIONS.md](DECISIONS.md).

`/cs-next` reads this + the dependency graph, replenishes the queue from ARCHITECTURE when it's thin, breaks work into units, and hands the next to `/cs-implement`. **Test suites are the spec** — a unit is "done" when its suite is green + Miri-clean, merged to `main`. Breakdown + prioritization is autonomous; the human only edits ARCHITECTURE.

## Dependency order

Bottom-up; never advance a unit whose dependencies are red.

```
host → stamp → clientid → elementid → scalar → counter → register → element → map
                                                    ↓
                          op envelope → doc (transact/apply, buffering, displacement)
                                                    ↓
                       list (Fugue) → text (codepoint) → binary codec
                                                    ↓
              wire framing → room hub → session driver → connection registry → websocket
                                                    ↓
                            persistence → state codec → compaction → client session
                                                    ↓
                                    SDKs (FFI / wasm / Python / Go)
```

Element + Map are one coupled unit (Map slots hold Elements; Element forwards lifecycle to composites).

**Zones** — **Zones Unit 3 — zone-scoped subscription + per-zone replication streams + wire redaction (server) — DONE (#278).** Turns the Unit-2 op-envelope `zone` substrate into actual per-zone STREAMS: **with this the zone isolation model is functionally real** (declaration → resolution → enforcement → per-zone streams). `Message::Subscribe` gains a `zone: Vec<u8>` selector (empty = whole room = every zone the actor may read + root; a name = that partition + root; a no-`zones` room ignores it — one implicit root partition, byte-identical to today). `Resource::Zone { room, zone }` gates a zone subscription; a nonexistent name and an unauthorized zone are refused with the **same** generic `forbidden("read denied")`, so a refusal never confirms the zone exists (existence itself concealed). Zone default-visible within a readable room; an explicit per-zone deny carves the partition out — expressible on the shipped `Acl` via a new `ResourceMatch::Zone`. Streams are **one channel per room + a per-recipient `op.zone` filter** at the exact seam doc-ACL redaction / awareness auth-filter use (not a channel-per-zone): fan-out, catch-up delta, and cold-start snapshot each redact by partition so an unauthorized zone's ops/state/structure/count/existence — and any clock jump (per-zone clocks) — are **wholly absent**, not redacted-but-present. Snapshot to a zone-limited subscriber is projected (`Document::project_zones`) to its authorized partitions. **Refines Unit 2:** a container-create now belongs to the created *child's* zone (`zone_of_op`), not the parent it targets — so a zone owns its root container's creation and it is hidden from an unauthorized subscriber on every seam, closing a stream↔snapshot divergence (Unit-2 clock specs updated); `MapCursor::map`/`into_map` also stop re-emitting a redundant `MapCreate` for a live map. Server-plus-one-core-frame; specs `crates/server/tests/zone_streams.rs` (9) + updated `crates/core/tests/zone_lamport.rs`; local review (4 finder angles — fixed the create divergence + the shipped-Acl gap; documented the reachable-only-projection / doc-level-op-on-stream first-cut limitations). Design in DECISIONS (2026-07-14). **Zones Unit 5 (SDK subscribe surface) is the remaining zone work; Unit 4 (cross-zone token) stays deferred.** → *Authorization/zones*.

**Zones** — **Zones Unit 2 — per-zone lamport clocks + op-envelope `zone` dimension (core) — DONE (#276).** The core substrate for per-zone replication streams (Unit 3). `Op` gains `zone: Option<u32>` — a **compact zone id** (the target's index in the order-preserving `Schema::zones()`, `None` = the root/unzoned partition, `zone::zone_id_of`; `zone_of` now delegates to it). **Corrects the board's claim the field was "already reserved" — it was not on `Op`; this unit adds it** (pre-release op-wire/state break, no migration). The Document's lamport allocation is **partitioned**: the root rides `lamport`, declared zones ride `zone_clocks: HashMap<u32,u64>`. Stamp keeps its `(lamport,client)` shape — uniqueness holds within a partition, which suffices because a reachable container's zone is structurally fixed and cross-zone moves/anchors are forbidden, so no two ops that could collide on a stamp-derived id ever span zones. **Webhooks** — **Webhooks — outbound `EventSink` over the engine event bus (server) — DONE (#253).** Realizes the §Versioning "event bus = webhook substrate" item — the event bus (#191) built for exactly this. `WebhookSink` (`crates/server/src/webhook.rs`) is the second built-in `EventSink` after `AutoVersionSink`: on each room-bearing lifecycle event it serializes a JSON body and `try_send`s it to a bounded (1024) mpsc queue that a separate spawned reqwest worker drains, POSTing **off the hot path** — `on_event` never blocks or panics the hub commit path, so a slow/failing/absent endpoint can't stall emission or the op path. **Delivery = best-effort, at-most-once, no cross-restart ordering:** a full queue drops the event, a failed POST (transport/timeout/non-2xx) is logged and never retried, a single serial worker keeps in-run FIFO. **Worker placement:** the registry actor's `current_thread` runtime has no I/O reactor, so the worker is spawned on the main multi-thread runtime in `serve_with_authorizer` (before the registry thread); the sink carries only the channel `Sender` (Send) into the registry thread to register on the hub. **Event JSON:** `{"type": "<kebab>", …}` — `version-created {room,name}` / `version-renamed {room,from,to}` / `version-deleted {room,name}` / `compacted {room,floor}` / `subscribed {room}` / `after-restore {room,branch}` / `before-publish {room,branch}`; byte names as UTF-8 (lossy for non-UTF-8, as elsewhere); roomless connect/disconnect + reserved `before-migration` filtered out (same room-bearing filter as `AutoVersionSink`). Optional shared secret in the `X-Crdtsync-Webhook-Secret` header (over TLS, never the body). **Opt-in:** `ServeConfig.webhook: Option<WebhookConfig>`, `main.rs` reads `CRDTSYNC_WEBHOOK_URL` / `CRDTSYNC_WEBHOOK_SECRET`; unset ⇒ no sink, no worker, zero per-event cost (mirrors the optionally-wired admin plane). Server adds `reqwest` (rustls TLS) + `serde`, neither reaching core. Server-only diff (no `crates/core/src` → CI miri non-required/unaffected); spec `crates/server/tests/webhook.rs` (4, hermetic loopback axum capture server: shape+secret header, compaction floor + no-secret-header, no-sink-no-delivery, dead-endpoint-past-capacity never blocks/panics). Design in DECISIONS (2026-07-12). → *Versioning / Debugging (event bus = webhook substrate)*.

**Branches** — **Branches Unit 5 — publish/draft + per-user branches (server) — DONE (#249).** The draft→publish gate over the active-HEAD + branch-write machinery, closing the last reserved branch engine event. **Publish/draft:** `Hub::publish(room, published)` (`Registry::publish` wraps it to drive the auto-version drain; default target `PUBLISHED_BRANCH = b"published"`) points a **read-only `published` branch's HEAD** at the active editor branch's (`main` by default, or a restored HEAD) current materialized state — a snapshot fork owning that state as its base, so read-only consumers subscribe `(room, published)` and are served the state as it stood at the last publish while editors keep writing `main`. **Republish** repoints `published`'s base to the newer editor state; each published state is first captured as an immutable named version `publish/<published>/<source>@<seq>` (keyed on the source branch + its head, so the same head number on a switched editor branch stays a distinct rollback), so **the old published state stays reachable as a version** — apps roll published state back independently of the editor branch. **Read-only target:** a client `Ops` write to a `published` branch is refused recoverably (`OpsRejected{Forbidden}`, the same idiom as an authz denial — the author keeps its ops), enforced by a durable `Branch.published` flag; a published branch never diverges, so a stale tail under its name (a name reused from a former writable fork) is dropped on load, never folded onto the base. **Engine event:** `BeforePublish` (the last reserved branch event) now fires *before* the repoint; `AutoVersionSink` maps it to `TriggerEvent::BeforePublish`, so a schema's `autoVersion: [{on: before-publish}]` trigger captures at the publish point — the branch/migration engine-event reservations are now fully consumed on the branch side (only `BeforeMigration` remains, for the migration layer). **Per-user branches:** the same primitive — a `user/<id>` branch forked via `fork_branch`/`fork_branch_from_version` edits in isolation (Unit 3 branch-scoped write/fan-out); confirmed with test coverage proving isolation (neither user's writes reach the base or each other), no new machinery. Server-only diff (no `crates/core/src` → CI miri unaffected; `TriggerEvent::BeforePublish` already existed in core); specs `crates/server/tests/publish.rs` (10, incl. durable reopen + switched-branch-rollback + orphan-tail regressions), `engine_events.rs` (+before-publish fires-once). Two correctness-review agents (caught a rollback-name collision across a switched editor branch + a partial-rollback tail hazard → both fixed with regression tests). Design in DECISIONS (2026-07-12). **Deferred:** branch SDK surface (Unit 6), schema-aware branch diff (Unit 7), cross-branch merge (out of v0.x), per-user branch *authorization* (ACL slice 4). → *Versioning and Branches*.

**Branches** — **Branches Unit 4b — restore-as-branch + active-HEAD switch + `AfterRestore` event (server) — DONE (#248).** Completes Branches Unit 4 over the 4a fork-from-snapshot base. **Active HEAD:** a durable per-room default-served-branch pointer (`Hub.active_branch`, `<room>.active`, absent ⇒ `main`); `session::step`'s Subscribe resolves an *empty* branch to `hub.active_branch(room)` (replacing the old empty⇒`main` normalize), while an explicit name (incl. `main`) is taken as given. The resolved branch is stored on the channel, so a channel bound *before* a restore keeps its branch — an offline op in flight against the old HEAD lands on the old branch (Unit 3 branch-scoped write), never on the restored state; a new plain subscribe *after* restore follows the restored branch. **`Hub::restore_as_branch(room, version, new_branch)`:** forks `new_branch` from the version snapshot (4a), captures an **audit version** of the pre-restore live `main` state (`audit/restore/<branch>@<seq>` — keyed on captured seq so a reused branch name still audits each restore), switches the active HEAD, and emits `EngineEvent::AfterRestore`. Rewrites no history, resets no sequence; the old branch stays immutable + subscribable by name; deleting the active branch resets the HEAD to `main`. Durable throughout → the whole switch **replays on reload** (`.active` + restored `.branches`/`.bbase` + audit `.versions`; a dangling `.active` falls back to `main`). **Engine event:** `AfterRestore` (event bus #191) now fires; `AutoVersionSink` maps it to `TriggerEvent::AfterRestore` and `Registry::restore_as_branch` drives the auto-version drain, so a schema's `autoVersion: [{on: after-restore}]` trigger **captures the restored state — closing the parked branch-trigger item**; `BeforePublish` stays reserved for Unit 5. Server-only diff (no `crates/core/src` → CI miri unaffected); specs `crates/server/tests/restore_as_branch.rs` (9), `auto_version.rs` (+after-restore), `engine_events.rs` (+fires-once); two correctness-review agents (a caught audit-name-collision-on-reuse bug → keyed on seq + regression test). Design in DECISIONS (2026-07-12). → *Versioning and Branches*.

**Branches** — **Branches Unit 4a — fork-from-snapshot (server) — DONE (#247).** Lands the shared-base-below-a-fork-point machinery Unit 3 deferred: a branch may fork from a **named version's snapshot**, not only a live point in `main`'s log. **Base machinery:** `Hub::fork_branch_from_version(room, new, version)` records the branch pointer at `fork_point = version.seq` and copies the version's encoded whole-replica state into the branch's **owned base** — `Hub.branch_bases: room → branch → Vec<u8>` in memory, a durable `<room>_<branch>.bbase` file (state verbatim, whole-file read, room+branch hex-joined like `.blog`) — so the branch survives the source version's deletion. The presence of a base entry marks a branch a snapshot fork; a live-log fork (Unit 3) has no entry and reads `main`'s log as before. **Catch-up:** `catch_up_branch` serves the owned base ++ tail, never `main`'s post-version ops — a fresh subscriber (below `fork_point`) gets the base with the tail folded in as one `Catchup::Snapshot { seq: fork_point + tail.len() }` (decode → apply tail → encode); one already past the base gets just the tail as `Catchup::Ops`. **Lifecycle:** the `.bbase` lands before the `.branches` pointer (rolled back on a pointer-persist error), is dropped with the branch (mem + disk, so a re-fork inherits no stale base), and is reseeded from disk on reload — an orphan base whose branch is not a registered fork is pruned, so it never shadows a live-log fork reusing the name. **Counter `dec` end-to-end** — a Counter decrement surfaced through the whole binding stack: `path::dec` → FFI `crdtsync_doc_dec`/`crdtsync_client_dec` → Python/Go/wasm `dec`, mirroring `inc` (#96). Closes an SDK gap (the PN-counter's negative direction had no path/FFI/SDK entry point) and readies nested-counter undo.

**UndoManager (nested paths)** — undo is now path-addressed: every `UndoManager`/`Batch` method takes an encoded path, so a scalar slot inside a nested Map undoes exactly as a root one; a group can revert root + nested edits as one intention (#97). Backed by `path::register` (generic scalar) + `path::get_register`. List/text revival is the remaining undo slice.

**UndoManager (list revival)** — undo of a list insert deletes the node it minted (addressed by stable id, not a drifting index); undo of a list delete revives the removed value as a fresh insert at its place — the op log has no un-tombstone (#98). Backed by `List::live_index`, `ListCursor::delete_id`, `path::list_delete_id`/`list_live_index`. Text revival is the last undo slice.

**UndoManager (text revival)** — undo of a text insert deletes the run's char_ids; undo of a text delete revives the captured substring as a fresh run at its place (#99). Backed by `Text::live_index`, `TextCursor::delete_ids`, `path::text_delete_ids`/`text_run_ids`/`text_live_index`. Core undo now spans every value type (scalar / counter / list / text, root or nested); only the SDK surface remains.

**UndoManager (FFI)** — a `CrdtUndo` handle over the C ABI: `crdtsync_undo_new`/`_free` plus register_int/inc/dec/delete/list_insert/list_delete/text_insert/text_delete (each returns the ops to broadcast) and undo/redo/can_undo/can_redo (#100). The manager is a handle distinct from the `CrdtDoc` it drives, named on every call. Bytes-in-Register has no existing reader, so it is not exposed. Python/Go/wasm wrappers next.

**UndoManager (SDK wrappers)** — Python `Undo`, Go `Undo`, and wasm `WasmUndo` over the FFI/core surface: register_int/inc/dec/delete/list_insert/list_delete/text_insert/text_delete + undo/redo/can_undo/can_redo, each naming the document it drives (#101). Undo/redo now works end-to-end in every SDK; the subsystem is complete.

**Atomic transactions (core)** — `Document::atomic_transact` tags a group's ops with `Tx { id, count }` (op envelope + codec extended from the reserved `tx` slot); a receiver buffers members until the whole group is present and its external deps resolve, then applies them together in seq order — an all-or-nothing *view* boundary that preserves convergence (verified against the convergence harness) (#102). Buffered partials ride the existing op-buffer, so they survive a snapshot. Design logged in DECISIONS. Wire/server/client drive path + SDK surface are the follow-on slices.

**Atomic transactions (drive path)** — `ClientSession::atomic_edit` mirrors `edit` but tags the group as one transaction, sent as an ordinary `Message::Ops` (membership rides on the ops, no new wire message) (#103). The server needed no change: `Hub::ingest` returns every fresh op regardless of buffering, so a whole atomic batch fans out intact and a fresh subscriber's catch-up carries the full group — verified end to end (client peer folds it in all-or-nothing; hub preserves the tags through ingest + catch-up). SDK `atomic_edit`/`atomic_transact` is the remaining slice.

**Atomic transactions (SDK surface, doc-level)** — a begin/commit atomic API on the `Document` seat, since the C ABI has no closures: core `Document::begin_atomic`/`commit_atomic`/`is_atomic` (edits between them accumulate into one group; `atomic_transact` is now a thin wrapper), exposed as `crdtsync_doc_begin_atomic`/`_commit_atomic` (FFI), `begin_atomic`/`commit_atomic` (Python, Go), and `beginAtomic`/`commitAtomic` (wasm) (#104). Client-seat (per-channel) atomic wrappers over `ClientSession::atomic_edit` are the last slice.

**Atomic transactions (SDK surface, client-seat)** — per-channel begin/commit on the wire client: core `ClientSession::begin_atomic(channel)`/`commit_atomic(channel)` (the latter returns the group as one `Message::Ops`), exposed as `crdtsync_client_begin_atomic`/`_commit_atomic` (FFI) and `begin_atomic`/`commit_atomic` (Python, Go) / `beginAtomic`/`commitAtomic` (wasm) (#105). Edits recorded between them travel as one atomic group over the wire — a peer folds them all-or-nothing. **Atomic transactions are now complete end-to-end** (core → drive path → doc + client SDK seats).

Realizes ARCHITECTURE §Transactions "a transaction is naturally an undo intention": an `Intention` carries an `atomic` flag and `apply` wraps an atomic intention's inverse ops in a fresh atomic tx via the doc's `begin_atomic`/`commit_atomic`. Core-only, matching `group` (closure-shaped, not in the SDK surface).

**Schema-aware diff (core)** — `core::diff::diff(old, new) -> Vec<Change>` computes the structural changes between two replica snapshots by walking the Element trees in lockstep (#109): slots `Added`/`Removed` (with kind), scalar/register `Value` and `Counter` old→new value changes, nested maps recursed so a deep edit reports at its own path. Path-addressed (shared `path` encoding) and ordered by path, so diffing a pair is deterministic. Backed by `Map::keys()`. Pairs with named versions + export/import (compare any two snapshots).

**Schema-aware diff — change-list codec** — `core::diff::encode_changes`/`decode_changes` serialize a `Vec<Change>` to a tag-led byte buffer (`u32` count then each change) and back, so a diff crosses the language-SDK boundary as one buffer (#112). Round-trips every variant, decodes totally (`BadTag`/`TrailingBytes`/`UnexpectedEof`), reuses the op-codec primitives (`put_scalar`/`Cursor`). Not durable — a diff is transient. Backed by `ElementKind::from_tag`.

**Schema-aware diff — FFI entry point** — `crdtsync_diff(old_state, new_state) -> CrdtBuf` over the C ABI: decodes two snapshot buffers, diffs them, returns the encoded change list (#113). Stateless (no doc handle) — the inputs are any two state blobs (a `version_state`, an exported room, `encode_state`); empty on malformed input or a bad snapshot, never a panic. Header symbol asserted, Miri leak-clean.

**Schema-aware diff — wasm surface** — `WasmDocument.diff(oldState, newState)` returns an array of structural change objects, each with an `op` tag, a `path` (Uint8Array), and its variant's fields (#115). A scalar is a tagged `{ t, v }` object (`int`/`bool`/`null`/`bytes`/`blobref`) so binary values and full-range ints read unambiguously; list items are `{ scalar }` or `{ kind }`. Throws on a malformed snapshot. This establishes the cross-language change representation.

**Schema-aware diff — Python surface** — module-level `diff(old_state, new_state)` returns a list of change dicts over the C ABI `crdtsync_diff` (#116): a pure-Python `_Reader` decodes the change-list bytes into the same shape as wasm — `{"op", "path", ...}` with tagged `{"t","v"}` scalars and `{"scalar"}`/`{"kind"}` list items. Raises `ValueError` on a malformed snapshot.

**Schema-aware diff — Go surface** — `crdtsync.Diff(oldState, newState) ([]Change, error)` over the C ABI `crdtsync_diff` (#117): a `changeReader` decodes the change-list bytes into typed `Change` structs (`Op`/`Path` + per-variant fields), with a tagged `Scalar` and `Item` (scalar-or-kind), mirroring the wasm/Python shape. Errors on a malformed snapshot. **Diff runs end to end across every SDK** (core → codec → C ABI → wasm/Python/Go).

**Schema-aware diff — default renderer** — `core::diff::render(changes) -> Vec<String>` gives the engine's default human-readable rendering, one line per change (#119): `+`/`-`/`~` for add/remove/change, slash-joined paths, plainly-printed scalars, sequence runs with index + contents. Realizes ARCHITECTURE §Schema-Aware-Diff "engine ships sensible default renderers; apps can override" (an app renders the structured `Change`s its own way). XmlElement/marks/attrs diffs remain blocked on those unbuilt primitives.

**Schema-aware diff — sequence detail** — Text and List diff to runs by stable id: Text → `TextInsert{index,text}`/`TextDelete{index,text}` codepoint runs (#110), List → `ListInsert{index,items}`/`ListDelete{index,items}` item runs (`SeqItem::Scalar`/`Composite`) (#111). Stable-id identity makes both exact (no heuristic alignment); consecutive same-op elements coalesce into runs, deletes (old index) before inserts (new index). Text is codepoint-indexed. Realizes ARCHITECTURE §Schema-Aware-Diff "Text values produce char-level diffs" + structural list change lists. **Core diff now spans every built primitive** (map/register/counter/list/text). XmlElement/marks/attrs diffs (unbuilt primitives), renderers, and the SDK surface remain follow-ons (see Queue).

**RelativePosition / anchors (core)** — `core::anchor::RelativePosition`, a stable cursor/selection position in a List or Text sequence: `Start`/`End` boundaries + `Before(id)`/`After(id)` bound to a CRDT item id (left/right gravity), not an integer offset. `List::relative_position(index, side)` / `resolve_position` (+ Text codepoint wrappers): capture binds to the id at the gap; resolve returns the live index — a live binding to its item's edge, a deleted one walking the retained tombstones (#125) to the nearest live neighbour on the gravity side (clamping to 0/len), boundaries to 0/len. So an insert before shifts it, an insert at the gap lands per gravity, a delete of the bound item doesn't strand it — and it resolves identically on converged replicas. Binary codec (`encode`/`decode`, total-decode) so a position can cross the wire (awareness cursors carry one). Spec `crates/core/tests/anchor.rs` (#134). Realizes §Anchors "Exposed at SDK level as RelativePosition ... survive concurrent inserts and deletes without drifting. Core primitive." Unblocks marks/comments/RangedElement + awareness cursors; the SDK `RelativePosition` surface is the follow-on (see Next).

**Snapshot export / import** — a room's whole-replica state is portable across hubs: `Hub::export_room` hands back the snapshot bytes (`None` for an unknown room); `Hub::import_room` rebuilds a fresh room from them, persisting the snapshot durably before it commits so the import survives a restart (#107). Backup / cross-server-move / debug-repro: the merged state, element/client identities, and dedup set all ride the bytes, so a client resending its ops is deduped exactly as against the origin, and a fresh subscriber is caught up with the imported snapshot. Import is create-only (refuses an existing room rather than clobbering live state) and rejects malformed bytes as `InvalidData`, never a panic; sequences renumber from the imported op count (server-local, so no collision with the origin). Cloning under a *new* room id — clock bumps + id namespacing so two live copies can't collide — stays a follow-on (see Queue).

## 🔍 Revisit / tech-debt (accepted decisions flagged for a later look)

- **File-log vs. embedded DB for the query/metadata side** — the append-only file log is right for the op hot-path, but admin UI / op-log viewer / audit-query / retention want queryability, and durability is hand-rolled (a dir-fsync crash bug already shipped + fixed). Reconsider SQLite/redb for the metadata/index side when those consumers land. Checkpoint, not a reversal.
- **Cold-start snapshot CPU** — a below-floor subscriber triggers a whole-replica `encode_state` regenerated live per cold-start (O(state) CPU/connection). Fine now; cache the encoded snapshot per compaction floor if snapshots grow or cold-starts get frequent.
- **Cluster gossip failure detection — deferred refinements (Unit 7b, #PR).** The first cut ships detection + dissemination + refutation. Follow-ons, none blocking: (1) **SWIM indirect probing** (ping-req via k relays) — a suspected node is probed indirectly through other members before being declared dead, cutting false positives from one node's bad link. Detection is currently driven only by a node's *own initiated* probe outcomes; the inbound (dialed) side merges a frame but cannot identify the dialer to credit it reachable (the relay Hello carries a `ClientId`, not a `NodeId`). So under a **one-way/asymmetric partition** — A can reach B but B cannot reach A — B keeps failing its own probes to A and escalates A to `Dead` even while receiving A's gossip every round; A refutes (bump) so there is no *permanent* false death, but it flaps until the refutation lands. Indirect probing (or threading a sender id into the gossip frame so the dialed side credits it) closes this; symmetric failures (a crashed/dead host — the common case) detect correctly today. (2) **Member reaping** — a `Dead` member stays in the placement set marked down; a permanent leave/eviction that removes it (and reshuffles placement) is unbuilt, so a genuinely-gone node lingers as a placement replica forever. (3) **Late-joiner replication-dial wiring** — a member learned by gossip after boot now gets a real liveness reading, but the existing nodes' replication *peer-connections* to it are still dialed from the static boot set (`spawn_peers`), so a dynamically-joined node is routed to but not yet replicated to. (4) **Epoch persistence** (from 6b) — per-room leadership epoch is in-memory. Reconsider all four when a real multi-node deployment exercises churn.
